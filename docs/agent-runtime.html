2.1 运行时模块：为什么需要它？

场景一：嘴上说说，手上没做。 你对 ChatGPT 说"帮我把 parse_date 改成支持 ISO 格式"，它会输出一段漂亮的代码——然后等你自己去改文件。你对 OpenClaw 说同样的话，它直接读文件、定位函数、改代码、写回磁盘——文件真的变了。

场景二：做了一步就停了。 你让 Agent 帮你调研一个 API 的价格，它调用了一次搜索工具，返回了一个不完整的结果，然后告诉你"以下是搜索结果"——它不知道该继续搜还是该总结。你不得不再发一条消息让它继续。

场景三：做过头了停不下来。 Agent 调用了一个代码执行工具，结果有个 bug 触发了无限循环。每次循环它都认为"需要再试一次"。你眼睁睁看着 API 账单飙升，直到 token 上限被打满，Agent 才被动停止。

这三个场景对应运行时的三个核心职责：驱动 LLM 从"说"变成"做"、控制循环的推进逻辑、以及知道什么时候该停下来。没有它，LLM 只能输出文字；有了它，LLM 能真正做事——但也可能做过头。

运行时的设计质量，决定了你对 Agent 行为的掌控力。

2.3 核心算法：ReAct

运行时的核心算法叫 ReAct（Reasoning + Acting，Yao et al., ICLR 2023）。核心思想一句话：让 LLM 交替推理和行动——先想，再做，看结果，再想，直到任务完成。

一个完整的循环轨迹：

Thought 1: 用户想订明天下午3点的会议室，我先查有哪些空闲的
Action 1:  search_rooms(date="2025-03-01", time="15:00")
Obs 1:     空闲: 301会议室(6人), 502会议室(12人)
Thought 2: 有两个选项，用户没指定人数，选较小的301
Action 2:  book_room(room="301", date="2025-03-01", time="15:00", duration="1h")

Thought 3: 预订完成，告诉用户

翻译成代码，核心逻辑只需要 7 行：

while not exceeding_max_iterations:
    response = call_llm(tools, messages)
    if response.is_final_answer:
        return response.content
    for tool_call in response.tool_calls:
        result = execute_tool(tool_call)
        messages.append(result)

50 行 Python 就能实现一个可运行的版本（Lab 2.6 会让你亲手验证）。但这 7 行没有重试、没有权限过滤、没有过程追踪、没有并发隔离——生产级的五条评价标准（C1-C5）全部不及格。

为什么是 ReAct？

把已有的范式按"推理"和"行动"两个维度排列，ReAct 的定位就很清楚了：

范式 类型 核心机制 局限
CoT 纯推理 LLM 内部逐步推导 无法调用工具，中间结果可能是幻觉
Tree-of-Thought 纯推理 + 分支搜索 多条推理路径并行探索 搜索空间爆炸，仍无工具交互
Plan-and-Execute 先推理，后行动 先生成完整计划，再逐步执行 执行中发现新信息无法修正计划
Reflexion 行动 → 反思 → 重试 失败后自我反思，修正策略 反思步骤增加延迟和 token 成本
ReAct 推理与行动交织 Thought → Action → Observation 循环 通用 Agent 的事实标准

纯推理（CoT）的天花板：中间步骤没有真实数据验证，"事实"可能是幻觉。纯行动（早期 function calling）的天花板：遇到多步任务无法基于上一步结果做决策。ReAct 把两者交织在每一步——Thought 基于上一轮的真实 Observation，避免幻觉；Action 由 Thought 引导，避免盲目。

ReAct 成为事实标准还有一个现实原因：OpenAI、Anthropic、Google、DeepSeek 等主流提供商都原生支持 function calling / tool use——你在请求中声明工具的 JSON Schema，LLM 直接返回结构化的工具名 + 参数，不需要从自由文本中正则提取。这正是 ReAct 的 Action 步骤，开箱即用。

循环里的典型故障

7 行代码能跑通 happy path，但真实环境有三类常见故障：

  1. 幻觉工具名：LLM 编造不存在的工具（如 send_email，但你只注册了 get_weather）。不检查就会让 execute_tool() 崩溃。
  2. 参数错误：LLM 传了错误类型或漏填必填参数（如 get_weather(city=42) 而非字符串）。Schema 校验是必需品。
  3. 死循环：LLM 反复调用同一个工具不收敛（如搜索无结果时换关键词重搜，无限重复）。max_iterations 是最后防线。

这三类故障正是 C1-C5 评价标准存在的根本原因。C2 防止幻觉工具名绕过权限，C1 处理参数校验失败后的重试，C4 确保一个用户的死循环不阻塞其他用户。

2.4.1 循环引擎：抽成独立 SDK

📐 ADR-03: 为什么把循环引擎抽成独立 SDK？

  • 背景：ReAct 循环本身是一个已解决的问题，但 OpenClaw 的运行时还包含大量"循环之外"的工程（认证轮换、上下文压缩、策略过滤）
  • 决策：把循环引擎抽成独立 npm 包（Pi Agent Core），主项目只需两行调用
  • 放弃的替代方案：内联实现（与 50 万行代码纠缠，无法独立演进）；使用 LangChain AgentExecutor（抽象过厚，控制力不够）
  • 后果：循环引擎和"循环之外"的工程可以独立演进、独立测试，但多了一层版本管理开销

OpenClaw 选择了一个关键的架构决策：把 ReAct 循环抽成独立 SDK（内部称 Pi Agent Core）。

为什么要拆？因为 2.3 节已经说明——ReAct 循环本身是一个已解决的问题。把它留在 50 万行的主项目里，会和安全、渠道、调度的代码纠缠在一起。拆成独立 SDK 后，循环引擎和"循环之外"的工程可以独立演进。

这个架构决策直接服务于评价标准 C5（可扩展性）——SDK 内部封装了 4 个 LLM 提供商（OpenAI / Anthropic / Bedrock / Ollama）的 API 差异，主项目只需两行就能启动完整循环：

const session = createAgentSession(tools, messages, config);
const result = await session.prompt();

（此为简化示意，实际 createAgentSession() 接受包含工作目录、认证存储等字段的配置对象，如 {cwd, agentDir, authStorage, ...}）

注意：Pi Agent Core 是闭源包，读者无法直接查阅其源码。但"把循环引擎抽成独立模块"这个模式是通用的——如果你在自己的项目中需要类似的循环引擎，以下开源方案可以替代：

2.4.2 C1 可靠性：认证轮换 + 上下文压缩

LLM 提供商会限流、会宕机、会返回格式错误的响应。

OpenClaw 维护多个 Auth Profile（多把 API 钥匙），当一个 profile 失败时，自动标记冷却期并切换到下一个。如果上下文窗口溢出（token 超限），不是报错，而是自动压缩会话历史再重试。核心是一个双重 try-catch：外层捕获认证失败（切换 profile 重试），内层捕获上下文溢出（压缩后重试）。

上下文压缩（compaction.ts）分三层，从轻到重依次执行：

  1. 工具输出截断——长工具返回值（如大文件内容）在进入上下文前被截断，防止单次调用就撑爆窗口
  2. 历史分块裁剪——将对话历史按 token 比例切成 N 块，从最早的块开始丢弃，同时修复丢弃后产生的孤立 tool_use/tool_result 配对
  3. LLM 摘要生成——用 LLM 将被丢弃的历史压缩成一段摘要，保留关键决策、待办事项和约束条件，然后拼在剩余历史前面

这样 Agent 既不会因为上下文溢出而崩溃，也不会丢失之前对话中的关键信息——它只是"记忆变模糊了"。

2.4.3 C2 安全性：九层策略过滤

Agent 有 30+ 工具，其中包括 exec_shell_command。谁能用哪些工具？答案不是一个全局开关能解决的——管理员需要 Shell，普通用户不需要；同一个用户用 Claude 调用时某些工具不可用，换 GPT-4o 又可以。权限必须按用户 × 场景 × 模型三个维度组合控制。

OpenClaw 的做法：工具列表在到达 LLM 之前依次通过九层过滤（pi-tools.ts:434-460），每层只能缩小列表、不能扩大。每层携带一个 { deny: [...], allow: [...] } 策略，deny 优先——命中 deny 直接移除，否则必须显式 allow 才保留。

看一个具体例子就明白了：

30 个工具
  ─ Profile（用户级）        管理员，不过滤               → 30
  ─ ProviderProfile          无配置                       → 30
  ─ Global（全局级）         deny ["gateway"]             → 29
  ─ GlobalProvider           无配置                       → 29
  ─ Agent（实例级）          allow ["exec*","read*","write*","search*"]  → 18
  ─ AgentProvider            无配置                       → 18
  ─ Group（群组级）          deny ["exec_shell*"]         → 17
  ─ Sandbox（沙箱级）        无配置                       → 17
  ─ Subagent（子Agent级）    deny ["sessions*","cron","memory*"]        → 14
最终 LLM 只看到 14 个工具

九层分五个级别，每个级别管一个维度的权限：

级别 包含的层 管什么
用户级 Profile, ProviderProfile 这个人能用什么（管理员 vs 普通用户）
全局级 Global, GlobalProvider 所有人都不能用什么（危险管理工具）
实例级 Agent, AgentProvider 这个 Agent 角色能用什么（客服只能查询）
群组级 Group 这个通道/群组能用什么（家庭群禁用代码工具）
运行时级 Sandbox, Subagent 这个执行环境能用什么（Docker 容器、子 Agent）

2.4.4 C3 可观测性：事件订阅模型

Agent 的行为不确定——同样的输入可能走不同的工具路径。你需要在运行过程中实时看到它在想什么、做什么，而不是等 30 秒后只拿到一个最终结果。

对比两种方式：

# 黑盒等待
result = agent.run(query)       # 30 秒... 什么都看不到
print(result)                    # 只有最终结果

# 事件流
for event in agent.stream(query):
    if event.type == "tool_execution_start":
        print(f"[调用] {event.tool}({event.args})")
    elif event.type == "message_update":
        print(event.text, end="")
    elif event.type == "agent_end":
        print(f"\n[完成] {event.tokens} tokens")

OpenClaw 选方式 B——监听 SDK 事件流（pi-embedded-subscribe.ts），不侵入循环内部即可观测全过程：

事件 触发时机 用途
tool_execution_start Agent 决定调工具 记录工具名和参数，UI 展示执行进度
tool_execution_end 工具返回结果 格式化结果，注入上下文
message_update LLM 输出文本片段 流式推送给用户
agent_end 循环结束 汇总 token 用量，计费

当 Agent 调了三个工具但结果不对，事件流让你精确定位是哪次调用返回了错误数据——print(result) 永远做不到这一点。

2.4.5 C4 隔离性：命令通道（Lane）

问题：如果两个用户同时给 Agent 发消息，共享的 messages 列表会交叉污染：

messages = []  # 如果全局共享...

# 用户 A 问天气（messages 被注入天气上下文）
messages.append({"role": "user", "content": "北京天气怎么样？"})
# → LLM 调用 get_weather → 结果注入 messages
messages.append({"role": "tool", "content": "北京：晴 15°C"})

# 用户 B 同时问股价（此时 messages 里已经有 A 的天气上下文！）
messages.append({"role": "user", "content": "苹果股价多少？"})
# → LLM 看到天气上下文，可能回答"北京今天苹果价格..."

这不是"偶尔出 bug"——只要有并发就必然出错。根源是 messages 列表被共享了。

解法——Lane（命令通道）：OpenClaw 维护一个 Map<string, Queue>（process/command-queue.ts），每个 lane 是一个独立的 FIFO 队列，默认 maxConcurrent = 1（串行）。核心函数 enqueueCommandInLane(lane, task) 的逻辑：

  1. 把 task 推入指定 lane 的队列
  2. 如果 lane 当前没有正在执行的任务 → 立即执行
  3. 如果 lane 已有任务在跑 → 排队等待，前一个完成后自动执行下一个

Lane 的命名决定了隔离边界：

cron 定时任务 定时任务不阻塞用户请求

三条 Lane 互相并行（不同用户不等待），但每条 Lane 内部严格串行（同一用户的请求排队执行）。这样每个请求看到的 messages 列表、工具上下文都是自己 session 独占的，不会交叉污染。并发控制不是性能优化，是正确性问题。

2.4.6 C5 可扩展性：让"换一个"变得廉价

生产级 Agent 会不断面临"换一个"的需求——换提供商、加工具、改过滤规则。如果每次都要改散落在多处的代码，系统会在迭代中腐化。OpenClaw 通过两层隔离把常见变更收敛到一个位置：

变更场景 靠哪层隔离 改什么 不改什么
换 LLM 提供商 SDK 分离——循环引擎封装了 4 家提供商的 API 差异 一个环境变量 业务代码
加一个新工具 工厂函数——createOpenClawCodingTools() 集中管理工具创建 工厂内加一个定义 所有调用方
改一条过滤规则 九层过滤链（C2） 链中插入一层 工具定义和调用方
换参数校验库（Zod → AJV） 工厂函数内部 替换校验逻辑 工具定义和调用方

注：Zod 和 AJV 都是参数校验库——当 LLM 返回 get_weather(city=42) 时，校验库在执行前拦截类型错误（city 应为字符串）。

两层隔离的核心思想一致：把变化的部分封装起来，让调用方不感知变化。 Lab 中的 create_demo_tools() 和 create_llm() 也是工厂函数——从最简实现就建立这个抽象，后续扩展才不痛苦。

这是三个模块中最大的，因为 C1-C5 的核心实现全部集中于此。

auto-reply/（~21k 行）——用户的"前台"

用户的消息不会直接触发 Agent 循环——它先经过一条流水线：

  1. 入队去抖：多条快速连续的消息被合并，避免为每条都启动一轮循环
  2. 模型分发：根据消息内容和用户配置选择模型——简单问候直接用轻量模型回复，复杂任务才启动完整 Agent
  3. 并发控制：限制同时运行的 Agent 实例数，防止资源耗尽（C4）
  4. 回复格式化：Agent 输出的 Markdown / 代码块 / <thinking> 标签需要根据目标通道（WhatsApp、Web、CLI）适配

commands/（~29k 行）——CLI 管理界面

这个模块是用户和运维人员通过命令行与系统交互的入口。最大的一块是 onboarding 流程（30+ 文件）：首次设置、API 密钥配置、OAuth 认证、通道接入、技能安装。其次是运维管理：系统健康检查（status）、模型列表与切换（models）、会话管理、沙箱管理。agentCommand() 是 Agent 的 CLI 启动入口，解析参数后直接调用 agents/ 模块，跳过 auto-reply/ 的消息队列。

分层的价值：每一层为上一层屏蔽复杂度。auto-reply/ 不需要知道九层过滤怎么运作，agents/ 不需要知道消息从哪个通道来。这样任何一层的内部重构都不会波及其他层——这本身就是 C5（可扩展性）在架构层面的体现。

"""mini_agent/runtime.py -- 运行时模块：ReAct 循环引擎

模块规格（见 2.2 节）：
  输入: 用户消息 + 工具集 + 系统提示词 + 对话历史
  输出: 最终回答 + 副作用 + 执行轨迹

  1. OpenAI     -- 设置 OPENAI_API_KEY
  2. DeepSeek   -- 设置 DEEPSEEK_API_KEY
  3. Ollama     -- 本地运行 ollama serve
  4. MockLLM    -- 零依赖后备

单独运行: python -m mini_agent.runtime
依赖: openai (pip install openai)
"""
from __future__ import annotations

import json, os
from dataclasses import dataclass
from typing import Any, Callable, Protocol


# -- 工具抽象 -----------------------------------------------------------

@dataclass
class Tool:
    """单个工具定义: 名称 + 描述 + JSON Schema + 处理函数"""
    name: str
    description: str
    parameters: dict[str, dict]
    handler: Callable[..., str]

    def schema(self) -> dict:
        return {"type": "function", "function": {
            "name": self.name, "description": self.description,
            "parameters": {"type": "object",
                           "properties": self.parameters,
                           "required": list(self.parameters.keys())}}}


class ToolRegistry:
    """工具注册表 -- 按名称注册和查找工具"""

    def __init__(self):
        self._tools: dict[str, Tool] = {}

    def register(self, tool: Tool) -> None:
        self._tools[tool.name] = tool

    def get(self, name: str) -> Tool | None:
        return self._tools.get(name)

    def schemas(self) -> list[dict]:
        return [t.schema() for t in self._tools.values()]

    def names(self) -> list[str]:
        return list(self._tools.keys())


# -- LLM 后端 (C5 可扩展性: 统一接口) ------------------------------------
# 归一化返回: (finish_reason, content, tool_calls)
# tool_calls = [(call_id, function_name, args_dict), ...]

class LLMBackend(Protocol):
    def call(self, messages: list[dict], tools: list[dict]) -> tuple: ...


class OpenAICompatibleLLM:
    """OpenAI 兼容 API 后端 -- 支持 OpenAI / DeepSeek / Groq / Ollama 等"""

    def __init__(self, model: str, api_key: str,
                 base_url: str | None = None, label: str = ""):
        from openai import OpenAI
        self.client = OpenAI(api_key=api_key, base_url=base_url)
        self.model = model
        self.label = label or model

    def call(self, messages: list[dict], tools: list[dict]) -> tuple:
        kwargs: dict[str, Any] = {"model": self.model, "messages": messages}
        if tools:
            kwargs["tools"] = tools
        resp = self.client.chat.completions.create(**kwargs)
        c = resp.choices[0]
        tc = [(t.id, t.function.name, json.loads(t.function.arguments))
              for t in (c.message.tool_calls or [])]
        return (c.finish_reason, c.message.content, tc)


class MockLLM:
    """关键词匹配 Mock -- 零依赖后备，可体验 ReAct 循环基本流程"""
    label = "MockLLM (建议设置 API key 使用真实 LLM)"

    def call(self, messages: list[dict], tools: list[dict]) -> tuple:
        import re
        if any(m.get("role") == "tool" for m in messages[-8:]):
            results = [m["content"] for m in messages if m.get("role") == "tool"]
            return ("stop",
                    "[Mock] 根据工具返回结果:\n" + "\n".join(f"  - {r}" for r in results),
                    [])
        user_msg = next(
            (m["content"] for m in reversed(messages) if m.get("role") == "user"), "")
        tool_names = {t["function"]["name"] for t in tools}
        calls, n = [], 0
        if "get_weather" in tool_names:
            for city in ["北京", "上海", "深圳", "Beijing", "Shanghai"]:
                if city in user_msg:
                    calls.append((f"mock-{n}", "get_weather", {"city": city})); n += 1

            calls.append((f"mock-{n}", "search_web", {"query": user_msg[:50]})); n += 1
        if "calculator" in tool_names:
            expr = re.search(r'[\d(][\d+\-*/.()\s]+[\d)]', user_msg)
            if expr:
                calls.append((f"mock-{n}", "calculator", {"expression": expr.group().strip()}))
        if calls:
            return ("tool_calls", None, calls)
        return ("stop", f"[Mock] 收到: {user_msg}. 可用工具: {list(tool_names)}", [])

# -- ReAct 循环引擎 (核心算法, 见 2.3 节) ---------------------------------

class ReactRuntime:
    """运行时模块的核心实现

    可选集成 (鸭子类型, 无硬依赖):
      memory  -- 提供 add(role, content) 和 context() -> list[dict]
      sandbox -- 提供 check(tool_name) -> bool
    """

    def __init__(self, llm: LLMBackend, tools: ToolRegistry,
                 *, memory=None, sandbox=None, max_iterations: int = 10):
        self.llm = llm
        self.tools = tools
        self.memory = memory
        self.sandbox = sandbox
        self.max_iter = max_iterations

    def run(self, user_message: str, system_prompt: str | None = None) -> str:
        if self.memory:
            self.memory.add("user", user_message)

        sys_msg = system_prompt or "你是一个有用的助手。需要时使用工具回答问题。"
        if self.memory:
            messages = [{"role": "system", "content": sys_msg},
                        *self.memory.context()]
        else:
            messages = [{"role": "system", "content": sys_msg},
                        {"role": "user", "content": user_message}]

        schemas = self.tools.schemas()

        for i in range(self.max_iter):
            finish, content, tool_calls = self.llm.call(messages, schemas)

            assistant_msg: dict[str, Any] = {"role": "assistant", "content": content}
            if tool_calls:
                assistant_msg["tool_calls"] = [
                    {"id": cid, "type": "function",
                     "function": {"name": fn, "arguments": json.dumps(args)}}
                    for cid, fn, args in tool_calls]
            messages.append(assistant_msg)

            if finish == "stop":
                if self.memory and content:
                    self.memory.add("assistant", content)
                return content or ""

            for call_id, func_name, args in tool_calls:
                tool = self.tools.get(func_name)
                if not tool:
                    result = f"Error: unknown tool '{func_name}'"
                elif self.sandbox and not self.sandbox.check(func_name):
                    result = f"Error: '{func_name}' blocked by sandbox"
                else:
                    try:
                        result = str(tool.handler(**args))
                    except Exception as e:
                        result = f"Error: {e}"
                print(f"[tool_call] {func_name}({args})")
                print(f"         → {result}")
                messages.append({"role": "tool", "tool_call_id": call_id,
                                 "content": result})

        return "达到最大迭代次数"

2.1 The Runtime Module: Why Do We Need It?

Scenario 1: All talk, no action. You ask ChatGPT to "refactor parse_date to support ISO format" and it outputs a nice code block — for you to apply yourself. You tell OpenClaw the same thing, and it reads the file, locates the function, edits the code, and writes it back to disk — the file actually changes.

Scenario 2: One step and stop. You ask the Agent to research an API’s pricing. It calls a search tool once, gets an incomplete result, then replies "here are the search results" — it doesn’t know whether to search more or summarize. You have to send another message to keep it going.

Scenario 3: Overdoing it and not stopping. The Agent invokes a code execution tool with a bug that triggers an infinite loop. Each iteration it thinks "need to try again." You watch the API bill climb until the token limit is hit and the Agent stops passively.

These three scenarios map to the runtime’s three core responsibilities: driving the LLM from "talking" to "doing," controlling loop progression, and knowing when to stop. Without it, the LLM can only output text; with it, the LLM can actually act — but it can also overdo it.

The design quality of the runtime determines how much control you have over Agent behavior.

2.3 The Core Algorithm: ReAct

The runtime's core algorithm is called ReAct (Reasoning + Acting, Yao et al., ICLR 2023). The idea in one sentence: have the LLM alternate between reasoning and acting — think, act, observe the result, think again, until the task is done.

A complete loop trace:

Thought 1: The user wants to book a meeting room for tomorrow 3PM. Let me check availability
Action 1:  search_rooms(date="2025-03-01", time="15:00")
Obs 1:     Available: Room 301 (6 pax), Room 502 (12 pax)

Action 2:  book_room(room="301", date="2025-03-01", time="15:00", duration="1h")

Translated into code, the core logic is just 7 lines:

while not exceeding_max_iterations:
    response = call_llm(tools, messages)
    if response.is_final_answer:
        return response.content
    for tool_call in response.tool_calls:
        result = execute_tool(tool_call)
        messages.append(result)

50 lines of Python yields a working version (Lab 2.6 lets you verify this). But these 7 lines have no retry, no permission filtering, no tracing, no concurrency isolation — all five production criteria (C1-C5) fail.

Why ReAct?

Arrange existing paradigms along the "reasoning" vs "acting" dimensions, and ReAct's position becomes clear:

Paradigm Type Core Mechanism Limitation
CoT Reasoning only LLM derives step by step internally Cannot call tools; intermediate "facts" may be hallucinations
Tree-of-Thought Reasoning + branch search Explores multiple reasoning paths in parallel Search space explosion; still no tool interaction
Plan-and-Execute Reason first, act later Generate full plan, then execute sequentially New info during execution cannot revise the plan
Reflexion Act → reflect → retry Self-reflect after failure, revise strategy Reflection adds latency and token cost
ReAct Reasoning and acting interleaved Thought → Action → Observation loop De facto standard for general Agents

Pure reasoning (CoT) hits a ceiling: intermediate steps have no real-data verification, so "facts" may be hallucinations. Pure action (early function calling) hits a different ceiling: multi-step tasks need decisions based on previous results, which require reasoning. ReAct interleaves both at every step — Thought is grounded in the previous Observation's real data; Action is guided by Thought rather than called blindly.

ReAct becoming the de facto standard also has a practical reason: OpenAI, Anthropic, Google, DeepSeek, and other major providers all natively support function calling / tool use — you declare tool JSON Schemas in the request, the LLM returns structured tool name + arguments, no regex extraction from free text needed. This maps directly to ReAct's Action step, ready out of the box.

Typical Failures Inside the Loop

The 7-line code handles the happy path, but real environments have three common failure modes:

  1. Hallucinated tool names: The LLM invents a nonexistent tool (e.g., send_email when only get_weather is registered). Without validation, execute_tool() crashes.
  2. Parameter errors: Wrong types or missing required params (e.g., get_weather(city=42) instead of a string). Schema validation is non-negotiable.
  3. Infinite loops: The LLM calls the same tool repeatedly without converging (e.g., search returns nothing, tries another keyword, ad infinitum). max_iterations is the last line of defense.

These failures are the fundamental reason C1-C5 exist. C2 prevents hallucinated tools from bypassing permissions, C1 handles retry after validation failures, C4 ensures one user's infinite loop doesn't block others.

The next section examines how OpenClaw closes these gaps.

2.4.2 C1 Reliability: Auth Rotation + Context Compression

LLM providers rate-limit, crash, and return malformed responses.

OpenClaw maintains multiple Auth Profiles (multiple API keys). When one profile fails, it's automatically marked for cooldown and the system switches to the next. If the context window overflows (token limit exceeded), instead of erroring, it automatically compresses conversation history and retries. The core is a double try-catch: the outer catches auth failures (switch profile and retry), the inner catches context overflow (compress and retry).

Context compaction (compaction.ts) operates in three layers, from lightest to heaviest:

  1. Tool output truncation — verbose tool returns (e.g., large file contents) are truncated before entering the context, preventing a single call from blowing the window
  2. History chunk pruning — conversation history is split into N chunks by token share; oldest chunks are dropped first, with orphaned tool_use/tool_result pairs repaired after each drop
  3. LLM summary generation — the LLM summarizes dropped history into a condensed paragraph, preserving key decisions, TODOs, and constraints, then prepends it to the remaining history

This way the Agent neither crashes on context overflow nor loses critical information from earlier in the conversation — it just "remembers less precisely."

2.4.3 C2 Security: Nine-Layer Policy Filtering

The Agent has 30+ tools, including exec_shell_command. Who gets to use which tools? A single global toggle can't solve this — admins need Shell, regular users don't; the same user may have different tools available depending on which model they're using. Permissions must be controlled along three dimensions: user × context × model.

OpenClaw's approach: the tool list passes through nine filter layers before reaching the LLM (pi-tools.ts:434-460), each layer can only shrink the list, never expand it. Each layer carries a { deny: [...], allow: [...] } policy with deny-first logic — matching deny removes the tool immediately, otherwise it must be explicitly allowed.

A concrete example makes this clear:

2.4.4 C3 Observability: Event Subscription Model

Agent behavior is non-deterministic — the same input may take different tool paths. You need to see what it's thinking and doing in real time, not wait 30 seconds and get only a final result.

Compare two approaches:

# Black-box wait
result = agent.run(query)       # 30 seconds... see nothing
print(result)                    # Only the final result

# Event stream
for event in agent.stream(query):
    if event.type == "tool_execution_start":
        print(f"[Calling] {event.tool}({event.args})")
    elif event.type == "message_update":
        print(event.text, end="")
    elif event.type == "agent_end":
        print(f"\n[Done] {event.tokens} tokens")

OpenClaw uses approach B — listening to SDK event streams (pi-embedded-subscribe.ts), achieving full observability without intruding into the loop:

Event Fires when Purpose
tool_execution_start Agent decides to call a tool Log tool name + args, show progress in UI
tool_execution_end Tool returns result Format result, inject into context
message_update LLM outputs a text fragment Stream to user in real time
agent_end Loop completes Aggregate token usage, billing

When an Agent calls three tools but produces wrong results, event streams let you pinpoint exactly which call returned bad data — print(result) can never do that.

2.4.5 C4 Isolation: Command Lane

Consider a concrete concurrency conflict scenario:

messages = []  # If globally shared...

# User A asks about weather (weather context injected into messages)
messages.append({"role": "user", "content": "What's the weather in Beijing?"})
# → LLM calls get_weather → result injected into messages
messages.append({"role": "tool", "content": "Beijing: Sunny 15°C"})

# User B simultaneously asks about stocks (messages already has A's weather context!)
messages.append({"role": "user", "content": "What's Apple's stock price?"})
# → LLM sees weather context, might answer "Beijing's apple prices today..."

This isn't an "occasional bug" — with concurrency it will fail. The root cause is a shared messages list.

Solution — Lanes (Command Channels): OpenClaw maintains a Map<string, Queue> (process/command-queue.ts), where each lane is an independent FIFO queue with maxConcurrent = 1 (serial by default). The core function enqueueCommandInLane(lane, task):

  1. Push task into the specified lane's queue
  2. If the lane has no active task → execute immediately
  3. If the lane already has a running task → wait in queue; auto-execute when the previous one completes

The lane key determines the isolation boundary:

Lane Key Source Effect
session:<UserA-WhatsApp> Session ID All of User A's requests execute serially, no interleaving
session:<UserB-Telegram> Session ID User B runs independently in parallel, not blocked by A
cron Scheduled tasks Cron jobs don't block user requests
subagent Sub-Agents Sub-Agent tasks don't block the parent Agent

The three lanes run in parallel (different users don't wait for each other), but within each lane requests are strictly serial (same user's requests queue up). This way each request sees its own session-exclusive messages list and tool context with no cross-contamination. Concurrency control isn't performance optimization — it's a correctness requirement.

2.4.6 C5 Extensibility: Making "Swap One Thing" Cheap

A production Agent constantly faces "swap one thing" demands — switch providers, add tools, change filtering rules. If every swap touches scattered code, the system rots over time. OpenClaw uses two layers of isolation to confine common changes to a single location:

Change Scenario Isolation Layer What Changes What Doesn't
Switch LLM provider SDK separation — loop engine encapsulates 4 providers' API differences One env variable Business code
Add a new tool Factory function — createOpenClawCodingTools() centralizes tool creation One definition inside the factory All callers
Change a filtering rule Nine-layer chain (C2) Insert one layer Tool definitions and callers
Swap validation library (Zod → AJV) Factory internals Replace validation logic Tool definitions and callers

Note: Zod and AJV are both parameter validation libraries — when the LLM returns get_weather(city=42), the validator catches the type error (city should be a string) before execution.

The core idea behind both layers is the same: encapsulate what varies so callers don't feel the change. The Lab's create_demo_tools() and create_llm() are also factory functions — establishing this abstraction from the start makes future extensions painless.

User messages don't directly trigger the Agent loop — they first pass through a pipeline:

  1. Queue & debounce: Multiple rapid successive messages are merged, avoiding a full loop launch for each one
  2. Model dispatch: Selects the right model based on message content and user config — simple greetings get a lightweight model, complex tasks get the full Agent
  3. Concurrency control: Limits simultaneous Agent instances to prevent resource exhaustion (C4)
  4. Reply formatting: Agent output containing Markdown / code blocks / <thinking> tags is adapted for the target channel (WhatsApp, Web, CLI)

commands/ (~29k lines) — The CLI Management Interface

This module is the command-line entry point for users and operators. The largest chunk is the onboarding flow (30+ files): first-time setup, API key configuration, OAuth authentication, channel onboarding, and skill installation. Next is operations management: system health checks (status), model listing and switching (models), session management, and sandbox management. agentCommand() is the Agent's CLI launch entry — it parses arguments and calls agents/ directly, bypassing auto-reply/'s message queue.

The value of layering: Each layer shields the one above from complexity. auto-reply/ doesn't need to know how nine-layer filtering works; agents/ doesn't need to know which channel the message came from. This means internal refactoring of any layer won't ripple into the others — which is itself C5 (extensibility) manifested at the architecture level.

Reference Implementation: mini_agent/runtime.py (~290 lines) 
"""mini_agent/runtime.py -- Runtime Module: ReAct Loop Engine

Module specification (see Section 2.2):
  Inputs: User message + tool set + system prompt + conversation history
  Outputs: Final answer + side effects + execution trace
  Algorithm: ReAct (Thought -> Action -> Observation loop)

Supported LLM providers (auto-detected, priority high to low):
  1. OpenAI     -- set OPENAI_API_KEY
  2. DeepSeek   -- set DEEPSEEK_API_KEY
  3. Ollama     -- run ollama serve locally
  4. MockLLM    -- zero-dependency fallback

Standalone run: python -m mini_agent.runtime
Dependency: openai (pip install openai)
"""
from __future__ import annotations

import json, os
from dataclasses import dataclass
from typing import Any, Callable, Protocol


# -- Tool Abstraction ---------------------------------------------------

@dataclass
class Tool:
    """Single tool definition: name + description + JSON Schema + handler"""
    name: str
    description: str
    parameters: dict[str, dict]
    handler: Callable[..., str]

    def schema(self) -> dict:
        return {"type": "function", "function": {
            "name": self.name, "description": self.description,
            "parameters": {"type": "object",
                           "properties": self.parameters,
                           "required": list(self.parameters.keys())}}}


class ToolRegistry:
    """Tool registry -- register and lookup tools by name"""

    def __init__(self):
        self._tools: dict[str, Tool] = {}

    def register(self, tool: Tool) -> None:
        self._tools[tool.name] = tool

    def get(self, name: str) -> Tool | None:
        return self._tools.get(name)

    def schemas(self) -> list[dict]:
        return [t.schema() for t in self._tools.values()]

    def names(self) -> list[str]:
        return list(self._tools.keys())


# -- LLM Backends (C5 Extensibility: unified interface) -----------------
# Normalized return: (finish_reason, content, tool_calls)
# tool_calls = [(call_id, function_name, args_dict), ...]

class LLMBackend(Protocol):
    def call(self, messages: list[dict], tools: list[dict]) -> tuple: ...


class OpenAICompatibleLLM:
    """OpenAI-compatible API backend -- supports OpenAI / DeepSeek / Groq / Ollama"""

    def __init__(self, model: str, api_key: str,
                 base_url: str | None = None, label: str = ""):
        from openai import OpenAI
        self.client = OpenAI(api_key=api_key, base_url=base_url)
        self.model = model
        self.label = label or model

    def call(self, messages: list[dict], tools: list[dict]) -> tuple:
        kwargs: dict[str, Any] = {"model": self.model, "messages": messages}
        if tools:
            kwargs["tools"] = tools
        resp = self.client.chat.completions.create(**kwargs)
        c = resp.choices[0]
        tc = [(t.id, t.function.name, json.loads(t.function.arguments))
              for t in (c.message.tool_calls or [])]
        return (c.finish_reason, c.message.content, tc)


class MockLLM:
    """Keyword-matching Mock -- zero-dependency fallback for basic ReAct demo"""
    label = "MockLLM (set an API key for real LLM)"

    def call(self, messages: list[dict], tools: list[dict]) -> tuple:
        import re
        if any(m.get("role") == "tool" for m in messages[-8:]):
            results = [m["content"] for m in messages if m.get("role") == "tool"]
            return ("stop",
                    "[Mock] Based on tool results:\n" + "\n".join(f"  - {r}" for r in results),
                    [])
        user_msg = next(
            (m["content"] for m in reversed(messages) if m.get("role") == "user"), "")
        tool_names = {t["function"]["name"] for t in tools}
        calls, n = [], 0
        if "get_weather" in tool_names:
            for city in ["Beijing", "Shanghai", "Shenzhen"]:
                if city.lower() in user_msg.lower():
                    calls.append((f"mock-{n}", "get_weather", {"city": city})); n += 1
        if "search_web" in tool_names and re.search(r"search|find|look up", user_msg, re.I):
            calls.append((f"mock-{n}", "search_web", {"query": user_msg[:50]})); n += 1
        if "calculator" in tool_names:
            expr = re.search(r'[\d(][\d+\-*/.()\s]+[\d)]', user_msg)
            if expr:
                calls.append((f"mock-{n}", "calculator", {"expression": expr.group().strip()}))
        if calls:
            return ("tool_calls", None, calls)
        return ("stop", f"[Mock] Received: {user_msg}. Available tools: {list(tool_names)}", [])


def create_llm(config: dict | None = None) -> OpenAICompatibleLLM | MockLLM:
    """Factory function -- auto-detect available LLM provider

    Detection priority: OpenAI -> DeepSeek -> Ollama -> MockLLM
    Can also manually specify provider / model / api_key / base_url via config
    """
    cfg = config or {}

    if cfg.get("api_key"):
        return OpenAICompatibleLLM(
            model=cfg.get("model", "gpt-4o-mini"),
            api_key=cfg["api_key"],
            base_url=cfg.get("base_url"),
            label=cfg.get("label", ""))

    if key := os.environ.get("OPENAI_API_KEY"):
        return OpenAICompatibleLLM("gpt-4o-mini", key, label="OpenAI gpt-4o-mini")

    if key := os.environ.get("DEEPSEEK_API_KEY"):
        return OpenAICompatibleLLM(
            "deepseek-chat", key,
            base_url="https://api.deepseek.com",
            label="DeepSeek deepseek-chat")

    try:
        import urllib.request
        urllib.request.urlopen("http://localhost:11434/api/tags", timeout=1)
        return OpenAICompatibleLLM(
            "qwen2.5:7b", "ollama",
            base_url="http://localhost:11434/v1",
            label="Ollama qwen2.5:7b")
    except Exception:
        pass

    return MockLLM()


# -- ReAct Loop Engine (core algorithm, see Section 2.3) ----------------

class ReactRuntime:
    """Core implementation of the runtime module

    Optional integrations (duck typing, no hard dependencies):
      memory  -- provides add(role, content) and context() -> list[dict]
      sandbox -- provides check(tool_name) -> bool
    """

    def __init__(self, llm: LLMBackend, tools: ToolRegistry,
                 *, memory=None, sandbox=None, max_iterations: int = 10):
        self.llm = llm
        self.tools = tools
        self.memory = memory
        self.sandbox = sandbox
        self.max_iter = max_iterations

    def run(self, user_message: str, system_prompt: str | None = None) -> str:
        if self.memory:
            self.memory.add("user", user_message)

        sys_msg = system_prompt or "You are a helpful assistant. Use tools when needed."
        if self.memory:
            messages = [{"role": "system", "content": sys_msg},
                        *self.memory.context()]
        else:
            messages = [{"role": "system", "content": sys_msg},
                        {"role": "user", "content": user_message}]

        schemas = self.tools.schemas()

        for i in range(self.max_iter):
            finish, content, tool_calls = self.llm.call(messages, schemas)

            assistant_msg: dict[str, Any] = {"role": "assistant", "content": content}
            if tool_calls:
                assistant_msg["tool_calls"] = [
                    {"id": cid, "type": "function",
                     "function": {"name": fn, "arguments": json.dumps(args)}}
                    for cid, fn, args in tool_calls]
            messages.append(assistant_msg)

            if finish == "stop":
                if self.memory and content:
                    self.memory.add("assistant", content)
                return content or ""

            for call_id, func_name, args in tool_calls:
                tool = self.tools.get(func_name)
                if not tool:
                    result = f"Error: unknown tool '{func_name}'"
                elif self.sandbox and not self.sandbox.check(func_name):
                    result = f"Error: '{func_name}' blocked by sandbox"
                else:
                    try:
                        result = str(tool.handler(**args))
                    except Exception as e:
                        result = f"Error: {e}"
                print(f"[tool_call] {func_name}({args})")
                print(f"         → {result}")
                messages.append({"role": "tool", "tool_call_id": call_id,
                                 "content": result})

        return "Max iterations reached"

def _get_weather(city: str) -> str:
    data = {"Beijing": "Sunny 15°C", "Shanghai": "Cloudy 18°C",
            "Shenzhen": "Showers 24°C"}
    return data.get(city, f"Weather not found for {city}")

def _search_web(query: str) -> str:
    return f"Search '{query}': Top frameworks 2025 -- LangChain, CrewAI, AutoGen, Smolagents"

def _calculator(expression: str) -> str:
    allowed = set("0123456789+-*/.() ")
    if not all(c in allowed for c in expression):
        return "Error: illegal characters"
    try:
        return str(eval(expression))
    except Exception as e:
        return f"Error: {e}"

def create_demo_tools() -> ToolRegistry:
    reg = ToolRegistry()
    reg.register(Tool("get_weather", "Query city weather",
                       {"city": {"type": "string", "description": "City name"}},

    reg.register(Tool("search_web", "Search the web",
                       {"query": {"type": "string", "description": "Search keywords"}},
                       _search_web))
    reg.register(Tool("calculator", "Math calculation",
                       {"expression": {"type": "string", "description": "Math expression"}},
                       _calculator))
    return reg

2.7 Chapter Summary

Back to the opening question: what problem does the runtime module solve?

Given a user request and a set of tools, drive the LLM to repeatedly reason and invoke tools until a final answer is produced.

The core algorithm (ReAct) takes just 7 lines, but a production-grade runtime module must satisfy five evaluation criteria:
