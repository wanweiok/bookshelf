8.1 组装模块：为什么需要它？

过去六章你分别构建了六个模块：运行时、记忆、通道、安全、调度、技能。每个模块都能独立运行、独立测试。

然后你把它们放在一起——什么都不工作了。

记忆层需要数据库连接，运行时需要记忆层，技能层需要运行时来注册工具——谁先启动？调度器要调用运行时，但运行时还没注册完工具。渠道要发消息给运行时，但安全层还没就绪。

这就是涌现复杂度：单独能跑的模块，放在一起却不能启动。

组装模块解决的核心问题：

按正确的依赖顺序初始化六层架构，统一管理配置，在生产环境中保持系统稳定运行，并提供可观测性。

8.4 生产级实现：从 10 行到完整系统

C1 依赖解析：谁先启动？

OpenClaw 有 50+ 插件，不能硬编码初始化顺序。它的解决方案是三阶段插件生命周期：

  1. Load——四层优先级扫描（config → workspace → global → bundled）+ manifest 加载
  2. Validate——启用状态检查 + 配置 schema 校验
  3. Register——调用 register(api) 注册 tools/hooks/channels

这本质上是一个依赖注入容器的简化版。

NanoClaw 的整个启动逻辑在 main()（index.ts:456-504）里，固定顺序：容器检查 → initDB → loadState → shutdown 处理器 → 子系统初始化 → recoverPendingMessages。没有插件系统——依赖关系通过代码调用顺序隐式表达。

# OpenClaw 风格 —— 动态组合（运行时决定）
def boot_with_plugins(plugin_dirs):
    manifests = discover(plugin_dirs)       # Load
    validated = validate(manifests)          # Validate
    for plugin in validated:
        plugin.register(shared_services)     # Register

# NanoClaw 风格 —— 静态组合（编译时确定）
def main():
    db = init_database()
    state = load_state(db)
    whatsapp = connect_whatsapp(state)
    start_message_loop(whatsapp)

设计模式就地展开——Dependency Injection（依赖注入）

解决的问题：模块之间硬编码依赖导致无法替换、无法测试。

解决的方法：依赖由外部注入而非内部硬编码。OpenClaw 的插件 register() 接收 api 对象，通过它注入 tools/channels/hooks——插件不需要知道运行时内部结构。

💡 设计法则：组件 < 10 且组合固定 → 简单调用顺序。组件 > 10 或需运行时决定 → 插件框架或 DI 容器。

C2 配置管理：30+ 字段怎么管？

OpenClaw 的配置来自三层：

来源 优先级 示例
环境变量 最高 OPENAI_API_KEY
配置文件 中 JSON/YAML 模型参数
代码默认值 最低 max_iterations=10

TypeBox 做运行时类型校验——加载后立即验证。启动日志打印所有非敏感配置的当前值。

NanoClaw 只需 .env + 少量环境变量。没有类型校验，没有多来源合并——功能集小到不需要。

# 三层配置合并
def load_config(path="config.json"):
    cfg = dict(DEFAULTS)                    # ③ 代码默认值
    if os.path.exists(path):
        cfg.update(json.load(open(path)))   # ② 配置文件覆盖
    for key in os.environ:                  # ① 环境变量最高优先
        if key.startswith("AGENT_"):
            cfg[key[6:].lower()] = os.environ[key]
    return cfg

设计模式就地展开——Builder（构建者）

解决的问题：复杂对象（配置 / 系统）需要分步构造，每步可独立验证。

解决的方法：从 defaults → file → env 三步合并，每步产出中间结果。NanoClaw 的 main() 也是构建者模式——initDB → loadState → WhatsApp → 子系统 → messageLoop，每步成功是下步前置条件。

C3 优雅启停：崩溃后怎么恢复？

OpenClaw：乐观启动 + 运行时降级。 所有插件尽可能启动，失败的标记为 disabled，运行时返回降级响应。结合 launchd/systemd/schtasks 实现跨平台守护进程自动重启。

NanoClaw：严格启动 + 外部重启。 任何一步失败就终止进程。Docker Compose 自动重启失败容器。ensureContainerSystemRunning() 在启动时清理孤儿容器。

# 优雅关闭 —— 逆序释放
async def shutdown(agent):
    log.info("Shutting down...")
    if agent.scheduler: agent.scheduler.stop()    # ⑤ 调度
    if agent.channels: await agent.channels.disconnect_all()  # ⑥ 渠道
    if agent.memory: agent.memory.close()          # ① 记忆
    log.info("Goodbye.")

💡 选择信号：有容器编排 → 严格启动 + 外部重启。单机裸跑 → 应用层降级。

C4 部署策略：单进程还是多容器？

维度 OpenClaw NanoClaw
进程模型 单进程 Node.js 多容器 Docker Compose
最小部署 npx openclaw docker compose up
存储 SQLite + 本地目录 容器卷挂载
故障隔离 无（单点故障） 有（容器独立）
适用场景 个人电脑常驻 服务器 / VPS

OpenClaw 的单进程哲学：所有层在同一 Node.js 进程中运行。resolveGatewayService()（daemon/service.ts:66-155）抽象了跨平台守护进程——macOS LaunchAgent、Linux systemd、Windows Scheduled Task。

NanoClaw 的容器隔离哲学：API、WhatsApp 连接器、代码执行器各自是独立容器。故障隔离——代码执行容器崩了不影响消息接收。

📐 设计法则：个人 Agent → 单进程。团队/多用户 → 容器化。先简单后复杂。

"""mini_agent/main.py — 六层 Agent 组装

第八章 Lab · 启动序列 → 主循环 → 优雅关闭
真正导入并组装前六章的所有模块 — 不是重写，是组装。

运行: python -m mini_agent
依赖: openai（可选）, croniter（可选）
"""
from __future__ import annotations

import asyncio, signal, json, os, logging

from mini_agent.runtime import ReactRuntime, create_llm, MockLLM
from mini_agent.memory import DualMemory
from mini_agent.channels import ChannelManager, CLIChannel, HTTPChannel
from mini_agent.security import Sandbox, ToolPolicy, AuditLog
from mini_agent.scheduler import Scheduler
from mini_agent.skills import SkillRegistry, create_builtin_skills

logging.basicConfig(level=logging.INFO,
                    format="%(asctime)s [%(name)s] %(message)s")
log = logging.getLogger("agent")


DEFAULT_CONFIG = {
    "model": "gpt-4o-mini",
    "memory_db": ":memory:",
    "memory_window": 20,
    "allowed_tools": ["*"],
    "denied_tools": ["run_command", "delete_file"],
    "http_port": None,
    "scheduler_enabled": False,
    "skills_dir": "./skills",
}


def load_config(path: str = "agent_config.json") -> dict:
    cfg = dict(DEFAULT_CONFIG)
    if os.path.exists(path):
        cfg.update(json.loads(open(path, encoding="utf-8").read()))
    for key in ("OPENAI_API_KEY", "DEEPSEEK_API_KEY", "OPENAI_MODEL"):
        val = os.environ.get(key)
        if val:
            cfg[key.lower()] = val
    return cfg


class Agent:
    """六层 Agent — 导入并组装前六章的所有模块

    Boot 顺序（不可变）:
      ① Memory (Ch03) → ② Sandbox (Ch05) → ③ Skills (Ch07)
      → ④ Runtime (Ch02) → ⑤ Scheduler (Ch06) → ⑥ Channel (Ch04)
    """

    def __init__(self, config: dict | None = None):
        self.config = config or load_config()
        self.memory: DualMemory | None = None
        self.sandbox: Sandbox | None = None
        self.skills: SkillRegistry | None = None
        self.runtime: ReactRuntime | None = None
        self.scheduler: Scheduler | None = None
        self.channels: ChannelManager | None = None

    def boot(self) -> "Agent":
        log.info("═══ Boot sequence ═══")

        # ① Memory (Ch03)
        self.memory = DualMemory(
            db_path=self.config["memory_db"],
            window=self.config["memory_window"])
        log.info("① Memory initialized")

        # ② Sandbox (Ch05)
        self.sandbox = Sandbox(
            policy=ToolPolicy(
                allow=self.config["allowed_tools"],
                deny=self.config["denied_tools"]),
            audit=AuditLog("audit.jsonl"))
        log.info("② Sandbox initialized")

        # ③ Skills → Tools (Ch07)
        self.skills = SkillRegistry()
        for skill in create_builtin_skills(self.memory):
            self.skills.register(skill)
        from mini_agent.skills import SkillLoader
        for skill in SkillLoader.discover(self.config["skills_dir"]):
            self.skills.register(skill)
        tools = self.skills.to_registry()
        log.info(f"③ Skills: {[s.name for s in self.skills.list_skills()]}"
                 f" ({len(tools.names())} tools)")

        # ④ Runtime (Ch02)
        llm = create_llm(self.config)
        self.runtime = ReactRuntime(
            llm, tools,
            memory=self.memory,
            sandbox=self.sandbox)
        mode = "Mock" if isinstance(llm, MockLLM) else "LLM"
        log.info(f"④ Runtime initialized ({mode} mode)")

        # ⑤ Scheduler (Ch06)
        if self.config.get("scheduler_enabled"):
            self.scheduler = Scheduler()
            log.info("⑤ Scheduler initialized")

        # ⑥ Channels (Ch04)
        self.channels = ChannelManager()
        self.channels.register(CLIChannel())
        if self.config.get("http_port"):
            self.channels.register(
                HTTPChannel(self.config["http_port"]))
        log.info("⑥ Channels registered")

        log.info("═══ All 6 layers assembled ═══\n")
        return self

    async def run(self) -> None:
        assert self.channels and self.runtime
        await self.channels.connect_all()

        if self.scheduler:
            asyncio.create_task(self.scheduler.start())

        async for msg in self.channels.listen_all():
            log.info(f"[{msg.channel}] {msg.sender}: {msg.content}")

            if msg.content.strip().lower() in ("exit", "quit", "bye"):
                await self.channels.send(
                    msg.channel, msg.sender, "再见！")
                break

            answer = self.runtime.run(msg.content)
            await self.channels.send(msg.channel, msg.sender, answer)

    async def shutdown(self) -> None:
        log.info("Shutting down...")
        if self.scheduler:
            self.scheduler.stop()
        if self.channels:
            await self.channels.disconnect_all()
        if self.memory:
            self.memory.close()
        log.info("Goodbye.")


async def main():
    agent = Agent().boot()
    for sig in (signal.SIGINT, signal.SIGTERM):
        try:
            signal.signal(sig, lambda *_: None)
        except (OSError, ValueError):
            pass
    try:
        await agent.run()
    except (KeyboardInterrupt, EOFError):
        pass
    finally:
        await agent.shutdown()


if __name__ == "__main__":
    asyncio.run(main())

8.1 The Assembly Module: Why Do We Need It?

Over the past six chapters you built six modules independently: Runtime, Memory, Channel, Security, Scheduler, and Skills. Each could run and be tested on its own.

Then you put them together — nothing worked.

The Memory layer needs a database connection, Runtime needs Memory, the Skills layer needs Runtime to register tools — who starts first? The Scheduler needs to call Runtime, but Runtime hasn't finished registering tools. Channels need to send messages to Runtime, but the Security layer isn't ready yet.

This is emergent complexity: modules that run fine alone fail to start when combined.

The core problem the Assembly module solves:

Initialize the six-layer architecture in the correct dependency order, manage configuration uniformly, keep the system running stably in production, and provide observability.

8.4 Production Implementation: From 10 Lines to a Complete System

C1 Dependency Resolution: Who Starts First?

OpenClaw has 50+ plugins — it cannot hard-code the initialization order. Its solution is a three-phase plugin lifecycle:

  1. Load — Four-priority scanning (config → workspace → global → bundled) + manifest loading
  2. Validate — Enable-status check + config schema validation
  3. Register — Call register(api) to register tools/hooks/channels

This is essentially a simplified dependency injection container.

NanoClaw's entire startup logic lives in main() (index.ts:456-504), with a fixed order: container check → initDB → loadState → shutdown handler → subsystem initialization → recoverPendingMessages. No plugin system — dependencies are expressed implicitly through code call order.

# OpenClaw style — dynamic composition (decided at runtime)
def boot_with_plugins(plugin_dirs):
    manifests = discover(plugin_dirs)       # Load
    validated = validate(manifests)          # Validate
    for plugin in validated:
        plugin.register(shared_services)     # Register

# NanoClaw style — static composition (decided at compile time)
def main():
    db = init_database()
    state = load_state(db)
    whatsapp = connect_whatsapp(state)
    start_message_loop(whatsapp)

Design Pattern In-Place — Dependency Injection

Problem it solves: Hard-coded dependencies between modules make them impossible to replace or test.

How it solves it: Dependencies are injected from the outside rather than hard-coded internally. OpenClaw's plugin register() receives an api object through which tools/channels/hooks are injected — the plugin doesn't need to know the runtime's internal structure.

💡 Design Principle: Components < 10 and composition is fixed → simple call order. Components > 10 or composition decided at runtime → plugin framework or DI container.

C2 Configuration Management: How to Manage 30+ Fields?

OpenClaw's configuration comes from three layers:

Source Priority Example
Environment variables Highest OPENAI_API_KEY
Config file Medium JSON/YAML model parameters
Code defaults Lowest max_iterations=10

TypeBox performs runtime type validation — values are verified immediately after loading. Startup logs print the current value of all non-sensitive configuration fields.

NanoClaw needs only .env + a few environment variables. No type validation, no multi-source merging — its feature set is small enough not to need it.

# Three-layer configuration merge
def load_config(path="config.json"):
    cfg = dict(DEFAULTS)                    # ③ Code defaults
    if os.path.exists(path):
        cfg.update(json.load(open(path)))   # ② Config file override
    for key in os.environ:                  # ① Env vars highest priority
        if key.startswith("AGENT_"):
            cfg[key[6:].lower()] = os.environ[key]
    return cfg

Design Pattern In-Place — Builder

Problem it solves: Complex objects (configuration / systems) need step-by-step construction, with each step independently verifiable.

How it solves it: Merge in three steps from defaults → file → env, each producing an intermediate result. NanoClaw's main() is also a Builder pattern — initDB → loadState → WhatsApp → subsystems → messageLoop, where each step's success is a precondition for the next.

C3 Graceful Startup/Shutdown: How to Recover After a Crash?

OpenClaw: Optimistic startup + runtime degradation. All plugins are started as aggressively as possible; failures are marked as disabled, and the runtime returns degraded responses. Combined with launchd/systemd/schtasks for cross-platform daemon auto-restart.

NanoClaw: Strict startup + external restart. Any step failure terminates the process. Docker Compose automatically restarts failed containers. ensureContainerSystemRunning() cleans up orphan containers at startup.

# Graceful shutdown — reverse-order release
async def shutdown(agent):
    log.info("Shutting down...")
    if agent.scheduler: agent.scheduler.stop()    # ⑤ Scheduler
    if agent.channels: await agent.channels.disconnect_all()  # ⑥ Channels
    if agent.memory: agent.memory.close()          # ① Memory
    log.info("Goodbye.")

💡 Selection Signal: Have container orchestration → strict startup + external restart. Running bare-metal → application-layer degradation.

C4 Deployment Strategy: Single Process or Multi-Container?

Dimension OpenClaw NanoClaw
Process model Single-process Node.js Multi-container Docker Compose
Minimum deployment npx openclaw docker compose up
Storage SQLite + local directory Container volume mounts
Fault isolation None (single point of failure) Yes (containers are independent)

OpenClaw's single-process philosophy: All layers run within the same Node.js process. resolveGatewayService() (daemon/service.ts:66-155) abstracts cross-platform daemons — macOS LaunchAgent, Linux systemd, Windows Scheduled Task.

NanoClaw's container-isolation philosophy: API, WhatsApp connector, and code executor are each independent containers. Fault isolation — a crashed code-execution container doesn't affect message reception.

📐 Design Principle: Personal Agent → single process. Team / multi-user → containerized. Start simple, evolve later.

C5 Runtime Monitoring: How Do You Know What the Agent Is Thinking?

"""mini_agent/main.py — Six-Layer Agent Assembly

Chapter 8 Lab · Boot Sequence → Main Loop → Graceful Shutdown
Actually imports and assembles all modules from the previous six chapters — not rewriting, assembling.

Run: python -m mini_agent
Dependencies: openai (optional), croniter (optional)
"""
from __future__ import annotations

import asyncio, signal, json, os, logging

from mini_agent.runtime import ReactRuntime, create_llm, MockLLM
from mini_agent.memory import DualMemory
from mini_agent.channels import ChannelManager, CLIChannel, HTTPChannel
from mini_agent.security import Sandbox, ToolPolicy, AuditLog
from mini_agent.scheduler import Scheduler
from mini_agent.skills import SkillRegistry, create_builtin_skills

logging.basicConfig(level=logging.INFO,
                    format="%(asctime)s [%(name)s] %(message)s")
log = logging.getLogger("agent")


DEFAULT_CONFIG = {
    "model": "gpt-4o-mini",
    "memory_db": ":memory:",
    "memory_window": 20,
    "allowed_tools": ["*"],
    "denied_tools": ["run_command", "delete_file"],
    "http_port": None,
    "scheduler_enabled": False,
    "skills_dir": "./skills",
}


def load_config(path: str = "agent_config.json") -> dict:
    cfg = dict(DEFAULT_CONFIG)
    if os.path.exists(path):
        cfg.update(json.loads(open(path, encoding="utf-8").read()))
    for key in ("OPENAI_API_KEY", "DEEPSEEK_API_KEY", "OPENAI_MODEL"):
        val = os.environ.get(key)
        if val:
            cfg[key.lower()] = val
    return cfg


class Agent:
    """Six-Layer Agent — imports and assembles all modules from the previous six chapters

    Boot order (immutable):
      ① Memory (Ch03) → ② Sandbox (Ch05) → ③ Skills (Ch07)
      → ④ Runtime (Ch02) → ⑤ Scheduler (Ch06) → ⑥ Channel (Ch04)
    """

    def __init__(self, config: dict | None = None):
        self.config = config or load_config()
        self.memory: DualMemory | None = None
        self.sandbox: Sandbox | None = None
        self.skills: SkillRegistry | None = None
        self.runtime: ReactRuntime | None = None
        self.scheduler: Scheduler | None = None
        self.channels: ChannelManager | None = None

    def boot(self) -> "Agent":
        log.info("═══ Boot sequence ═══")

        # ① Memory (Ch03)
        self.memory = DualMemory(
            db_path=self.config["memory_db"],
            window=self.config["memory_window"])
        log.info("① Memory initialized")

        # ② Sandbox (Ch05)
        self.sandbox = Sandbox(
            policy=ToolPolicy(
                allow=self.config["allowed_tools"],
                deny=self.config["denied_tools"]),
            audit=AuditLog("audit.jsonl"))
        log.info("② Sandbox initialized")

        # ③ Skills → Tools (Ch07)
        self.skills = SkillRegistry()
        for skill in create_builtin_skills(self.memory):
            self.skills.register(skill)
        from mini_agent.skills import SkillLoader
        for skill in SkillLoader.discover(self.config["skills_dir"]):
            self.skills.register(skill)
        tools = self.skills.to_registry()
        log.info(f"③ Skills: {[s.name for s in self.skills.list_skills()]}"
                 f" ({len(tools.names())} tools)")

        # ④ Runtime (Ch02)
        llm = create_llm(self.config)
        self.runtime = ReactRuntime(
            llm, tools,
            memory=self.memory,
            sandbox=self.sandbox)
        mode = "Mock" if isinstance(llm, MockLLM) else "LLM"
        log.info(f"④ Runtime initialized ({mode} mode)")

        # ⑤ Scheduler (Ch06)
        if self.config.get("scheduler_enabled"):
            self.scheduler = Scheduler()
            log.info("⑤ Scheduler initialized")

        # ⑥ Channels (Ch04)
        self.channels = ChannelManager()
        self.channels.register(CLIChannel())
        if self.config.get("http_port"):
            self.channels.register(
                HTTPChannel(self.config["http_port"]))
        log.info("⑥ Channels registered")

        log.info("═══ All 6 layers assembled ═══\n")
        return self

    async def run(self) -> None:
        assert self.channels and self.runtime
        await self.channels.connect_all()

        if self.scheduler:
            asyncio.create_task(self.scheduler.start())

        async for msg in self.channels.listen_all():
            log.info(f"[{msg.channel}] {msg.sender}: {msg.content}")

            if msg.content.strip().lower() in ("exit", "quit", "bye"):
                await self.channels.send(
                    msg.channel, msg.sender, "再见！")
                break

            answer = self.runtime.run(msg.content)
            await self.channels.send(msg.channel, msg.sender, answer)

    async def shutdown(self) -> None:
        log.info("Shutting down...")
        if self.scheduler:
            self.scheduler.stop()
        if self.channels:
            await self.channels.disconnect_all()
        if self.memory:
            self.memory.close()
        log.info("Goodbye.")


async def main():
    agent = Agent().boot()
    for sig in (signal.SIGINT, signal.SIGTERM):
        try:
            signal.signal(sig, lambda *_: None)
        except (OSError, ValueError):
            pass
    try:
        await agent.run()
    except (KeyboardInterrupt, EOFError):
        pass
    finally:
        await agent.shutdown()


if __name__ == "__main__":
    asyncio.run(main())

C2 Configuration Management Three-layer merge (env > file > defaults) + validate-and-print at startup, preventing config drift where "it works locally but breaks in production."
C3 Graceful Startup/Shutdown Have container orchestration → strict startup + external restart. Running bare-metal → application-layer degradation. Reverse-order shutdown is a fundamental discipline.
C4 Deployment Strategy Personal → single process (simple and sufficient). Team → containerized (isolated and scalable). Start simple, evolve later.
C5 Runtime Monitoring Every LLM call must record the full prompt/response — this is the only way to trace back in a non-deterministic system.

Where to Go Next

This book covers the core skeleton of a single Agent. The following directions are out of scope but equally important:

  • Multimodal: Voice I/O (Whisper + TTS), image understanding (GPT-4o Vision)
  • Multi-Agent Collaboration: AutoGen / CrewAI / LangGraph
  • Evaluation & Fine-tuning: LMSYS Chatbot Arena, OpenAI Evals
  • Production Deployment: LangSmith observability, vLLM self-hosting, Kubernetes

The book's core belief: An AI Agent's architecture reflects its architect's understanding of what makes a "good assistant." There is no best architecture — only the architecture that best matches your constraints. Start with a 50-line ReAct loop, and let pain points guide you to the next layer.
