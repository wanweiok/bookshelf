4.1 通道模块：为什么需要它？

你在 WhatsApp 群里 @了 Agent："帮我查一下北京天气。"十秒后你收到回复——但不是在 WhatsApp 上，而是在 Telegram 上。你的手机亮了两次，两个 App 上各有半句话。

或者更常见的情况：Agent 的运行时引擎完美运行，记忆系统正确检索，但用户发的消息根本没有到达 Agent——因为 WhatsApp 的 WebSocket 断了，而没有任何重连机制。Agent 在那里等着消息，用户在那里等着回复，双方都以为对方沉默了。

这就是没有通道模块的 Agent——一个困在隔音室里的天才。

通道模块解决的核心问题只有一个：

将来自不同平台（WhatsApp、Telegram、Slack、Web、CLI...）的消息统一为标准格式送达运行时，并将运行时的回复路由回正确的平台和用户。

如果运行时是"神经系统"、记忆是"海马体"，那么通道层就是 Agent 的五官和嘴巴——它决定了 Agent 能"听"到什么、"说"给谁。

4.3 核心算法：stdin/stdout

通道模块的最简实现只需 3 行 Python：

while True:
    user_input = input("You: ")          # stdin 入站
    print(f"Agent: echo {user_input}")   # stdout 出站

对照评价标准打分：

标准 最简实现 得分
C1 渠道抽象 硬编码 stdin/stdout，无接口 ❌ 无法切换渠道
C2 消息路由 无路由，直接 echo ❌ 单用户单 Agent
C3 可靠投递 阻塞式 I/O，无重试 ❌ 断开即丢失
C4 跨进程安全 同一进程内 ❌ 无隔离

5 条全部不及格。接下来看生产项目如何逐条补齐。

C1 渠道抽象：怎么隔离平台差异？

📐 ADR-05: 统一协议层 vs 适配器模式？

  • 背景：15 个渠道，每个协议差异巨大（HTTP/WebSocket/长轮询/OAuth）
  • 决策：适配器模式——每个渠道内部用最合适的原生 SDK，只在边界处归一化为统一 Message
  • 放弃的替代方案：统一协议抽象层（如 Matrix 协议——太重，且无法覆盖所有平台特性）；每个渠道独立写完整逻辑（代码爆炸）
  • 后果：新增渠道只需实现一个接口（~200 行），但每个适配器需要熟悉对应平台 SDK

当你只支持一个渠道时，"抽象"是可选的。当你支持 15 个渠道时，"抽象"是生存必需。

NanoClaw 的做法：6 方法接口，够用就好。 Channel 接口（types.ts:80-93）只有 connect()、sendMessage()、isConnected()、ownsJid()、disconnect() 和可选的 setTyping()。WhatsApp 的所有行为——QR 认证、群组同步、LID 映射——全部硬编码在 WhatsAppChannel 类内部。

OpenClaw 的做法：20+ 可选适配器的组合式接口。 ChannelPlugin（types.plugin.ts:48-84）把渠道能力分解为 config、outbound、gateway、security、groups、mentions、threading、actions 等独立适配器。简单渠道只需实现 3 个核心适配器；功能丰富的渠道可实现全部。

# NanoClaw 风格 —— 6 方法极简接口
class Channel(Protocol):
    def connect(self) -> None: ...
    def send_message(self, to: str, text: str) -> None: ...
    def is_connected(self) -> bool: ...
    def owns_jid(self, jid: str) -> bool: ...
    def disconnect(self) -> None: ...
    def set_typing(self, jid: str, on: bool) -> None: ...   # 可选

# OpenClaw 风格 —— 组合式适配器（只实现你需要的）
class ChannelPlugin:
    config: ConfigAdapter         # 必选
    outbound: OutboundAdapter     # 必选
    gateway: GatewayAdapter       # 必选
    security: SecurityAdapter     # 可选
    groups: GroupsAdapter         # 可选
    mentions: MentionsAdapter     # 可选
    threading: ThreadingAdapter   # 可选
    # ... 20+ 更多可选适配器

设计模式就地展开——Adapter（适配器模式）

mermaid flowchart TB A["ChannelPlugin 接口<br/>统一消息收发协议"] --> B["WhatsApp 适配器<br/>WebSocket + QR 认证"] A --> C["Telegram 适配器<br/>HTTP 长轮询"] A --> D["Discord 适配器<br/>Gateway 事件"] A --> E["Slack 适配器<br/>Events API"] B --> F["Baileys SDK<br/>@whiskeysockets/baileys"] C --> G["grammY SDK"] D --> H["discord.js"] E --> I["Bolt SDK"]

解决的问题：每个平台的 SDK 接口完全不同——Baileys 用 WebSocket 事件，grammY 用 HTTP 长轮询，discord.js 用 Gateway。如果运行时直接调用这些 SDK，每加一个平台就要改运行时代码。

解决的方法：定义统一接口（ChannelPlugin），每个平台的"方言"在适配器内部翻译为"普通话"。运行时只说"普通话"，不关心背后是 WhatsApp 还是 Telegram。OpenClaw 的组合式变体更进一步——不是一个大接口强制全实现，而是多个小接口按需组合。

C2 消息路由：消息应该发给哪个 Agent？

NanoClaw 不需要路由——只有一个 Agent，所有消息都交给同一个 Claude 实例。路由逻辑就是遍历 channels 数组，找到 ownsJid(jid) 返回 true 的那个（router.ts:31-39）。

OpenClaw 的 7 层瀑布路由。resolveAgentRoute()（resolve-route.ts:295-345）从精确到宽泛逐层尝试：

  1. Peer 精确匹配 — 消息来自已绑定的特定用户
  2. Parent Peer — 父级用户匹配
  3. Guild + Roles — 特定服务器 + 特定角色
  4. Guild — 特定服务器
  5. Team — 团队级匹配
  6. Account — 账户级匹配
  7. Channel 通配 — 兜底：默认 Agent

7 层路由的本质是权限模型的倒影：层级越多，多租户能力越强，运维复杂度也成正比增长。

设计模式就地展开——Middleware（中间件模式）

mermaid flowchart LR M(["入站消息"]) --> A["身份验证<br/>检查 Token"] A --> B["消息解析<br/>归一化格式"] B --> C["频率限制<br/>防刷保护"] C --> D["路由分发<br/>7 层瀑布"] D --> E(["Agent 处理"])

解决的问题：入站消息到达 Agent 前需要经过身份验证、格式解析、频率限制、路由分发等多个步骤。如果把所有逻辑写在一个函数里，代码会变成不可维护的巨型条件链。

解决的方法：消息通过一串处理函数依次流过，每个函数（中间件）专注一个职责。Telegram 渠道使用 grammY 框架的中间件链：bot.use(session()) → bot.use(rateLimit()) → bot.on('message', handler)。每个中间件可以读取/修改上下文、调用 next() 传递、或直接返回终止链条。

C3 可靠投递：消息丢了怎么办？

网络不稳定、进程崩溃、平台 API 限流——消息投递失败是常态。

OpenClaw：预写日志（WAL）。 deliverOutboundPayloads()（deliver.ts:184-263）在发送前先写入持久化队列，成功后确认，失败则保留供重试。与数据库事务日志思路一致——先持久化意图，再执行动作。

NanoClaw：内存队列 + 指数退避。 outgoingQueue（whatsapp.ts:29-44）在断连时缓存消息，重连后刷新。scheduleRetry()（group-queue.ts:220-241）对失败采用指数退避：5s → 10s → 20s...（最多重试 5 次）

# 方式 A：WAL 预写日志 —— 进程崩溃不丢消息
def send_reliable(msg):
    wal.write(msg)            # 先持久化
    try:
        platform.send(msg)    # 再发送
        wal.ack(msg)          # 成功确认
    except:
        pass                  # 重启后从 WAL 重投

# 方式 B：内存队列 —— 简单但进程崩溃全丢
queue = []
def send_best_effort(msg):
    try:
        platform.send(msg)
    except:
        queue.append(msg)     # 下次重试
        # 但如果进程崩溃，queue 全丢

NanoClaw 不用 WAL 的原因：入站消息本身持久化在 SQLite 中。即使出站队列丢失，用户再说一遍 @Andy，Agent 会重新处理。在个人助手场景下，"用户驱动的重试"是可接受的。

C4 跨进程安全：进程隔离下怎么通信？

NanoClaw 的 Agent 运行在 Apple Container（Linux VM）中，不能直接调用宿主的 WhatsApp 连接。它选择了一个出人意料的方案：用文件系统作为消息总线。

startIpcWatcher()（ipc.ts:34-56）定期扫描 ipc/{groupFolder}/messages/ 目录。Agent 容器写入 JSON 文件，宿主读取后发送到 WhatsApp，再删除文件。每个群组一个子目录，天然隔离。

# NanoClaw 风格 —— 文件系统即消息总线
# Agent 容器（写入者）
def send_from_container(group_jid, text):
    path = f"ipc/{group_jid}/messages/{uuid4()}.json"
    write_file(path, {"text": text})     # 原子写入

# 宿主进程（读取者）
def poll_ipc():
    for f in listdir("ipc/*/messages/"):
        msg = read_json(f)
        whatsapp.send(msg["text"])
        delete(f)                         # 处理后删除

设计模式就地展开——Event-Driven（事件驱动模式）

mermaid flowchart TB WA["Baileys WebSocket"] -->|"messages.upsert"| H["消息处理器"] WA -->|"connection.update"| R["重连逻辑"] WA -->|"creds.update"| S["凭证持久化"] H --> DB["storeMessage()"] DB --> P["轮询循环"] P --> C["启动容器"]

解决的问题：入站消息的到达时间不可预测，如果用同步轮询会浪费资源或增加延迟。同时，WebSocket 连接状态变化（断连、重连）需要独立处理，不能阻塞消息流。

解决的方法：组件之间通过事件解耦。Baileys 的 sock.ev.on('messages.upsert') 触发入站处理，写入 SQLite；轮询循环定时读取新消息；容器处理后通过 IPC 文件触发出站。三个组件通过 SQLite 和文件系统解耦，互不依赖。

IPC 的安全模型遵循最小权限原则（ipc.ts:75-95）：只有 main 群组的容器可以跨群组发消息，普通群组只能回复自己所属群组。这防止了恶意 prompt 注入导致 Agent 向其他群组发送未授权消息。

OpenClaw 不需要 IPC——Agent 和渠道运行在同一进程中，通过 ChannelManager 直接调用各插件的 outbound.sendText() 方法。延迟最低但隔离最弱。

C5 新渠道扩展：加第二个渠道有多痛？

这是评价渠道架构的终极测试。假设 NanoClaw 明天要支持 Telegram：

  1. 写一个 TelegramChannel 实现 6 方法接口——简单
  2. 修改 router.ts——ownsJid() 假设 JID 是 WhatsApp 格式
  3. 修改 SQLite Schema——messages 表有 WhatsApp 特有字段
  4. 修改触发逻辑——@Andy 在 Telegram 中的解析方式不同
  5. 修改 IPC 授权——当前按 JID 目录隔离

估算：5-8 个文件需要修改，涉及存储层、路由层和 IPC 层。

而 OpenClaw 加一个新渠道？写一个 npm 包实现 ChannelPlugin 接口，注册到插件表。核心代码零修改。

加第 2 个渠道的成本 加第 15 个渠道的成本
NanoClaw 5-8 文件，中等重构 不可行（需完全重新设计）
OpenClaw 1 个 npm 包，零核心修改 同样 1 个 npm 包

但别急着下结论。OpenClaw 的抽象层本身就是代价——650+ 文件、20+ 适配器接口、插件注册表、动态加载机制。如果你只需要 WhatsApp？这 97% 的渠道代码都是为假设的未来买的保险。

YAGNI vs 未来证明——NanoClaw 的作者明确说："我只用 WhatsApp。" OpenClaw 确实从 3 个渠道增长到了 30+。不要为假设的未来编码，但也不要堵死扩展的门。

Prompt 1 — 构建 CLI + HTTP 双渠道

在 mini_agent/channels.py 中实现多渠道通信抽象：

  1. Message 数据类 — 统一消息格式（id, sender, content, channel, timestamp）
  2. Channel 协议 — connect(), disconnect(), listen() (async generator), send(to, content)
  3. CLIChannel — stdin/stdout 适配器
  4. HTTPChannel — POST /message 入站 + GET /poll 出站，用 stdlib http.server + threading
  5. ChannelManager — 注册多渠道、合并消息流、路由回复
  6. 集成第二章的 ReactRuntime：收到消息后用 Agent 处理（而不是简单 echo）

"""mini_agent/channels.py — 多渠道通信抽象层

第四章 Lab · CLI + HTTP 渠道适配器（适配器模式）+ 消息路由

单独运行: python -m mini_agent.channels
依赖: 仅 Python 标准库
"""
from __future__ import annotations

import asyncio, json, sys
from dataclasses import dataclass, field
from datetime import datetime
from http.server import HTTPServer, BaseHTTPRequestHandler
from threading import Thread
from typing import AsyncGenerator, Protocol
from urllib.parse import urlparse, parse_qs
from uuid import uuid4


@dataclass
class Message:
    """所有渠道的标准消息格式"""
    id: str
    sender: str
    content: str
    channel: str
    timestamp: datetime = field(default_factory=datetime.now)


class Channel(Protocol):
    name: str
    async def connect(self) -> None: ...
    async def disconnect(self) -> None: ...
    def listen(self) -> AsyncGenerator[Message, None]: ...
    async def send(self, to: str, content: str) -> None: ...


class CLIChannel:
    name = "cli"

    async def connect(self) -> None:
        print("[CLI] 已连接 — 输入消息后按回车发送")

    async def disconnect(self) -> None:
        print("[CLI] 已断开")

    async def listen(self) -> AsyncGenerator[Message, None]:
        loop = asyncio.get_event_loop()
        while True:
            try:
                line = await loop.run_in_executor(None, sys.stdin.readline)
                if not line:
                    break
                line = line.strip()
                if line:
                    yield Message(id=uuid4().hex[:8], sender="cli-user",
                                  content=line, channel=self.name)
            except (EOFError, KeyboardInterrupt):
                break

    async def send(self, to: str, content: str) -> None:
        print(f"  [Agent → {to}] {content}")


class HTTPChannel:
    name = "http"

    def __init__(self, port: int = 3000):
        self.port = port
        self._server: HTTPServer | None = None
        self._queue: asyncio.Queue[Message] = asyncio.Queue()
        self._outbox: dict[str, list[str]] = {}
        self._loop: asyncio.AbstractEventLoop | None = None

    async def connect(self) -> None:
        self._loop = asyncio.get_event_loop()
        channel_ref = self

        class Handler(BaseHTTPRequestHandler):
            def do_POST(self):
                if self.path == "/message":
                    length = int(self.headers.get("Content-Length", 0))
                    body = json.loads(self.rfile.read(length))
                    msg = Message(
                        id=uuid4().hex[:8], sender=body["from"],
                        content=body["content"], channel="http")
                    asyncio.run_coroutine_threadsafe(
                        channel_ref._queue.put(msg), channel_ref._loop)
                    self.send_response(200)
                    self.send_header("Content-Type", "application/json")
                    self.end_headers()
                    self.wfile.write(
                        json.dumps({"ok": True, "id": msg.id}).encode())
                else:
                    self.send_response(404); self.end_headers()

            def do_GET(self):
                if self.path.startswith("/poll"):
                    uid = parse_qs(
                        urlparse(self.path).query).get("userId", [""])[0]
                    msgs = channel_ref._outbox.pop(uid, [])
                    self.send_response(200)
                    self.send_header("Content-Type", "application/json")
                    self.end_headers()
                    self.wfile.write(json.dumps({"messages": msgs}).encode())
                else:
                    self.send_response(404); self.end_headers()

            def log_message(self, *_): pass

        self._server = HTTPServer(("", self.port), Handler)
        Thread(target=self._server.serve_forever, daemon=True).start()
        print(f"[HTTP] 监听 http://localhost:{self.port}")

    async def disconnect(self) -> None:
        if self._server: self._server.shutdown()

    async def listen(self) -> AsyncGenerator[Message, None]:
        while True:
            msg = await self._queue.get()
            yield msg

    async def send(self, to: str, content: str) -> None:
        self._outbox.setdefault(to, []).append(content)


class ChannelManager:
    def __init__(self):
        self._channels: dict[str, Channel] = {}

    def register(self, ch: Channel) -> None:
        self._channels[ch.name] = ch

    async def connect_all(self) -> None:
        for ch in self._channels.values():
            await ch.connect()

    async def disconnect_all(self) -> None:
        for ch in self._channels.values():
            await ch.disconnect()

    async def listen_all(self) -> AsyncGenerator[Message, None]:
        merged: asyncio.Queue[Message] = asyncio.Queue()
        async def _pipe(ch):
            async for msg in ch.listen():
                await merged.put(msg)
        for ch in self._channels.values():
            asyncio.create_task(_pipe(ch))
        while True:
            yield await merged.get()

    async def send(self, channel: str, to: str, content: str) -> None:
        ch = self._channels.get(channel)
        if ch: await ch.send(to, content)


async def _demo():
    mgr = ChannelManager()
    mgr.register(CLIChannel())
    mgr.register(HTTPChannel(3000))
    await mgr.connect_all()

    print("\nAgent 就绪 — CLI 输入或 HTTP POST 均可")
    print("  curl -X POST http://localhost:3000/message \\")
    print('    -H "Content-Type: application/json" \\')
    print("    -d '{\"from\":\"u1\",\"content\":\"hello\"}'\n")

    async for msg in mgr.listen_all():
        print(f"[{msg.channel}] {msg.sender}: {msg.content}")
        await mgr.send(msg.channel, msg.sender, f"Echo: {msg.content}")


if __name__ == "__main__":
    asyncio.run(_demo())

4.1 The Channel Module: Why Do We Need It?

You @mentioned the Agent in a WhatsApp group: "Check the weather in Beijing for me." Ten seconds later you get a reply — but not on WhatsApp. It comes through on Telegram. Your phone lights up twice, with half a sentence on each app.

Or a more common scenario: the Agent's runtime engine runs perfectly, the memory system retrieves correctly, but the user's message never reaches the Agent — because the WhatsApp WebSocket disconnected and there's no reconnection mechanism. The Agent sits there waiting for a message, the user sits there waiting for a reply, and each assumes the other has gone silent.

This is what an Agent without a channel module looks like — a genius trapped in a soundproof room.

The channel module solves exactly one core problem:

Unify messages from different platforms (WhatsApp, Telegram, Slack, Web, CLI...) into a standard format for the runtime, and route the runtime's replies back to the correct platform and user.

If the runtime is the "nervous system" and memory is the "hippocampus," then the channel layer is the Agent's eyes, ears, and mouth — it determines what the Agent can "hear" and whom it "speaks" to.

C1 Channel Abstraction: How to Isolate Platform Differences?

📐 ADR-05: Unified Protocol Layer vs. Adapter Pattern?

  • Context: 15 channels, each with vastly different protocols (HTTP/WebSocket/long polling/OAuth)
  • Decision: Adapter pattern — each channel uses the most suitable native SDK internally, and normalizes to a unified Message only at the boundary
  • Rejected alternatives: Unified protocol abstraction layer (e.g. Matrix — too heavy, cannot cover all platform features); each channel implementing full logic independently (code explosion)
  • Consequences: Adding a new channel requires implementing one interface (~200 lines), but each adapter must be familiar with the corresponding platform SDK

When you support only one channel, "abstraction" is optional. When you support 15, "abstraction" is a survival necessity.

NanoClaw's approach: a 6-method interface — just enough. The Channel interface (types.ts:80-93) has only connect(), sendMessage(), isConnected(), ownsJid(), disconnect(), and an optional setTyping(). All WhatsApp-specific behavior — QR authentication, group sync, LID mapping — is hardcoded inside the WhatsAppChannel class.

OpenClaw's approach: 20+ optional adapters with a composable interface. ChannelPlugin (types.plugin.ts:48-84) decomposes channel capabilities into independent adapters: config, outbound, gateway, security, groups, mentions, threading, actions, and more. Simple channels only need to implement 3 core adapters; feature-rich channels can implement them all.

# NanoClaw style — 6-method minimal interface
class Channel(Protocol):
    def connect(self) -> None: ...
    def send_message(self, to: str, text: str) -> None: ...
    def is_connected(self) -> bool: ...
    def owns_jid(self, jid: str) -> bool: ...
    def disconnect(self) -> None: ...
    def set_typing(self, jid: str, on: bool) -> None: ...   # optional

# OpenClaw style — composable adapters (implement only what you need)
class ChannelPlugin:
    config: ConfigAdapter         # required
    outbound: OutboundAdapter     # required
    gateway: GatewayAdapter       # required
    security: SecurityAdapter     # optional
    groups: GroupsAdapter         # optional
    mentions: MentionsAdapter     # optional
    threading: ThreadingAdapter   # optional
    # ... 20+ more optional adapters

Design Pattern In-Place — Adapter Pattern

mermaid flowchart TB A["ChannelPlugin Interface<br/>Unified messaging protocol"] --> B["WhatsApp Adapter<br/>WebSocket + QR auth"] A --> C["Telegram Adapter<br/>HTTP long polling"] A --> D["Discord Adapter<br/>Gateway events"] A --> E["Slack Adapter<br/>Events API"] B --> F["Baileys SDK<br/>@whiskeysockets/baileys"] C --> G["grammY SDK"] D --> H["discord.js"] E --> I["Bolt SDK"]

Problem it solves: Each platform's SDK has a completely different interface — Baileys uses WebSocket events, grammY uses HTTP long polling, discord.js uses Gateway. If the runtime called these SDKs directly, every new platform would require changes to the runtime code.

How it solves it: Define a unified interface (ChannelPlugin), and each platform's "dialect" is translated into a "common language" inside its adapter. The runtime speaks only the "common language" and doesn't care whether the backend is WhatsApp or Telegram. OpenClaw's composable variant goes further — instead of one large interface forcing full implementation, multiple small interfaces are composed on demand.

C2 Message Routing: Which Agent Should Receive the Message?

NanoClaw doesn't need routing — there's only one Agent, and all messages go to the same Claude instance. The routing logic simply iterates through the channels array to find the one where ownsJid(jid) returns true (router.ts:31-39).

OpenClaw's 7-layer waterfall routing. resolveAgentRoute() (resolve-route.ts:295-345) tries layers from most specific to most general:

  1. Peer exact match — message is from a specifically bound user
  2. Parent Peer — parent-level user match
  3. Guild + Roles — specific server + specific roles
  4. Guild — specific server
  5. Team — team-level match
  6. Account — account-level match
  7. Channel wildcard — fallback: default Agent

The essence of 7-layer routing is a mirror image of the permission model: the more layers, the stronger the multi-tenancy capability — but operational complexity grows proportionally.

Design Pattern In-Place — Middleware Pattern

mermaid flowchart LR M(["Inbound msg"]) --> A["Auth<br/>check token"] A --> B["Parse<br/>normalize format"] B --> C["Rate limit<br/>anti-flood"] C --> D["Route<br/>7-layer waterfall"] D --> E(["Agent"])

Problem it solves: Inbound messages must go through authentication, format parsing, rate limiting, and route dispatching before reaching the Agent. If all this logic were in a single function, the code would become an unmaintainable giant conditional chain.

How it solves it: Messages flow through a chain of handler functions, each middleware focusing on a single responsibility. The Telegram channel uses the grammY framework's middleware chain: bot.use(session()) → bot.use(rateLimit()) → bot.on('message', handler). Each middleware can read/modify the context, call next() to pass through, or return directly to terminate the chain.

C3 Reliable Delivery: What If Messages Get Lost?

Network instability, process crashes, platform API throttling — delivery failures are the norm, not the exception.

OpenClaw: Write-Ahead Log (WAL). deliverOutboundPayloads() (deliver.ts:184-263) writes to a persistent queue before sending, acknowledges on success, and retains on failure for retry. The approach mirrors database transaction logs — persist the intent first, then execute the action.

NanoClaw: In-memory queue + exponential backoff. outgoingQueue (whatsapp.ts:29-44) buffers messages during disconnection and flushes on reconnect. scheduleRetry() (group-queue.ts:220-241) applies exponential backoff to failures: 5s → 10s → 20s... (max 5 retries)

# Approach A: WAL Write-Ahead Log — no message loss on process crash
def send_reliable(msg):
    wal.write(msg)            # persist first
    try:
        platform.send(msg)    # then send
        wal.ack(msg)          # acknowledge on success
    except:
        pass                  # retry from WAL after restart

# Approach B: In-memory queue — simple but loses everything on crash
queue = []
def send_best_effort(msg):
    try:
        platform.send(msg)
    except:
        queue.append(msg)     # retry next time
        # but if the process crashes, queue is gone

Why NanoClaw doesn't use WAL: inbound messages are already persisted in SQLite. Even if the outbound queue is lost, the user can simply say @Andy again and the Agent will reprocess. In a personal assistant scenario, "user-driven retry" is acceptable.

C4 Cross-Process Security: How to Communicate Under Process Isolation?

NanoClaw's Agent runs inside an Apple Container (Linux VM) and cannot directly access the host's WhatsApp connection. It chose a surprisingly simple approach: using the file system as a message bus.

startIpcWatcher() (ipc.ts:34-56) periodically scans the ipc/{groupFolder}/messages/ directory. The Agent container writes JSON files, the host reads and sends them to WhatsApp, then deletes the files. Each group gets its own subdirectory — natural isolation.

# NanoClaw style — file system as message bus
# Agent container (writer)
def send_from_container(group_jid, text):
    path = f"ipc/{group_jid}/messages/{uuid4()}.json"
    write_file(path, {"text": text})     # atomic write

# Host process (reader)
def poll_ipc():
    for f in listdir("ipc/*/messages/"):
        msg = read_json(f)
        whatsapp.send(msg["text"])
        delete(f)                         # delete after processing

Design Pattern In-Place — Event-Driven Pattern

mermaid flowchart TB WA["Baileys WebSocket"] -->|"messages.upsert"| H["消息处理器"] WA -->|"connection.update"| R["重连逻辑"] WA -->|"creds.update"| S["凭证持久化"] H --> DB["storeMessage()"] DB --> P["轮询循环"] P --> C["启动容器"]

Problem it solves: Inbound messages arrive at unpredictable times. Synchronous polling would waste resources or increase latency. Meanwhile, WebSocket connection state changes (disconnect, reconnect) need independent handling without blocking message flow.

How it solves it: Components are decoupled through events. Baileys' sock.ev.on('messages.upsert') triggers inbound processing and writes to SQLite; a polling loop periodically reads new messages; the container processes them and triggers outbound via IPC files. The three components are decoupled through SQLite and the file system, with no direct dependencies.

The IPC security model follows the principle of least privilege (ipc.ts:75-95): only the main group's container can send cross-group messages; regular groups can only reply to their own group. This prevents malicious prompt injection from causing the Agent to send unauthorized messages to other groups.

OpenClaw doesn't need IPC — the Agent and channels run in the same process, communicating through ChannelManager directly calling each plugin's outbound.sendText() method. Lowest latency but weakest isolation.

C5 Channel Extensibility: How Painful Is Adding a Second Channel?

This is the ultimate test of channel architecture. Suppose NanoClaw needs to support Telegram tomorrow:

  1. Write a TelegramChannel implementing the 6-method interface — straightforward
  2. Modify router.ts — ownsJid() assumes WhatsApp JID format
  3. Modify SQLite schema — the messages table has WhatsApp-specific fields
  4. Modify trigger logic — @Andy is parsed differently in Telegram
  5. Modify IPC authorization — currently isolated by JID directory

Estimate: 5–8 files need modification, spanning the storage layer, routing layer, and IPC layer.

And adding a new channel to OpenClaw? Write an npm package implementing the ChannelPlugin interface and register it in the plugin table. Zero modifications to core code.

Cost of adding the 2nd channel Cost of adding the 15th channel
NanoClaw 5–8 files, moderate refactoring Infeasible (requires complete redesign)
OpenClaw 1 npm package, zero core changes Still just 1 npm package

But don't rush to conclusions. OpenClaw's abstraction layer itself is the cost — 650+ files, 20+ adapter interfaces, plugin registry, dynamic loading mechanism. If you only need WhatsApp? That 97% of channel code is insurance bought for a hypothetical future.

YAGNI vs Future-Proofing — NanoClaw's author explicitly said: "I only use WhatsApp." OpenClaw did grow from 3 channels to 30+. Don't code for a hypothetical future, but don't seal the door to extension either.

"""mini_agent/channels.py — 多渠道通信抽象层

第四章 Lab · CLI + HTTP 渠道适配器（适配器模式）+ 消息路由

单独运行: python -m mini_agent.channels
依赖: 仅 Python 标准库
"""
from __future__ import annotations

import asyncio, json, sys
from dataclasses import dataclass, field
from datetime import datetime
from http.server import HTTPServer, BaseHTTPRequestHandler
from threading import Thread
from typing import AsyncGenerator, Protocol
from urllib.parse import urlparse, parse_qs
from uuid import uuid4


@dataclass
class Message:
    """所有渠道的标准消息格式"""
    id: str
    sender: str
    content: str
    channel: str
    timestamp: datetime = field(default_factory=datetime.now)


class Channel(Protocol):
    name: str
    async def connect(self) -> None: ...
    async def disconnect(self) -> None: ...
    def listen(self) -> AsyncGenerator[Message, None]: ...
    async def send(self, to: str, content: str) -> None: ...


class CLIChannel:
    name = "cli"

    async def connect(self) -> None:
        print("[CLI] 已连接 — 输入消息后按回车发送")

    async def disconnect(self) -> None:
        print("[CLI] 已断开")

    async def listen(self) -> AsyncGenerator[Message, None]:
        loop = asyncio.get_event_loop()
        while True:
            try:
                line = await loop.run_in_executor(None, sys.stdin.readline)
                if not line:
                    break
                line = line.strip()
                if line:
                    yield Message(id=uuid4().hex[:8], sender="cli-user",
                                  content=line, channel=self.name)
            except (EOFError, KeyboardInterrupt):
                break

    async def send(self, to: str, content: str) -> None:
        print(f"  [Agent → {to}] {content}")


class HTTPChannel:
    name = "http"

    def __init__(self, port: int = 3000):
        self.port = port
        self._server: HTTPServer | None = None
        self._queue: asyncio.Queue[Message] = asyncio.Queue()
        self._outbox: dict[str, list[str]] = {}
        self._loop: asyncio.AbstractEventLoop | None = None

    async def connect(self) -> None:
        self._loop = asyncio.get_event_loop()
        channel_ref = self

        class Handler(BaseHTTPRequestHandler):
            def do_POST(self):
                if self.path == "/message":
                    length = int(self.headers.get("Content-Length", 0))
                    body = json.loads(self.rfile.read(length))
                    msg = Message(
                        id=uuid4().hex[:8], sender=body["from"],
                        content=body["content"], channel="http")
                    asyncio.run_coroutine_threadsafe(
                        channel_ref._queue.put(msg), channel_ref._loop)
                    self.send_response(200)
                    self.send_header("Content-Type", "application/json")
                    self.end_headers()
                    self.wfile.write(
                        json.dumps({"ok": True, "id": msg.id}).encode())
                else:
                    self.send_response(404); self.end_headers()

            def do_GET(self):
                if self.path.startswith("/poll"):
                    uid = parse_qs(
                        urlparse(self.path).query).get("userId", [""])[0]
                    msgs = channel_ref._outbox.pop(uid, [])
                    self.send_response(200)
                    self.send_header("Content-Type", "application/json")
                    self.end_headers()
                    self.wfile.write(json.dumps({"messages": msgs}).encode())
                else:
                    self.send_response(404); self.end_headers()

            def log_message(self, *_): pass

        self._server = HTTPServer(("", self.port), Handler)
        Thread(target=self._server.serve_forever, daemon=True).start()
        print(f"[HTTP] 监听 http://localhost:{self.port}")

    async def disconnect(self) -> None:
        if self._server: self._server.shutdown()

    async def listen(self) -> AsyncGenerator[Message, None]:
        while True:
            msg = await self._queue.get()
            yield msg

    async def send(self, to: str, content: str) -> None:
        self._outbox.setdefault(to, []).append(content)


class ChannelManager:
    def __init__(self):
        self._channels: dict[str, Channel] = {}

    def register(self, ch: Channel) -> None:
        self._channels[ch.name] = ch

    async def connect_all(self) -> None:
        for ch in self._channels.values():
            await ch.connect()

    async def disconnect_all(self) -> None:
        for ch in self._channels.values():
            await ch.disconnect()

    async def listen_all(self) -> AsyncGenerator[Message, None]:
        merged: asyncio.Queue[Message] = asyncio.Queue()
        async def _pipe(ch):
            async for msg in ch.listen():
                await merged.put(msg)
        for ch in self._channels.values():
            asyncio.create_task(_pipe(ch))
        while True:
            yield await merged.get()

    async def send(self, channel: str, to: str, content: str) -> None:
        ch = self._channels.get(channel)
        if ch: await ch.send(to, content)


async def _demo():
    mgr = ChannelManager()
    mgr.register(CLIChannel())
    mgr.register(HTTPChannel(3000))
    await mgr.connect_all()

    print("\nAgent 就绪 — CLI 输入或 HTTP POST 均可")
    print("  curl -X POST http://localhost:3000/message \\")
    print('    -H "Content-Type: application/json" \\')
    print("    -d '{\"from\":\"u1\",\"content\":\"hello\"}'\n")

    async for msg in mgr.listen_all():
        print(f"[{msg.channel}] {msg.sender}: {msg.content}")
        await mgr.send(msg.channel, msg.sender, f"Echo: {msg.content}")


if __name__ == "__main__":
    asyncio.run(_demo())

4.7 Chapter Summary

Criterion Core Insight
C1 Channel Abstraction The Adapter pattern translates platform "dialects" into a unified "common language." OpenClaw's composable variant lets each channel implement only the adapter subset it needs. Abstraction isn't free — but the cost of lacking abstraction is only revealed when you need to extend.
C2 Message Routing Routing layers = multi-tenancy capability. A single Agent needs no routing; multiple Agents need a 7-layer waterfall. Routing is essentially a mirror image of the permission model.
C3 Reliable Delivery WAL (persist intent → execute → acknowledge) is the enterprise-grade approach; in-memory queue + user-driven retry is the personal-grade approach. The Middleware pattern allows cross-cutting concerns like logging and rate limiting without intruding on core logic.
C4 Cross-Process Security "File system as message bus" seems primitive but simultaneously solves persistence and isolation. The principle of least privilege prevents prompt injection from escalating across groups.
C5 Channel Extensibility Extensibility cost is proportional to the number of channels supported. 1 channel = 6-method interface; 30 channels = plugin architecture + registry + dynamic loading. Both YAGNI and future-proofing have merit — the key is knowing where you sit on the spectrum.
