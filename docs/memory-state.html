3.1 记忆模块：为什么需要它？

你对 Agent 说："我叫 Alice，喜欢深色主题，请记住。"它回复："好的，记住了。"第二天你回来问："我叫什么名字？"它回答——

"抱歉，我无法获取你的个人信息。你能告诉我你的名字吗？"

一切从零开始。昨天的对话、你告诉它的偏好、你们讨论过的项目方案——全部消失了。

这就是没有记忆模块的 Agent——每次对话都是第一次见面。用户不得不反复自我介绍、重复需求、重新建立上下文。这不是 bug，而是架构缺失。

记忆模块解决的核心问题只有一个：

在有限的上下文窗口中，为 Agent 提供跨对话的、可检索的、结构化的记忆能力。

如果把 Agent 比作一个人，运行时是神经系统（第二章），那么记忆就是大脑中的海马体——它决定了哪些经历值得记住、如何组织这些记忆、以及在需要时如何回忆起来。

如何判断一个记忆模块的好坏？我们定义五条评价标准：

标准 问题 理想状态
C1 容量效率 上下文窗口满了怎么办？ 在有限 token 内最大化有用信息密度
C2 检索精度 用户说"我上次提到的那个项目"，能找到吗？ 语义级别的精确检索，不只是关键字匹配

后续每一节都将围绕这五条标准展开。先从最简实现开始。

3.3 核心算法：messages 列表

记忆模块的最简实现只需 5 行 Python：

messages = []

def add(role, content):
    messages.append({"role": role, "content": content})

def context():
    return messages[-20:]   # 滑动窗口，只保留最近 20 条

这就是大多数 ChatGPT Demo 的做法——一个全局列表，末尾截断。对照我们的评价标准打分：

标准 最简实现 得分
C1 容量效率 固定窗口截断，丢弃早期消息 ❌ 丢弃 = 遗忘
C2 检索精度 无检索能力，只有最近 20 条 ❌ 找不到旧记忆
C3 主动整理 无主动性，被动追加 ❌ 不会整理
C4 结构化共存 全部混在一个列表里 ❌ 无区分
C5 持久化安全 内存变量，重启即丢失 ❌ 不持久

5 条全部不及格。功能上"能用"——但距离生产级还有巨大差距。接下来我们看生产项目如何逐条补齐。

3.4 生产级实现：从 5 行到 3000+ 行

C1 容量效率：窗口满了怎么办？

📐 ADR-04: 上下文溢出时截断还是摘要？

  • 背景：LLM 上下文窗口有限（GPT-4 128K / Claude 200K），对话持续增长必然溢出
  • 决策：Letta 选择 LLM 摘要压缩（用 AI 总结旧消息），OpenClaw 选择 TTL + 会话上限
  • 放弃的替代方案：简单截断（丢失重要上下文）；RAG 外挂（延迟高、相关性不稳定）
  • 后果：摘要压缩保留了更多信息但每次消耗额外 LLM 调用；TTL 方案零成本但可能丢失长期上下文

对话越来越长，token 逼近上限。最简实现的做法是"截断丢弃"。生产项目有三种策略：

Letta 的做法——LLM 摘要压缩。 agent.py:433 有一行关键检查：if response.usage.total_tokens > self.agent_state.llm_config.context_window。一旦触发，调用 summarize_messages_inplace()（agent.py:1107）——取出早期消息，用 LLM 生成摘要，用摘要替换原始消息释放空间。

此外，agent_alerted_about_memory_pressure 标志位（agent.py:157）会在压力上升时提前警告 LLM 主动整理 Core Memory。这让 Agent 在窗口真正溢出之前就有机会做准备。

OpenClaw 的做法——会话 TTL 过期。 不压缩单次对话，而是在会话粒度管理生命周期。store.ts:231 定义了三个常量：30 天过期、500 条上限、10MB 文件轮转。pruneStaleEntries()（store.ts:301）定期清理过期会话。

NanoClaw 的做法——依赖 SDK 窗口。 从 db.ts:429 的 getSession() 获取会话 ID，再由 container-runner.ts 传给 Claude Agent SDK 的 resume 选项，上下文管理完全交给 Claude 自己。CLAUDE.md 一般不超过几千字，远低于 200K 窗口限制。

三种策略的本质区别是谁来承担压缩的认知成本：

# 策略 A：LLM 摘要 —— 质量高但费 token
summary = llm.summarize(old_messages)      # 额外一次 LLM 调用
messages = [system, summary] + recent[-3:]  # 压缩后上下文

# 策略 B：TTL 淘汰 —— 简单但可能丢重要记忆
if session.age > 30_days:
    delete(session)                         # 过期就删，不管内容

# 策略 C：依赖 SDK —— 最省事但最不可控
sdk.resume(session_id)                      # "不是我的问题"

C2 检索精度：怎么找到相关的记忆？

用户说"我上次提到的那个项目"，三个系统分别怎么找到它？

Letta 给 Agent 两把钥匙：conversation_search(query) 在对话历史中搜索，archival_memory_search(query) 在归档记忆中做向量检索。两者都作为工具暴露给 LLM——Agent 自己决定什么时候需要"回忆"。

OpenClaw 构建了一个混合搜索引擎。search() 方法（manager.ts:198）并行执行向量搜索和 FTS5 全文搜索，通过 mergeHybridResults()（hybrid.ts:41）用可配置权重（默认 vector 0.7 / text 0.3）融合排序。.catch(() => []) 保证任一搜索后端故障不影响另一个。

NanoClaw 不搜索——它全读。Agent 的工作目录设置为 groups/{group-name}/，SDK 自动加载当前目录和父目录的 CLAUDE.md。所有记忆一次性进入上下文，靠 Claude 的长窗口理解力来"找到"相关信息。

这里有一个根本性的权衡——搜索精度 vs 方案复杂度：

# 方案 A：混合搜索 —— 精确但复杂
vec_results = vector_search(query, weight=0.7)   # 语义："编程偏好" → "喜欢 Python"
fts_results = fts5_search(query, weight=0.3)     # 精确："Alice" → "Alice"
return merge_and_rank(vec_results, fts_results)   # 需要嵌入模型 + 向量索引

# 方案 B：全量加载 —— 零复杂度但不可扩展
context = read_file("CLAUDE.md")                  # 直接读，不搜索
# 用户说"编程偏好"时，能找到"喜欢 Python"吗？看 LLM 心情

设计模式就地展开——Composite（组合模式）

OpenClaw 的混合搜索是 Composite 模式的经典应用：

解决的问题：单一搜索策略总有盲区——向量搜索擅长语义关联（"编程偏好" → "喜欢 Python"），但精确匹配差；关键字搜索精确但没有语义理解。

解决的方法：将多个搜索策略组合成一个统一接口，结果加权融合。调用方不知道背后有几个引擎，只看到一个 search() 方法。权重可配置（0.7/0.3），允许不同场景调整偏好。

C3 主动整理：Agent 能自己决定记什么？

这是三种方案分歧最大的地方——Agent 在记忆管理中的自主权。

Letta 的答案：完全自主。 core_memory_append() 和 core_memory_replace()（memory.py:379-412）是标准工具函数，LLM 像调用天气 API 一样调用它们。ChatMemory（memory.py:415）默认创建 persona 和 human 两个 Block，Agent 自己决定往里写什么。更精妙的是，_render_memory_blocks_standard()（memory.py:110）在渲染时会暴露 chars_current/chars_limit 给 LLM，让 Agent 感知容量压力并主动整理。

OpenClaw 的答案：半自动。 两条路径——手动路径：Agent 调用 memory_store 工具显式存储。自动路径：shouldCapture()（memory-lancedb/index.ts:197）用正则规则判断对话是否值得记忆——匹配 remember、prefer、always 等触发词，且长度在 10-500 字符之间才捕获。

NanoClaw 的答案：Agent 直接写文件。 全局 CLAUDE.md 中的"元记忆指令"教 Agent 如何管理记忆——创建 customers.md、preferences.md 等文件，超过 500 行就拆分成文件夹。这是一种自引用设计：记忆管理的规则本身也存储在记忆中。

# Letta 风格 —— Agent 调工具写记忆（最自主）
agent.call_tool("core_memory_append", {
    "label": "human",
    "content": "Alice 偏好深色主题"
})

# OpenClaw 风格 —— 规则自动捕获（半自动）
if re.match(r"remember|prefer|always", user_msg):
    memory.store(user_msg, category="preference")

# NanoClaw 风格 —— Agent 直接编辑文件（最自由也最危险）
agent.write_file("CLAUDE.md", "## 用户偏好\n- 深色主题\n- 中文")

⚠️ 风险提示：自主权越高，风险越大。Letta 的完全自主意味着 LLM 可能写入垃圾记忆。OpenClaw 的正则触发太简单，可能漏掉"我下个月要搬到上海"。NanoClaw 的一次错误 write 可能覆盖整个 CLAUDE.md。

C4 结构化共存：精确数据和模糊印象怎么共存？

"用户叫 Alice，邮箱 alice@example.com"是结构化的。"她好像不太喜欢周一开会"是模糊的。系统怎么同时处理？

Letta 用 Block 分区。 每个 BaseBlock（block.py:13），Block 继承自它（block.py:82），有 label、value 和 limit。结构化信息和模糊记忆共存于同一个 Block 内，完全由 Agent 自己组织格式。灵活——但没有 Schema 保障。

OpenClaw 用 SQLite Schema 分层。 核心表 chunks 存储文本块和嵌入向量，files 表追踪文件元数据，FTS5 虚拟表索引文本。LanceDB 插件的 detectCategory() 自动分类为 preference（偏好）、decision（决策）、entity（实体，如联系方式）、fact（事实）等预定义类别。结构化和模糊记忆在不同的表/字段中共存。

NanoClaw 用目录结构物理隔离。 SQLite 只负责运行时状态（chats、messages、sessions），语义记忆完全在文件系统：CLAUDE.md 是自由文本。

设计模式就地展开——Singleton（单例模式）

解决的问题：OpenClaw 的每个 Agent 有自己的工作区，如果每次搜索都重建索引管理器，会重复加载向量索引、浪费内存、拖慢响应。

解决的方法：MemoryIndexManager.get() 按 agentId + workspace 组合键缓存实例——同一个 Agent 的同一个工作区永远返回同一个索引管理器，避免重复建索引和内存浪费。

NanoClaw 的"状态 vs 记忆"物理分离是最干净的架构——SQLite 升级不影响记忆文件，记忆迁移不碰数据库。Letta 最灵活但最不可预测。OpenClaw 最工程化但最复杂。

C5 持久化安全：记忆丢了怎么办？

重启后记忆还在吗？向量搜索后端挂了怎么办？

Letta 将所有记忆持久化到 PostgreSQL/SQLite，通过 update_memory_if_changed()（agent.py:200-234）在每轮对话后同步变更。数据库事务保证原子性。

OpenClaw 将会话记录存为 JSONL 文件（session-files.ts:21-33），索引存在 SQLite 中。文件 + 数据库双保险，任一损坏可从另一个恢复。

NanoClaw 的记忆就是文件系统中的 .md 文件——天然持久化，而且 Git 友好，可以版本控制。SQLite 只存路由状态（db.ts:10-75），丢了也不影响核心记忆。

设计模式就地展开——Fallback（降级模式）

解决的问题：OpenClaw 支持外部搜索服务（QMD），但外部服务可能宕机、超时、返回错误。如果搜索完全不可用，Agent 就"失忆"了。

解决的方法：FallbackMemoryManager（search-manager.ts:67-202）先尝试 QMD 外部服务，失败时无缝降级到内置 SQLite 向量搜索。用户无感知——只是结果可能略微不精确。这确保了记忆服务的可用性优先于精确性。

Prompt 1 — 构建双层记忆系统

在 mini_agent/memory.py 中实现双层 Agent 记忆系统：

  1. MemoryBlock — 带容量限制的可编辑文本块（Letta 风格），支持 append()、replace()、render()

"""mini_agent/memory.py — 双层 Agent 记忆系统

第三章 Lab · 短期记忆（Letta 风格 Block）+ 长期记忆（SQLite + FTS5 混合搜索）

单独运行: python -m mini_agent.memory
依赖: 仅 Python 标准库（+ 可选 openai 用于真实嵌入）
"""
from __future__ import annotations

import json, sqlite3, hashlib, math, os
from dataclasses import dataclass, field


# ── 短期记忆：Letta 风格可编辑 Block ─────────────────────

@dataclass
class MemoryBlock:
    """带容量限制的可编辑文本块 — 活在 LLM 上下文窗口内"""
    label: str
    value: str = ""
    limit: int = 2000
    description: str = ""

    def append(self, content: str) -> None:
        new = f"{self.value}\n{content}" if self.value else content
        if len(new) > self.limit:
            raise ValueError(f"[{self.label}] 溢出: {len(new)}/{self.limit}")
        self.value = new

    def replace(self, old: str, new: str) -> None:
        if old not in self.value:
            raise ValueError(f"'{old}' 不在 [{self.label}] 中")
        result = self.value.replace(old, new)
        if len(result) > self.limit:
            raise ValueError(f"[{self.label}] 替换后超限")
        self.value = result

    @property
    def usage(self) -> float:
        return len(self.value) / self.limit if self.limit else 0.0

    def render(self) -> str:
        return (f"<{self.label}>\n"
                f"  <description>{self.description}</description>\n"
                f"  <usage>{len(self.value)}/{self.limit} ({self.usage:.0%})</usage>\n"
                f"  <value>\n{self.value}\n  </value>\n"
                f"</{self.label}>")


class ShortTermMemory:
    """管理多个 Block + 最近对话消息的滑动窗口"""

    def __init__(self, window: int = 20):
        self.blocks: dict[str, MemoryBlock] = {}
        self._messages: list[dict] = []
        self.window = window

    def add_block(self, block: MemoryBlock) -> None:
        self.blocks[block.label] = block

    def get(self, label: str) -> MemoryBlock:
        return self.blocks[label]

    def add_message(self, role: str, content: str) -> None:
        self._messages.append({"role": role, "content": content})
        self._messages = self._messages[-self.window:]

    def messages(self) -> list[dict]:
        return list(self._messages)

    def compile(self) -> str:
        parts = ["<memory>"]
        for b in self.blocks.values():
            parts.append(b.render())
        parts.append("</memory>")
        return "\n".join(parts)


# ── 长期记忆：SQLite + 伪向量 + FTS5 混合搜索 ─────────────

def _char_vector(text: str) -> list[float]:
    """26 维字符频率向量 + L2 归一化（教学用伪向量，生产中换真嵌入模型）"""
    vec = [0.0] * 26
    for ch in text.lower():
        if 'a' <= ch <= 'z':
            vec[ord(ch) - ord('a')] += 1
    mag = math.sqrt(sum(v * v for v in vec))
    return [v / mag for v in vec] if mag else vec


def _cosine_sim(a: list[float], b: list[float]) -> float:
    return max(0.0, min(1.0, sum(x * y for x, y in zip(a, b))))


class LongTermMemory:
    """SQLite 后端 + 向量相似度 + FTS5 全文检索
    融合权重: vector=0.7, text=0.3（模仿 OpenClaw mergeHybridResults）"""

    def __init__(self, db_path: str = ":memory:"):
        self.conn = sqlite3.connect(db_path)
        self.conn.executescript("""
            CREATE TABLE IF NOT EXISTS memories (
                id TEXT PRIMARY KEY, text TEXT, category TEXT DEFAULT 'general',
                embedding TEXT, created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP);
            CREATE VIRTUAL TABLE IF NOT EXISTS memories_fts
                USING fts5(text, id UNINDEXED);
        """)

    def store(self, text: str, category: str = "general") -> str:
        mid = hashlib.md5(text.encode()).hexdigest()[:12]
        emb = json.dumps(_char_vector(text))
        self.conn.execute(
            "INSERT OR REPLACE INTO memories VALUES (?,?,?,?,datetime('now'))",
            (mid, text, category, emb))
        self.conn.execute(
            "INSERT OR REPLACE INTO memories_fts VALUES (?,?)", (mid, text))
        self.conn.commit()
        return mid

    def search(self, query: str, top_k: int = 5) -> list[dict]:
        qv = _char_vector(query)
        vec_hits: dict[str, dict] = {}
        for rid, txt, cat, emb_j in self.conn.execute(
                "SELECT id, text, category, embedding FROM memories"):
            s = _cosine_sim(qv, json.loads(emb_j))
            if s >= 0.3:
                vec_hits[rid] = {"id": rid, "text": txt,
                                 "category": cat, "v_score": s}
        fts_hits: dict[str, float] = {}
        try:
            for rid, rank in self.conn.execute(
                    "SELECT id, rank FROM memories_fts WHERE text MATCH ?",
                    (query,)):
                fts_hits[rid] = 1.0 / (1.0 + abs(rank))
        except Exception:
            pass
        merged: dict[str, dict] = {}
        for rid, h in vec_hits.items():
            merged[rid] = {**h, "score": 0.7 * h["v_score"]
                           + 0.3 * fts_hits.get(rid, 0)}
        for rid, fs in fts_hits.items():
            if rid not in merged:
                row = self.conn.execute(
                    "SELECT text, category FROM memories WHERE id=?",
                    (rid,)).fetchone()
                if row:
                    merged[rid] = {"id": rid, "text": row[0],
                                   "category": row[1],
                                   "v_score": 0, "score": 0.3 * fs}
        return sorted(merged.values(),
                      key=lambda x: x["score"], reverse=True)[:top_k]

    def close(self):
        self.conn.close()


# ── 整合层：双层记忆 + 溢出管理 ───────────────────────────

class DualMemory:
    """统一接口 — 短期 Block + 长期 SQLite + 溢出自动归档

    提供给 ReactRuntime 的鸭子类型接口:
      add(role, content)   — 记录消息
      context() -> list    — 返回上下文消息列表
    """

    def __init__(self, db_path: str = ":memory:", window: int = 20):
        self.short = ShortTermMemory(window)
        self.long = LongTermMemory(db_path)
        self.short.add_block(
            MemoryBlock("persona", "我是 AI 助手。", 2000, "Agent 身份"))
        self.short.add_block(
            MemoryBlock("human", "", 2000, "用户信息"))

    def add(self, role: str, content: str) -> None:
        self.short.add_message(role, content)

    def context(self) -> list[dict]:
        return self.short.messages()

    def remember(self, label: str, content: str) -> None:
        self.short.get(label).append(content)

    def archive(self, text: str, category: str = "general") -> str:
        return self.long.store(text, category)

    def recall(self, query: str, top_k: int = 5) -> list[dict]:
        return self.long.search(query, top_k)

    def compile_blocks(self) -> str:
        return self.short.compile()

    def check_overflow(self, label: str, threshold: float = 0.8) -> bool:
        block = self.short.get(label)
        if block.usage <= threshold:
            return False
        self.long.store(block.value, f"archived_{label}")
        lines = block.value.split("\n")
        keep = max(1, len(lines) // 5)
        block.value = "[早期内容已归档]\n" + "\n".join(lines[-keep:])
        return True

    def close(self):
        self.long.close()

3.1 The Memory Module: Why Do We Need It?

You tell the Agent: "My name is Alice, I like dark themes, please remember that." It replies: "Got it, I'll remember." The next day you come back and ask: "What's my name?" It answers:

"Sorry, I can't access your personal information. Could you tell me your name?"

Everything starts from scratch. Yesterday's conversation, the preferences you shared, the project plan you discussed. All gone.

This is what an Agent without a memory module looks like. Every conversation is a first meeting. Users must repeatedly introduce themselves, restate their needs, and rebuild context. The cause is a missing architectural component.

The memory module solves one core problem:

Within a limited context window, provide the Agent with cross-conversation, retrievable, structured memory capabilities.

If we compare an Agent to a human, the runtime is the nervous system (Chapter 2), while memory is the hippocampus. It determines which experiences are worth remembering, how to organize those memories, and how to recall them when needed.

3.2 Module Specification: Inputs, Outputs, Interfaces, Evaluation Criteria

The Memory Module's Position in the Six-Layer Architecture

The arrows in the diagram represent data flow. The memory module is the runtime's primary data supplier: before each LLM call, the runtime retrieves context from the memory module; after each conversation, the runtime writes new messages back to memory. The skill layer uses tool calls (such as remember, archive) to let the Agent actively operate on memory. The security layer determines who has permission to read and write which memories (critical in multi-tenant scenarios).

Evaluation Criteria

How do we judge the quality of a memory module? We define five evaluation criteria:

Criterion Question Ideal State
C1 Capacity Efficiency What happens when the context window is full? Maximize useful information density within limited tokens
C2 Retrieval Precision When the user says "that project I mentioned last time," can it be found? Semantic-level precise retrieval, not just keyword matching
C3 Active Curation Can the Agent decide what to remember and what to forget? The Agent proactively summarizes, organizes, and prunes memories like a human
C4 Structured Coexistence How do "name=Alice" and "she doesn't seem to like Java" coexist? Precise data and fuzzy impressions each have optimal storage and retrieval paths
C5 Persistence Safety Are memories still there after a restart? What if the search backend goes down? No data loss + graceful degradation when services fail

Every subsequent section will revolve around these five criteria. Let's start with the minimal implementation.

3.3 Core Algorithm: The Messages List

The simplest implementation of a memory module requires only 5 lines of Python:

messages = []

def add(role, content):
    messages.append({"role": role, "content": content})

def context():
    return messages[-20:]   # 滑动窗口，只保留最近 20 条

This is how most ChatGPT demos work: a global list, truncated at the tail. Scoring against our evaluation criteria:

Criterion Minimal Implementation Score
C1 Capacity Efficiency Fixed window truncation, discards early messages ❌ Discard = forget
C2 Retrieval Precision No retrieval capability, only the last 20 messages ❌ Can't find old memories
C3 Active Curation No proactivity, passive appending only ❌ No curation
C4 Structured Coexistence Everything mixed in a single list ❌ No differentiation
C5 Persistence Safety In-memory variable, lost on restart ❌ Not persistent

All 5 criteria fail. Functionally it "works" — but the gap to production-grade is enormous. Let's see how production projects close each gap.

C1 Capacity Efficiency: What Happens When the Window Is Full?

📐 ADR-04: Truncate or Summarize on Context Overflow?

  • Context: LLM context windows are finite (GPT-4 128K / Claude 200K), so ongoing conversations will eventually overflow
  • Decision: Letta chooses LLM summary compression (AI summarizes old messages); OpenClaw chooses TTL + session limits
  • Rejected alternatives: Simple truncation (loses critical context); external RAG (high latency, unstable relevance)
  • Consequences: Summary compression preserves more information but incurs extra LLM calls each time; the TTL approach has zero cost but may lose long-term context

As conversations grow longer, tokens approach the limit. The minimal implementation's approach is "truncate and discard." Production projects employ three strategies:

Letta's approach — LLM summary compression. agent.py:433 contains a critical check: if response.usage.total_tokens > self.agent_state.llm_config.context_window. Once triggered, it calls summarize_messages_inplace() (agent.py:1107), extracting early messages, generating a summary via LLM, and replacing original messages with the summary to free up space.

Additionally, the agent_alerted_about_memory_pressure flag (agent.py:157) warns the LLM in advance to proactively curate Core Memory when pressure rises. This gives the Agent an opportunity to prepare before the window actually overflows.

OpenClaw's approach: Session TTL expiration. Instead of compressing individual conversations, it manages lifecycles at the session granularity. store.ts:231 defines three constants: 30-day expiration, 500-entry limit, 10MB file rotation. pruneStaleEntries() (store.ts:301) periodically cleans up expired sessions.

NanoClaw's approach: Relying on SDK window. It retrieves the session ID via getSession() (db.ts:429), which container-runner.ts then passes to Claude Agent SDK's resume option, delegating context management entirely to Claude itself. CLAUDE.md typically doesn't exceed a few thousand characters, well below the 200K window limit.

The fundamental difference between the three strategies is who bears the cognitive cost of compression:

# 策略 A：LLM 摘要 —— 质量高但费 token
summary = llm.summarize(old_messages)      # 额外一次 LLM 调用
messages = [system, summary] + recent[-3:]  # 压缩后上下文

# 策略 B：TTL 淘汰 —— 简单但可能丢重要记忆
if session.age > 30_days:
    delete(session)                         # 过期就删，不管内容

# 策略 C：依赖 SDK —— 最省事但最不可控
sdk.resume(session_id)                      # "不是我的问题"

C2 Retrieval Precision: How Do You Find Relevant Memories?

When the user says "that project I mentioned last time," how does each system find it?

Letta gives the Agent two keys: conversation_search(query) searches conversation history, while archival_memory_search(query) performs vector retrieval on archived memories. Both are exposed to the LLM as tools. The Agent decides when it needs to "recall."

OpenClaw builds a hybrid search engine. The search() method (manager.ts:198) executes vector search and FTS5 full-text search in parallel, then merges results through mergeHybridResults() (hybrid.ts:41) with configurable weights (default vector 0.7 / text 0.3). .catch(() => []) ensures that a failure in one search backend doesn't affect the other.

NanoClaw doesn't search — it loads everything. The Agent's working directory is set to groups/{group-name}/, and the SDK automatically loads CLAUDE.md from the current directory and parent directories. All memories enter the context at once, relying on Claude's long-window comprehension to "find" the relevant information.

Here lies a fundamental trade-off, retrieval precision vs. solution complexity:

# 方案 A：混合搜索 —— 精确但复杂
vec_results = vector_search(query, weight=0.7)   # 语义："编程偏好" → "喜欢 Python"
fts_results = fts5_search(query, weight=0.3)     # 精确："Alice" → "Alice"
return merge_and_rank(vec_results, fts_results)   # 需要嵌入模型 + 向量索引

# 方案 B：全量加载 —— 零复杂度但不可扩展
context = read_file("CLAUDE.md")                  # 直接读，不搜索
# 用户说"编程偏好"时，能找到"喜欢 Python"吗？看 LLM 心情

Design Pattern In-Place: Composite Pattern

OpenClaw's hybrid search is a classic application of the Composite pattern:

The problem it solves: Any single search strategy has blind spots. Vector search excels at semantic associations ("programming preferences" → "likes Python") but is poor at exact matching; keyword search is precise but lacks semantic understanding.

How it solves it: Multiple search strategies are combined into a unified interface, with results merged by weighted fusion. The caller doesn't know how many engines are working behind the scenes. It only sees a single search() method. Weights are configurable (0.7/0.3), allowing different scenarios to tune preferences.

C3 Active Curation: Can the Agent Decide What to Remember?

This is where the three approaches diverge the most. The core question: how much autonomy does the Agent have in memory management?

Letta's answer: Full autonomy. core_memory_append() and core_memory_replace() (memory.py:379-412) are standard tool functions that the LLM calls just like a weather API. ChatMemory (memory.py:415) creates persona and human blocks by default, and the Agent decides what to write in them. More elegantly, _render_memory_blocks_standard() (memory.py:110) exposes chars_current/chars_limit to the LLM when rendering, allowing the Agent to perceive capacity pressure and proactively curate.

OpenClaw's answer: Semi-automatic. Two paths — Manual path: the Agent calls the memory_store tool to explicitly store memories. Automatic path: shouldCapture() (memory-lancedb/index.ts:197) uses regex rules to determine if a conversation is worth remembering, matching trigger words like remember, prefer, always, and capturing only if the content is between 10-500 characters.

NanoClaw's answer: The Agent writes files directly. "Meta-memory instructions" in the global CLAUDE.md teach the Agent how to manage memory — creating files like customers.md, preferences.md, and splitting into folders when content exceeds 500 lines. This is a self-referential design: the rules for memory management are themselves stored in memory.

# Letta 风格 —— Agent 调工具写记忆（最自主）
agent.call_tool("core_memory_append", {
    "label": "human",
    "content": "Alice 偏好深色主题"
})

# OpenClaw 风格 —— 规则自动捕获（半自动）
if re.match(r"remember|prefer|always", user_msg):
    memory.store(user_msg, category="preference")

# NanoClaw 风格 —— Agent 直接编辑文件（最自由也最危险）
agent.write_file("CLAUDE.md", "## 用户偏好\n- 深色主题\n- 中文")

⚠️ Risk Warning: The greater the autonomy, the greater the risk. Letta's full autonomy means the LLM may write garbage into memory. OpenClaw's regex triggers are too simple and may miss "I'm moving to Shanghai next month." NanoClaw's one bad write could overwrite the entire CLAUDE.md.

C4 Structured Coexistence: How Do Precise Data and Fuzzy Impressions Coexist?

"The user is named Alice, email alice@example.com" is structured. "She doesn't seem to like Monday meetings" is fuzzy. How does the system handle both?

Letta uses Block partitioning. Each BaseBlock (block.py:13), with Block inheriting from it (block.py:82), has a label, value, and limit. Structured information and fuzzy memories coexist within the same Block, organized entirely by the Agent itself. Flexible, but with no schema guarantees.

OpenClaw uses SQLite Schema layering. The core chunks table stores text blocks and embedding vectors, the files table tracks file metadata, and the FTS5 virtual table indexes text. The LanceDB plugin's detectCategory() automatically classifies entries into predefined categories: preference, decision, entity, and fact. Structured and fuzzy memories coexist in different tables/fields.

NanoClaw uses directory structure for physical isolation. SQLite only handles runtime state (chats, messages, sessions), while semantic memory lives entirely in the filesystem: CLAUDE.md is free-form text.

Design Pattern In-Place: Singleton Pattern

The problem it solves: Each Agent in OpenClaw has its own workspace. If the index manager were rebuilt on every search, it would redundantly load vector indices, waste memory, and slow down responses.

How it solves it: MemoryIndexManager.get() caches instances by a composite key of agentId + workspace. The same Agent's same workspace always returns the same index manager, avoiding redundant index building and memory waste.

NanoClaw's "state vs. memory" physical separation is the cleanest architecture. SQLite upgrades don't affect memory files, and memory migration doesn't touch the database. Letta is the most flexible but the least predictable. OpenClaw is the most engineered but the most complex.

C5 Persistence Safety: What If Memories Are Lost?

Are memories still there after a restart? What happens if the vector search backend goes down?

Letta persists all memories to PostgreSQL/SQLite, syncing changes after each conversation turn via update_memory_if_changed() (agent.py:200-234). Database transactions guarantee atomicity.

OpenClaw stores session records as JSONL files (session-files.ts:21-33), with indexes in SQLite. The file + database dual backup means either can recover from the other's corruption.

NanoClaw's memories are simply .md files in the filesystem — naturally persistent, and Git-friendly for version control. SQLite only stores routing state (db.ts:10-75), and losing it doesn't affect core memories.

Design Pattern In-Place: Fallback Pattern

The problem it solves: OpenClaw supports external search services (QMD), but external services can crash, time out, or return errors. If search becomes completely unavailable, the Agent effectively "loses its memory."

How it solves it: FallbackMemoryManager (search-manager.ts:67-202) first attempts the QMD external service, then seamlessly degrades to the built-in SQLite vector search on failure. The user doesn't notice; results may just be slightly less precise. This ensures that memory service availability takes priority over precision.

"""mini_agent/memory.py — 双层 Agent 记忆系统

第三章 Lab · 短期记忆（Letta 风格 Block）+ 长期记忆（SQLite + FTS5 混合搜索）

单独运行: python -m mini_agent.memory
依赖: 仅 Python 标准库（+ 可选 openai 用于真实嵌入）
"""
from __future__ import annotations

import json, sqlite3, hashlib, math, os
from dataclasses import dataclass, field


# ── 短期记忆：Letta 风格可编辑 Block ─────────────────────

@dataclass
class MemoryBlock:
    """带容量限制的可编辑文本块 — 活在 LLM 上下文窗口内"""
    label: str
    value: str = ""
    limit: int = 2000
    description: str = ""

    def append(self, content: str) -> None:
        new = f"{self.value}\n{content}" if self.value else content
        if len(new) > self.limit:
            raise ValueError(f"[{self.label}] 溢出: {len(new)}/{self.limit}")
        self.value = new

    def replace(self, old: str, new: str) -> None:
        if old not in self.value:
            raise ValueError(f"'{old}' 不在 [{self.label}] 中")
        result = self.value.replace(old, new)
        if len(result) > self.limit:
            raise ValueError(f"[{self.label}] 替换后超限")
        self.value = result

    @property
    def usage(self) -> float:
        return len(self.value) / self.limit if self.limit else 0.0

    def render(self) -> str:
        return (f"<{self.label}>\n"
                f"  <description>{self.description}</description>\n"
                f"  <usage>{len(self.value)}/{self.limit} ({self.usage:.0%})</usage>\n"
                f"  <value>\n{self.value}\n  </value>\n"
                f"</{self.label}>")


class ShortTermMemory:
    """管理多个 Block + 最近对话消息的滑动窗口"""

    def __init__(self, window: int = 20):
        self.blocks: dict[str, MemoryBlock] = {}
        self._messages: list[dict] = []
        self.window = window

    def add_block(self, block: MemoryBlock) -> None:
        self.blocks[block.label] = block

    def get(self, label: str) -> MemoryBlock:
        return self.blocks[label]

    def add_message(self, role: str, content: str) -> None:
        self._messages.append({"role": role, "content": content})
        self._messages = self._messages[-self.window:]

    def messages(self) -> list[dict]:
        return list(self._messages)

    def compile(self) -> str:
        parts = ["<memory>"]
        for b in self.blocks.values():
            parts.append(b.render())
        parts.append("</memory>")
        return "\n".join(parts)


# ── 长期记忆：SQLite + 伪向量 + FTS5 混合搜索 ─────────────

def _char_vector(text: str) -> list[float]:
    """26 维字符频率向量 + L2 归一化（教学用伪向量，生产中换真嵌入模型）"""
    vec = [0.0] * 26
    for ch in text.lower():
        if 'a' <= ch <= 'z':
            vec[ord(ch) - ord('a')] += 1
    mag = math.sqrt(sum(v * v for v in vec))
    return [v / mag for v in vec] if mag else vec


def _cosine_sim(a: list[float], b: list[float]) -> float:
    return max(0.0, min(1.0, sum(x * y for x, y in zip(a, b))))


class LongTermMemory:
    """SQLite 后端 + 向量相似度 + FTS5 全文检索
    融合权重: vector=0.7, text=0.3（模仿 OpenClaw mergeHybridResults）"""

    def __init__(self, db_path: str = ":memory:"):
        self.conn = sqlite3.connect(db_path)
        self.conn.executescript("""
            CREATE TABLE IF NOT EXISTS memories (
                id TEXT PRIMARY KEY, text TEXT, category TEXT DEFAULT 'general',
                embedding TEXT, created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP);
            CREATE VIRTUAL TABLE IF NOT EXISTS memories_fts
                USING fts5(text, id UNINDEXED);
        """)

    def store(self, text: str, category: str = "general") -> str:
        mid = hashlib.md5(text.encode()).hexdigest()[:12]
        emb = json.dumps(_char_vector(text))
        self.conn.execute(
            "INSERT OR REPLACE INTO memories VALUES (?,?,?,?,datetime('now'))",
            (mid, text, category, emb))
        self.conn.execute(
            "INSERT OR REPLACE INTO memories_fts VALUES (?,?)", (mid, text))
        self.conn.commit()
        return mid

    def search(self, query: str, top_k: int = 5) -> list[dict]:
        qv = _char_vector(query)
        vec_hits: dict[str, dict] = {}
        for rid, txt, cat, emb_j in self.conn.execute(
                "SELECT id, text, category, embedding FROM memories"):
            s = _cosine_sim(qv, json.loads(emb_j))
            if s >= 0.3:
                vec_hits[rid] = {"id": rid, "text": txt,
                                 "category": cat, "v_score": s}
        fts_hits: dict[str, float] = {}
        try:
            for rid, rank in self.conn.execute(
                    "SELECT id, rank FROM memories_fts WHERE text MATCH ?",
                    (query,)):
                fts_hits[rid] = 1.0 / (1.0 + abs(rank))
        except Exception:
            pass
        merged: dict[str, dict] = {}
        for rid, h in vec_hits.items():
            merged[rid] = {**h, "score": 0.7 * h["v_score"]
                           + 0.3 * fts_hits.get(rid, 0)}
        for rid, fs in fts_hits.items():
            if rid not in merged:
                row = self.conn.execute(
                    "SELECT text, category FROM memories WHERE id=?",
                    (rid,)).fetchone()
                if row:
                    merged[rid] = {"id": rid, "text": row[0],
                                   "category": row[1],
                                   "v_score": 0, "score": 0.3 * fs}
        return sorted(merged.values(),
                      key=lambda x: x["score"], reverse=True)[:top_k]

    def close(self):
        self.conn.close()


# ── 整合层：双层记忆 + 溢出管理 ───────────────────────────

class DualMemory:
    """统一接口 — 短期 Block + 长期 SQLite + 溢出自动归档

    提供给 ReactRuntime 的鸭子类型接口:
      add(role, content)   — 记录消息
      context() -> list    — 返回上下文消息列表
    """

    def __init__(self, db_path: str = ":memory:", window: int = 20):
        self.short = ShortTermMemory(window)
        self.long = LongTermMemory(db_path)
        self.short.add_block(
            MemoryBlock("persona", "我是 AI 助手。", 2000, "Agent 身份"))
        self.short.add_block(
            MemoryBlock("human", "", 2000, "用户信息"))

    def add(self, role: str, content: str) -> None:
        self.short.add_message(role, content)

    def context(self) -> list[dict]:
        return self.short.messages()

    def remember(self, label: str, content: str) -> None:
        self.short.get(label).append(content)

    def archive(self, text: str, category: str = "general") -> str:
        return self.long.store(text, category)

    def recall(self, query: str, top_k: int = 5) -> list[dict]:
        return self.long.search(query, top_k)

    def compile_blocks(self) -> str:
        return self.short.compile()

    def check_overflow(self, label: str, threshold: float = 0.8) -> bool:
        block = self.short.get(label)
        if block.usage <= threshold:
            return False
        self.long.store(block.value, f"archived_{label}")
        lines = block.value.split("\n")
        keep = max(1, len(lines) // 5)
        block.value = "[早期内容已归档]\n" + "\n".join(lines[-keep:])
        return True

    def close(self):
        self.long.close()

C1 Capacity Efficiency LLM summarization (Letta) is high-quality but costs tokens; TTL eviction (OpenClaw) is simple but may lose critical memories; relying on SDK (NanoClaw) is the easiest but least controllable. Your compression strategy reflects your judgment about "which memories can be discarded."
C2 Retrieval Precision Hybrid search (vector + full-text) offers the highest precision but the most complexity; full loading is the simplest but doesn't scale. The Composite pattern enables multiple search engines to work together with tunable weights.
C3 Active Curation Agent self-management vs. rule-based auto-capture vs. direct file editing. The greater the autonomy, the greater the potential and the greater the risk.
C4 Structured Coexistence Block partitioning (Letta), multi-table schema (OpenClaw), directory isolation (NanoClaw). Three ways to make two types of memory coexist. The Singleton pattern ensures indexes aren't redundantly created.
C5 Persistence Safety The Fallback pattern ensures search service degradation doesn't affect availability. NanoClaw's file-based approach is naturally Git-friendly. All approaches have scale boundaries. Knowing the boundary matters more than choosing the approach.
