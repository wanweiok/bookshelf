1.1 一条消息的一生

早上八点，你在 WhatsApp 上发了一句："今天东京天气怎么样？"

五秒后，你收到一条图文并茂的回复：温度 18°C，晴，适合穿单衣，下午 3 点可能有短暂阵雨。

这五秒里发生了什么？你的消息穿越了六层架构、经过了 34 个模块中的 12 个、触发了两次 LLM 调用和一次工具执行。让我们跟着它走一遍。

一条消息，六层接力：

阶段 架构层 做了什么 对应章节
入站 通道层 WhatsApp Baileys SDK 接收 → 归一化为统一 Message 格式 → 路由到正确的会话 第四章
准备 记忆层 加载最近 20 条对话 + 用户偏好（"中文回复"），拼入 system prompt 第三章
推理 运行时 ReAct（Reasoning + Acting）循环：LLM 判断需要查天气 → 发出 tool_call 第二章
拦截 安全层 检查 get_weather 是否在允许列表 → 参数校验 → 记审计日志 第五章
执行 技能层 从注册表找到 get_weather 插件 → 调用天气 API → 返回结果 第七章
出站 通道层 回复文本按 WhatsApp 字数限制自动分块 → 适配器推送 第四章
(后台) 调度层 检查是否有定时任务（如"每天 9 点推送新闻"）待执行 第六章

注意两个关键的解耦点：

  1. 入站归一化——通道层把 WhatsApp 的原始消息格式转换为统一的 Message 对象。从这一刻起，运行时完全不知道消息来自哪个平台。这意味着换成 Telegram、Slack、甚至 CLI，后面的全部逻辑零修改。

  2. 工具注册表——运行时通过名字查找工具，不关心工具的实现细节。get_weather 可以是一个 HTTP API 调用，也可以是一个本地 Python 函数，对运行时来说完全透明。

这就是 OpenClaw 能支持 15+ 渠道、50+ 工具、4 种 LLM 提供商而核心逻辑不变的原因。

💡 全书线索：后续每一章深入一层，都会回到"消息的一生"这条线上，展示该层在真实消息流中的角色。

1.2 规模感：数字里的 OpenClaw

跟着消息走了一遍，你可能好奇：这个系统到底有多大？

这是一个中型 SaaS 后端的规模——不是 demo，但也没大到需要专门团队维护构建系统。34 个模块意味着架构师做了认真的关注点分离，这是我们后续能"逐层拆解"的基础。

但最让人意外的是代码量的分布：

入口+配置和安全占据了近七成代码，而"智能"的部分（运行时 + 记忆）只占不到三成。这揭示了一个重要事实：造一个能用的 Agent 很容易，造一个安全、可配置、可运维的 Agent 才是真正的工程挑战。 这就是为什么本书不只讲 LLM 调用——六层中的每一层都有真实的工程复杂度。

⚠️ 数字基于分析时的版本快照（v2026.2.13）。OpenClaw 持续演进，但核心架构保持稳定。

1.3 六层架构：消息经过的每一层

§ 1.1 用时间线展示了消息如何穿过系统。现在换一个视角——空间结构：六层之间的依赖关系和数据流向。

读法：自顶向下是层级关系——用户通过接入层进入，引擎处理请求，能力层提供工具执行。箭头方向是数据流。

对照 § 1.1 的消息旅程：消息从接入层（通道）进入，经过引擎（记忆 + 运行时）处理，必要时经能力层（安全 + 技能）执行工具，最后回到接入层出站。调度层是特殊的——它不在消息流上，而是在时间轴上触发运行时（"每天 9 点推新闻"）。

这六层不是理论模型——它直接映射序章中个人助理的六组需求，并且可以在 OpenClaw 的实际代码结构中一一对应。

1.4 代码地图：34 个模块住在哪里

OpenClaw 的 src/ 下有 34 个一级模块。每个模块属于六层中的哪一层？

层（章节） 模块 一句话职责
Ch2 运行时 agents/ · auto-reply/ · openai/ · anthropic/ ReAct 循环、回复流水线、LLM 调用
Ch3 记忆 memory/ · media/ 会话索引、媒体存储
Ch4 通道 channels/ · telegram/ · web/ · slack/ · discord/ · signal/ · line/ · 8 more 注册表 + 14 个渠道适配器（彼此无依赖）
Ch5 安全 gateway/ · infra/ · security/ WebSocket 网关、TLS、认证、出站投递
Ch6 调度 cron/ · hooks/ · daemon/ 定时任务、事件钩子、后台进程
Ch7 技能 plugins/ · plugin-sdk/ 插件注册、第三方 SDK 类型
入口+配置 cli/ · commands/ · config/ · tui/ · wizard/ · logging/ · process/ 启动入口、Zod 配置、日志、进程管理

两个值得留意的设计：

  • config/ 是地基——被几乎所有模块依赖。OpenClaw 用 Zod 做运行时 schema 验证，配置错误在启动时就暴露，不会带到运行时。

  • 14 个通道模块彼此完全解耦——它们只通过 channels/ 注册表统一管理。新增一个渠道就是新增一个目录，不需要碰任何现有代码。这就是 § 1.1 里"入站归一化"解耦点在代码层面的体现。

📐 ADR-01: 为什么接 4 个 LLM 提供商？

  • 背景：生产环境中单一提供商是不可接受的风险——OpenAI 限流时整个系统瘫痪
  • 决策：多提供商 + 自动 failover
  • 放弃的替代方案：只接 OpenAI（简单但脆弱）；用 LiteLLM 统一代理（多一层抽象 + 延迟）
  • 后果：需要维护 4 套 API 适配代码，但换来了生产级可靠性

通道 SDK

平台 SDK 协议特点

WhatsApp Baileys Web 协议，WebSocket 长连接
Slack Bolt Events API + Web API
Discord discord.js Gateway WebSocket
LINE · Lark · Signal 各平台官方 SDK 协议各异

每个平台协议差异巨大——Telegram 是 HTTP 请求-响应，WhatsApp 是 WebSocket 长连接，Slack 用 Events API + OAuth。试图用统一协议层抹平差异是不现实的。务实做法：通道内用最合适的 SDK，通道之上做抽象。这就是第四章"渠道适配器模式"的来源。

存储

技术 用途 选型理由

YAML 配置文件 可读性好，支持注释

📐 ADR-02: 为什么是 SQLite 而不是 PostgreSQL？

  • 背景：OpenClaw 的哲学是 Local-first——运行在个人电脑上，npx openclaw 一条命令启动
  • 决策：SQLite + sqlite-vec（单文件完成关系查询 + 向量搜索，零运维）
  • 放弃的替代方案：PostgreSQL + pgvector（功能更强但需要额外进程）；纯文件存储（无法做复杂查询）
  • 后果：并发写入受限（单写锁），但对单用户个人助理完全够用。这个选型直接影响了第三章记忆层的设计

1.1 One Message's Lifetime

At eight in the morning, you send a message on WhatsApp: "What's the weather in Tokyo today?"

Five seconds later, you receive a rich reply: 18°C, sunny, suitable for light clothing, with a brief afternoon shower possible around 3 PM.

What happened in those five seconds? Your message traversed six architectural layers, passed through 12 of 34 modules, and triggered two LLM calls and one tool execution. Let's follow it through.

One message, six layers in relay:

Stage Layer What Happens Chapter
Inbound Channel Layer WhatsApp Baileys SDK receives → normalizes to unified Message format → routes to correct session Chapter 4
Prepare Memory Layer Load last 20 messages + user preference ("Chinese reply"), merge into system prompt Chapter 3
Reasoning Runtime ReAct (Reasoning + Acting) loop: LLM decides to query weather → emits tool_call Chapter 2
Intercept Security Layer Check whether get_weather is on allowlist → parameter validation → record audit log Chapter 5
Execute Skills Layer Find get_weather plugin in registry → call weather API → return result Chapter 7
Outbound Channel Layer Reply text auto-chunked for WhatsApp limits → adapter delivers Chapter 4
(background) Scheduler Layer Check for pending scheduled tasks (e.g., "push news at 9 AM daily") Chapter 6

Two key decoupling points:

  1. Inbound normalization. The channel layer converts raw WhatsApp message format into a unified Message object. From that moment on, the runtime has no idea which platform the message came from. Switching to Telegram, Slack, or even CLI requires zero changes to downstream logic.

  2. Tool registry. The runtime looks up tools by name; it doesn't care about implementation details. get_weather can be an HTTP API call or a local Python function, completely transparent to the runtime.

That's why OpenClaw supports 15+ channels, 50+ tools, and 4 LLM providers while core logic stays unchanged.

💡 Thread through the book: Every later chapter digs into one layer and traces back to "one message's lifetime," showing that layer's role in the real message flow.

1.2 Scale: OpenClaw in Numbers

After following a message through the system, you might wonder: how big is this thing?

This is mid-sized SaaS backend scale, not a demo, but not large enough to need a dedicated build team. 34 modules means the architect practiced serious separation of concerns; that's the foundation for "dissecting layer by layer."

The surprising part is how code is distributed:

Entry+Config and Security account for nearly 70% of the code, while the "intelligent" parts (runtime + memory) are under 30%. That reveals an important fact: building a usable Agent is easy; building one that is secure, configurable, and operable is the real engineering challenge. That's why this book doesn't stop at LLM calls. Every one of the six layers has real engineering complexity.

⚠️ Numbers are based on an analysis-time snapshot (v2026.2.13). OpenClaw evolves continuously, but core architecture remains stable.

1.3 Six-Layer Architecture: Every Layer the Message Passes Through

§ 1.1 showed how a message flows through the system on a timeline. Now switch perspective: spatial structure: the dependency relationships and data flow between the six layers.

How to read it: Top-to-bottom is the layer hierarchy. Users enter through the access layer, the engine processes requests, and the capability layer executes tools. Arrows indicate data flow.

Compare with § 1.1's message journey: the message enters from the access layer (channel), is processed by the engine (memory + runtime), and when needed goes through the capability layer (security + skills) for tool execution, then returns to the access layer for outbound delivery. The scheduler is special. It sits off the message path and triggers the runtime on the time axis (e.g., "push news at 9 AM daily").

These six layers are not a theoretical model. They map directly to the six groups of personal-assistant requirements from the preface, and correspond one-to-one to OpenClaw's real code structure.

1.4 Code Map: Where 34 Modules Live

Under OpenClaw's src/ there are 34 top-level modules. Which layer does each belong to?

Layer (Chapter) Modules One-Line Responsibility
Ch2 Runtime agents/ · auto-reply/ · openai/ · anthropic/ ReAct loop, reply pipeline, LLM calls
Ch3 Memory memory/ · media/ Session index, media storage
Ch4 Channels channels/ · telegram/ · web/ · slack/ · discord/ · signal/ · line/ · 8 more Registry + 14 channel adapters (zero inter-dependencies)

Two design choices worth noting:

  • config/ is the foundation, depended on by almost every module. OpenClaw uses Zod for runtime schema validation; config errors surface at startup, never at runtime.

  • The 14 channel modules are fully decoupled from each other. They're managed only through the channels/ registry. Adding a new channel means adding a new directory; no changes to existing code. That's the "inbound normalization" decoupling point from § 1.1, embodied in code.

📐 ADR-01: Why Four LLM Providers?

  • Context: A single provider in production is unacceptable risk. When OpenAI rate-limits, the whole system goes down
  • Decision: Multiple providers + automatic failover
  • Alternatives rejected: OpenAI only (simple but fragile); LiteLLM as unified proxy (extra abstraction + latency)
  • Consequences: Need to maintain four API adapter implementations, but gain production-grade reliability

Channel SDKs

Platform SDK Protocol Characteristics
Telegram grammY Bot API, HTTP polling / Webhook
WhatsApp Baileys Web protocol, WebSocket long connection
Slack Bolt Events API + Web API
Discord discord.js Gateway WebSocket
LINE · Lark · Signal Platform official SDKs Protocols vary widely

Protocol differences across platforms are large. Telegram uses HTTP request-response, WhatsApp uses WebSocket long connections, Slack uses Events API + OAuth. Trying to flatten them with a unified protocol layer is unrealistic. The pragmatic approach: use the best SDK within each channel, abstract above it. That's the origin of Chapter 4's "channel adapter pattern."

📐 ADR-02: Why SQLite Instead of PostgreSQL?

  • Context: OpenClaw's philosophy is Local-first: run on a personal machine, npx openclaw starts it with one command
  • Decision: SQLite + sqlite-vec (single file for relational queries + vector search, zero ops)
  • Alternatives rejected: PostgreSQL + pgvector (more capable but needs an extra process); pure file storage (no complex queries)
  • Consequences: Concurrency limited (single-writer lock), but enough for a single-user personal assistant. This choice directly shapes the memory layer design in Chapter 3
