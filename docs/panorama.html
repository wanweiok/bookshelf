1.5 消息的一生：一条 WhatsApp 消息的完整旅程

用户在 WhatsApp 上发了一句"今天东京天气怎么样？"——这条消息在 OpenClaw 内部经历了什么？

flowchart TD subgraph 入站 ["入站（接收 + 预处理）"] direction LR A(["用户发消息"]) --> B["接收 + 归一化 + 路由"] --> C["防抖 + 模型选择 + 加载历史"] end subgraph 执行 ["Agent 执行（ReAct 循环）"] direction LR D{"ReAct 循环"} -->|"需要工具"| E["工具调用"] -->|"结果"| D end subgraph 出站 ["出站（适配 + 投递）"] direction LR F["分块 + 平台适配"] --> G(["回复送达"]) end C --> D D -->|"最终答案"| F
阶段 模块 发生了什么
入站 web/ · channels/ 适配器接收消息，归一化为统一格式（不管来自哪个平台），去重后路由到会话
流水线 auto-reply/ 防抖（合并连续消息）→ 模型选择（OpenAI/Anthropic/Ollama）→ 加载历史 + 构建提示词
Agent 执行 agents/ ReAct 循环：LLM 决定调用 web_search 工具 → 获取结果 → 再次推理 → 生成最终回答
出站 infra/ 回复文本自动分块（适配平台长度限制），通过适配器推送给用户

关键设计：入站归一化和出站适配器形成两个解耦点——中间的所有逻辑完全不知道消息来自哪个平台。这就是 OpenClaw 能支持 14+ 通道而不用为每个通道写不同处理逻辑的原因。

1.7 十八大设计模式速览

分析 OpenClaw、Letta、NanoClaw 源码的过程中，我们识别出 18 个反复出现的设计模式——不是教科书理论，而是为解决真实问题而采用的模式。按章节分组：

第二章 · 运行时 - ReAct Loop — Reason → Act → Observe 循环，分步推理执行复杂任务 - Chain of Responsibility — 九层策略链式过滤不同工具集 - Factory — 工厂函数封装复杂的工具创建逻辑

第三章 · 记忆 - Composite — 向量 + 关键词多源搜索结果融合 - Singleton — 确保搜索索引只创建一个实例 - Fallback/Degradation — 主搜索引擎不可用时自动降级到内建方案

第四章 · 通道 - Adapter — 统一接口隔离 14+ 平台的协议差异 - Middleware — 请求经中间件链逐层处理，分离横切关注点 - Event-Driven — 事件发射器解耦组件通信

第五章 · 安全 - Chain of Responsibility — 九层安全策略链式过滤 - Proxy/Decorator — 透明包装调用，注入校验和审计

第六章 · 调度 - Event-Driven + Heartbeat — 事件循环 + LLM 主动续约心跳 - Checkpoint + Failover — 状态快照 + 崩溃恢复 - Command Lane — 会话专属串行通道，防止并发状态冲突

第七章 · 技能 - Registry — 中央注册表，运行时热加载插件 - Observer/Hook — 生命周期钩子通知，解耦扩展点 - Dynamic Loading — 目录扫描 + manifest 驱动，运行时加载技能

第八章 · 组装 - Dependency Injection — 外部注入依赖，解耦模块创建与使用 - Builder — 分步构建复杂 Agent 对象

每个模式嵌入在对应章节 x.4 节的评价标准展开中就地分析，包含 Mermaid 原理图 + 跨项目实现对比。第八章 8.4 节提供全书设计模式总览表。

1.5 Life of a Message: A WhatsApp Message's Complete Journey

A user sends "What's the weather like in Tokyo today?" on WhatsApp — what does this message go through inside OpenClaw?

flowchart TD subgraph Inbound ["Inbound (receive + preprocess)"] direction LR A(["User sends msg"]) --> B["Receive + Normalize + Route"] --> C["Debounce + Model select + Load history"] end subgraph Execution ["Agent Execution (ReAct loop)"] direction LR D{"ReAct Loop"} -->|"needs tools"| E["Tool call"] -->|"results"| D end subgraph Outbound ["Outbound (adapt + deliver)"] direction LR F["Chunk + Platform adapt"] --> G(["Delivered"]) end C --> D D -->|"final answer"| F
Stage Modules What Happens
Inbound web/ · channels/ Adapter receives message, normalizes to unified format (regardless of platform), dedupes, routes to session
Pipeline auto-reply/ Debounce (merge rapid messages) → model selection (OpenAI/Anthropic/Ollama) → load history + build prompt
Agent Execution agents/ ReAct loop: LLM decides to call web_search → gets results → reasons again → generates final answer
Outbound infra/ Reply text auto-chunked (for platform size limits), pushed to user via adapter

Key design: inbound normalization and outbound adapters form two decoupling points — all logic in between has zero knowledge of which platform the message came from. This is how OpenClaw supports 14+ channels without writing different processing logic for each one.

1.7 Eighteen Design Patterns at a Glance

Analyzing the source code of OpenClaw, Letta, and NanoClaw, we identified 18 recurring design patterns — not textbook theory, but patterns adopted to solve real problems. Grouped by chapter:

Chapter 2 · Runtime - ReAct Loop — Reason → Act → Observe cycle, step-by-step reasoning for complex tasks - Chain of Responsibility — Nine-layer policy chain filtering different tool sets - Factory — Factory functions encapsulate complex tool creation logic

Chapter 3 · Memory - Composite — Vector + keyword multi-source search result fusion - Singleton — Ensure only one search index instance is created - Fallback/Degradation — Auto-degrade to built-in when primary search engine is unavailable

Chapter 4 · Channels - Adapter — Unified interface isolates 14+ platforms' protocol differences - Middleware — Requests processed through middleware chain, separating cross-cutting concerns - Event-Driven — Event emitters decouple component communication

Chapter 5 · Security - Chain of Responsibility — Nine-layer security policy chain filtering - Proxy/Decorator — Transparent call wrapping, injecting validation and auditing

Chapter 6 · Scheduling - Event-Driven + Heartbeat — Event loop + LLM-initiated heartbeat renewal - Checkpoint + Failover — State snapshots + crash recovery - Command Lane — Session-specific serial channel, preventing concurrent state conflicts

Chapter 7 · Skills - Registry — Central registry, runtime plugin hot-loading - Observer/Hook — Lifecycle hook notifications, decoupled extension points - Dynamic Loading — Directory scanning + manifest-driven, runtime skill loading

Chapter 8 · Assembly - Dependency Injection — External dependency injection, decoupling module creation from usage - Builder — Step-by-step construction of complex Agent objects

Each pattern is embedded in the corresponding chapter's section x.4 evaluation criteria expansion and analyzed in place, with Mermaid principle diagrams + cross-project implementation comparisons. Chapter 8, section 8.4 provides a book-wide design pattern overview table.
