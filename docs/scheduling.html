6.1 调度模块：为什么需要它？

你对 Agent 说："每天早上 9 点给我一份新闻摘要。"Agent 回答："好的。"然后——什么都没有发生。第二天早上 9 点，你打开聊天窗口，没有新闻。

因为你的 Agent 只会"被问才答"。没有人发消息给它，它就什么都不做。本质上，它还是个聊天机器人。

调度模块解决的核心问题：

让 Agent 在没有人类明确输入的情况下，按计划或按需自主执行任务，并安全地管理任务优先级、并发、故障恢复。

三种"助理"的自主性光谱：

级别 类比 项目 驱动模型
Level 1 前台接待——有人来才抬头 NanoClaw 纯事件驱动
Level 2 行政助理——每天 9 点整理桌面 OpenClaw 事件 + Cron 混合
Level 3 私人管家——自己判断该不该多做一步 Letta 心跳驱动（自我驱动）

6.3 核心算法：sleep-loop

while True:
    for task in tasks:
        if time.time() >= task.next_run: task.handler()

6.4 生产级实现：从 3 行到 4000+ 行

C1 定时精度：怎么表达"明天 9 点提醒我"？

用户的意图是自然语言，系统需要转化为可执行的调度指令。

OpenClaw 提供三种调度类型：at（一次性绝对时间）、every（固定间隔 + 锚点对齐）、cron（标准 cron 表达式 + 时区）。核心是 computeNextRunAtMs()（schedule.ts:13-67），其中 every 模式采用锚点对齐算法——从 anchorMs 起按固定格栅计算，而非"上次运行 + 间隔"，从根本上防止长期运行时的时间漂移。

NanoClaw 的调度更简单：cron、interval（简单 Date.now() + ms 偏移）、once。interval 模式会因执行耗时产生漂移，但实现极其简洁——"个人 Agent 场景下简单优先"。

Letta 根本没有传统"定时任务"概念。Agent 通过 request_heartbeat 参数声明"我还没做完，给我下一轮"。对于真正的定时任务，使用 APScheduler + PostgreSQL advisory lock 实现 leader election。

# OpenClaw 风格 —— 锚点对齐防漂移
def compute_next_anchor(anchor_ms, interval_ms, now_ms):
    periods = max(1, int((now_ms - anchor_ms) / interval_ms) + 1)
    return anchor_ms + periods * interval_ms
    # 从锚点开始的固定格栅，不受执行耗时影响

# NanoClaw 风格 —— 简单偏移
def compute_next_simple(last_run_ms, interval_ms):
    return last_run_ms + interval_ms
    # 如果执行耗时 5s，每次偏移 5s，长期运行后漂移严重

⚠️ 常见陷阱：cron 表达式的时区处理远比看起来复杂。夏令时切换可能导致任务跳过（spring forward）或重复执行（fall back）。

C2 自我驱动：Agent 应该有"心跳"吗？

这是本章最具哲学意味的设计问题。

Letta 的心跳模型是三个项目中最独特的设计。 Agent 的每一个工具调用都可以附带 request_heartbeat 参数。当设为 true 时，系统立即给予 Agent 下一轮执行机会。Agent 可以自己决定是否需要继续思考。

stateDiagram-v2 state "NanoClaw: 纯反应" as NC { [*] --> 等待消息 等待消息 --> 处理: 收到消息 处理 --> 等待消息: 完成 } state "OpenClaw: 事件 + 定时" as OC { [*] --> 空闲 空闲 --> 执行: 消息到达 / cron 触发 执行 --> 空闲: 完成 空闲 --> 执行: setTimeout 唤醒 } state "Letta: 心跳链" as LT { [*] --> 接收输入 接收输入 --> 执行步骤 执行步骤 --> 执行步骤: heartbeat=true\n(≤100 步) 执行步骤 --> 完成: heartbeat=false 完成 --> [*] }

OpenClaw 没有心跳，但通过 cron 自调度实现类似效果——一个任务可以在执行过程中创建新任务。区别在于基于时钟（"5 分钟后再做"），而非即时的。

NanoClaw 是纯反应式的。Agent 想要"主动"做某事，唯一方式是通过 IPC 写入 schedule_task 文件。

💡 哲学思考：Agent 的"持续思考"和"失控循环"之间的边界在哪里？当 Agent 连续请求 100 次心跳时，它是在"深度思考一个复杂问题"，还是在"陷入无意义的循环"？max_chaining_steps 是硬性安全边界，但真正的答案需要更精细的判断。

设计模式就地展开——Event-Driven + Heartbeat

解决的问题：如何让 Agent 在"完全被动"和"完全自主"之间找到平衡点？纯事件驱动太被动（只能等消息），完全自主太危险（可能无限循环）。

解决的方法：事件驱动作为基础（消息到达触发），心跳机制作为扩展（Agent 请求自己的下一轮）。max_chaining_steps 作为安全边界，防止心跳失控。

C3 故障恢复：Agent 崩溃后 pending 任务怎么恢复？

进程会崩溃，这不是"如果"而是"什么时候"的问题。

OpenClaw 的 runMissedJobs()（timer.ts:360-394）在启动时扫描所有超过 nextRunAtMs 的任务并立即补执行。幂等性判断通过 runningAtMs（正在执行？）和 lastStatus（一次性任务已完成？）两个字段实现。结合 launchd/systemd 的 Restart=always，形成"崩溃→重启→补执行"完整链条。

NanoClaw 的 recoverPendingMessages()（index.ts:380-392）在启动时遍历所有 group 检查未处理消息。ensureContainerSystemRunning() 清理孤儿容器——扫描所有 nanoclaw- 开头的运行中容器并停止。

Letta 依托服务端架构。Agent 状态持久化在 PostgreSQL 中，调度器使用 pg_try_advisory_lock 实现 leader election——多实例中只有一个获得调度锁，leader 崩溃时另一个自动接管。

# 幂等恢复 —— OpenClaw 风格
def recover_missed(tasks, now):
    for t in tasks:
        if t.enabled and t.next_run and now >= t.next_run:
            if t.running_at:  continue      # 可能另一进程在跑
            if t.schedule_type == "once" and t.last_status == "ok":
                continue                     # 一次性任务已完成
            execute(t)                       # 补执行

# 没有幂等判断会怎样？
# 崩溃前任务已经执行了一半但标记为 pending
# 恢复时重复执行 → 发了两条每日摘要 → 用户收到两份一样的

🔄 幂等性是崩溃恢复的灵魂：没有幂等判断，崩溃恢复反而会导致任务重复执行——治病变成致病。

设计模式就地展开——Checkpoint + Failover（检查点 + 故障转移）

mermaid flowchart LR subgraph 正常 ["正常运行周期"] direction LR RUN["正常运行<br/>处理任务"] -->|"周期性"| SAVE["保存检查点<br/>JSON / SQLite"] SAVE --> RUN end subgraph 恢复 ["崩溃恢复周期"] direction LR CRASH["进程终止"] --> RESTART["自动重启<br/>launchd / systemd"] RESTART --> RECOVER["runMissedJobs()<br/>补执行遗漏"] end RUN -->|"崩溃"| CRASH RECOVER -->|"恢复"| RUN

解决的问题：进程崩溃后，已注册但未执行的任务会丢失吗？

解决的方法：周期性保存状态到持久化存储（JSON/SQLite/PostgreSQL），重启时扫描并补执行遗漏任务。幂等性判断防止重复执行。

C4 资源控制：定时任务和实时对话怎么不冲突？

当 Agent 既要处理实时消息又要执行定时任务时，并发冲突不可避免。

OpenClaw 使用 Lane（车道）机制（command-queue.ts:19-56）。每条 lane 是独立的任务队列——main 处理用户交互（串行），cron 处理定时任务（可并行）。generation 字段在进程内重启时递增，使旧任务回调失效。

NanoClaw 使用 GroupQueue（group-queue.ts:27-33），每个 group 同一时刻只有一个容器运行，全局最多 5 个并发。drainGroup() 实现优先级链式排水：任务 > 消息 > 其他等待组。

Letta 从根本上规避了这个问题——单线程链式执行。step() 在 while True 循环中运行，不存在并发冲突。

设计模式就地展开——Command Lane（命令车道）

mermaid flowchart TB subgraph LANES["命令车道"] direction LR L1["Session A<br/>串行队列"] L2["Session B<br/>串行队列"] L3["Cron 车道<br/>可并行"] end REQ1["用户消息 A"] --> L1 REQ2["用户消息 B"] --> L2 CRON["定时任务"] --> L3 L1 --> RT["Agent Runtime"] L2 --> RT L3 --> RT

解决的问题：多个 session 的消息和定时任务并发到达，如何防止状态污染？

解决的方法：将请求按类型分到独立"车道"。同一车道内串行执行，不同车道可并行。OpenClaw 的 main 车道保证用户交互的串行性，cron 车道允许定时任务的并行性。

C5 优先级抢占：紧急任务能插队吗？

NanoClaw 的 drainGroup() 有明确的优先级顺序：任务 > 消息 > 其他等待组。任务优先级高于消息，因为任务入队后不会被 SQLite 重新发现——如果不及时执行就会丢失。

OpenClaw 的 Lane 机制目前没有显式的优先级抢占——一个正在运行的 cron 任务无法被更高优先级的新任务打断。但可以通过配置不同 lane 的并发数实现间接优先级控制。

Letta 作为单线程模型，不存在抢占概念——一切串行化。

# NanoClaw 风格 —— 优先级排水
def drain_group(group):
    # 1. 先看有没有 scheduled task（最高优先级）
    task = group.task_queue.pop()
    if task:
        return execute(task)
    # 2. 再看有没有用户消息
    msg = group.message_queue.pop()
    if msg:
        return execute(msg)
    # 3. 最后看有没有其他等待的 group
    # ...

跨项目对比：三种自主性光谱

评价标准 NanoClaw（被动） OpenClaw（定时） Letta（自治）
C1 定时精度 interval 简单偏移，有漂移 锚点对齐，无漂移 N/A（即时链式）
C2 自我驱动 否（纯反应式） 间接（cron 自调度） 核心设计（heartbeat）

💡 核心洞察：自主性是一个光谱，不是开关。从纯反应式到心跳驱动，每一级都增加了能力，也增加了风险。给 Agent 油门的同时必须给刹车。

"""mini_agent/scheduler.py — 智能任务调度器

第六章 Lab · Cron + Interval + 心跳 + 事件驱动 + 崩溃恢复

单独运行: python -m mini_agent.scheduler
依赖: croniter（可选 — 无安装时 cron 任务自动跳过）
"""
from __future__ import annotations

import asyncio, json, time, os, logging
from dataclasses import dataclass, field
from typing import Callable, Awaitable

logging.basicConfig(level=logging.INFO,
                    format="%(asctime)s [sched] %(message)s")
log = logging.getLogger("scheduler")

BACKOFF = [30, 60, 300, 900, 3600]

try:
    from croniter import croniter as _croniter
except ImportError:
    _croniter = None


@dataclass
class ScheduledTask:
    id: str
    name: str
    schedule_type: str           # "cron" | "interval" | "once" | "heartbeat"
    schedule_value: str          # cron expr | ms | ISO date | ""
    handler: Callable[[], Awaitable[str]]
    enabled: bool = True
    next_run: float | None = None
    last_run: float | None = None
    last_status: str | None = None
    errors: int = 0
    anchor: float = field(default_factory=time.time)


def _compute_next(t: ScheduledTask, now: float) -> float | None:
    """计算下次执行时间（锚点对齐防漂移）"""
    if not t.enabled:
        return None
    if t.schedule_type == "cron":
        if _croniter is None:
            log.warning(f"croniter 未安装, cron 任务 '{t.name}' 已跳过")
            return None
        return _croniter(t.schedule_value, now).get_next(float)
    if t.schedule_type == "interval":
        sec = int(t.schedule_value) / 1000
        periods = max(1, int((now - t.anchor) / sec) + 1)
        return t.anchor + periods * sec
    if t.schedule_type == "once":
        ts = time.mktime(
            time.strptime(t.schedule_value, "%Y-%m-%dT%H:%M:%S"))
        return ts if ts > now else None
    if t.schedule_type == "heartbeat":
        return now
    return None


class Scheduler:
    """任务调度器 — cron/interval/heartbeat + 事件 + 崩溃恢复"""

    def __init__(self, event_dir: str = "./events"):
        self.tasks: dict[str, ScheduledTask] = {}
        self._event_handlers: dict[str, Callable] = {}
        self.event_dir = event_dir
        self._running = False
        os.makedirs(event_dir, exist_ok=True)

    def add_task(self, name: str, stype: str, sval: str,
                 handler: Callable[[], Awaitable[str]]) -> str:
        tid = f"t-{int(time.time()*1000)}-{os.urandom(2).hex()}"
        task = ScheduledTask(
            id=tid, name=name, schedule_type=stype,
            schedule_value=sval, handler=handler)
        task.next_run = _compute_next(task, time.time())
        self.tasks[tid] = task
        log.info(f"+ {name} ({stype}={sval or 'now'})")
        return tid

    def request_heartbeat(self, name: str, handler: Callable) -> str:
        return self.add_task(name, "heartbeat", "", handler)

    def on_event(self, event_type: str, handler: Callable) -> None:
        self._event_handlers[event_type] = handler

    async def start(self) -> None:
        self._running = True
        await self._recover_missed()
        log.info(f"Started — {len(self.tasks)} tasks")
        await asyncio.gather(self._tick_loop(), self._event_loop())

    def stop(self) -> None:
        self._running = False
        log.info("Scheduler stopped")

    async def _tick_loop(self):
        while self._running:
            now = time.time()
            nearest = float("inf")
            for t in list(self.tasks.values()):
                if t.enabled and t.next_run is not None \
                        and now >= t.next_run:
                    await self._execute(t)
                if t.enabled and t.next_run is not None:
                    nearest = min(nearest, t.next_run)
            wait = (max(0.1, min(nearest - time.time(), 60))
                    if nearest < float("inf") else 60)
            await asyncio.sleep(wait)

    async def _execute(self, t: ScheduledTask):
        log.info(f"▶ {t.name}")
        try:
            await asyncio.wait_for(t.handler(), timeout=600)
            t.last_status, t.errors = "ok", 0
        except Exception as e:
            t.last_status = "error"
            t.errors += 1
            log.error(f"✗ {t.name}: {e}")
        t.last_run = time.time()
        if t.schedule_type in ("once", "heartbeat"):
            t.enabled, t.next_run = False, None
        elif t.last_status == "error":
            normal = _compute_next(t, time.time())
            bo = BACKOFF[min(t.errors - 1, len(BACKOFF) - 1)]
            t.next_run = max(normal or (time.time() + bo), time.time() + bo)
        else:
            t.next_run = _compute_next(t, time.time())

    async def _event_loop(self):
        while self._running:
            try:
                for fname in os.listdir(self.event_dir):
                    if not fname.endswith(".json"):
                        continue
                    fp = os.path.join(self.event_dir, fname)
                    try:
                        evt = json.loads(open(fp, encoding="utf-8").read())
                        handler = self._event_handlers.get(evt.get("type"))
                        if handler:
                            await handler(evt.get("payload", {}))
                        os.unlink(fp)
                    except Exception:
                        err_dir = os.path.join(self.event_dir, "errors")
                        os.makedirs(err_dir, exist_ok=True)
                        os.rename(fp, os.path.join(err_dir, fname))
            except FileNotFoundError:
                pass
            await asyncio.sleep(1)

    async def _recover_missed(self):
        now = time.time()
        for t in list(self.tasks.values()):
            if t.enabled and t.next_run and now >= t.next_run:
                log.info(f"↻ 恢复遗漏: {t.name}")
                await self._execute(t)


async def _demo():

评价标准 核心洞察
C1 定时精度 锚点对齐算法从根本上解决时间漂移——从 anchorMs 起按固定格栅计算，不受执行耗时影响。简单的"上次+间隔"在长期运行后会严重漂移。
C2 自我驱动 Letta 的心跳模型让 Agent 能"自己决定继续思考"，是模拟"意识流"的设计。但 max_chaining_steps 是必须的安全边界——给油门的同时必须给刹车。
C3 故障恢复 幂等性是崩溃恢复的灵魂。没有幂等判断，恢复反而导致重复执行。OpenClaw 通过 runningAtMs + lastStatus 双字段实现幂等。
C4 资源控制 Command Lane 把不同类型的工作分到独立车道，互不干扰。GroupQueue 用全局仲裁统管所有工作。Letta 直接取消并发——一切串行化。
C5 优先级抢占 NanoClaw 的"任务 > 消息"排水顺序解决了"定时任务入队后不及时执行就会丢失"的问题。但没有抢占能力——正在运行的长任务无法被打断。

三条核心设计原则：

  1. 自主性是一个光谱，不是开关——选择适合你场景的位置。
  2. 给 Agent 油门的同时必须给刹车——每种自主性都需要对应的安全边界。
  3. 崩溃恢复是第一等公民——从第一天就要设计，幂等性检查防止恢复变灾难。

6.1 The Scheduling Module: Why Do We Need It?

You tell your Agent: "Give me a news digest every morning at 9 AM." The Agent replies: "Sure." And then — nothing happens. The next morning at 9, you open the chat window. No news.

Because your Agent only responds when spoken to. If nobody sends it a message, it does nothing. At its core, it is still a chatbot.

The core problem the scheduling module solves:

Enable the Agent to autonomously execute tasks on a schedule or on demand — without explicit human input — while safely managing task priorities, concurrency, and crash recovery.

Three levels of assistant autonomy:

Level Analogy Project Driving Model
Level 1 Receptionist — only looks up when someone arrives NanoClaw Purely event-driven
Level 2 Administrative assistant — tidies the desk every day at 9 OpenClaw Event + Cron hybrid
Level 3 Personal butler — decides on their own whether to take an extra step Letta Heartbeat-driven (self-driven)

6.4 Production Implementation: From 3 Lines to 4000+

C1 Timing Precision: How to Express "Remind Me Tomorrow at 9 AM"?

The user's intent is in natural language; the system must convert it into an executable scheduling instruction.

OpenClaw provides three schedule types: at (one-shot absolute time), every (fixed interval + anchor alignment), and cron (standard cron expression + timezone). The core is computeNextRunAtMs() (schedule.ts:13-67), where the every mode uses an anchor-alignment algorithm — it calculates from anchorMs on a fixed grid rather than "last run + interval," fundamentally preventing time drift over long-running periods.

NanoClaw has simpler scheduling: cron, interval (a straightforward Date.now() + ms offset), and once. The interval mode drifts due to execution duration, but the implementation is extremely concise — "simplicity first for a personal-Agent scenario."

Letta has no traditional "scheduled task" concept at all. The Agent declares "I'm not done yet, give me another turn" via the request_heartbeat parameter. For true scheduled tasks, it uses APScheduler + PostgreSQL advisory locks for leader election.

# OpenClaw style — anchor alignment to prevent drift
def compute_next_anchor(anchor_ms, interval_ms, now_ms):
    periods = max(1, int((now_ms - anchor_ms) / interval_ms) + 1)
    return anchor_ms + periods * interval_ms
    # Fixed grid from the anchor; unaffected by execution duration

# NanoClaw style — simple offset
def compute_next_simple(last_run_ms, interval_ms):
    return last_run_ms + interval_ms
    # If execution takes 5s, it drifts 5s each time; severe drift over long runs

⚠️ Common Pitfall: Timezone handling for cron expressions is far more complex than it appears. Daylight Saving Time transitions can cause tasks to be skipped (spring forward) or executed twice (fall back).

C2 Self-Driven: Should an Agent Have a "Heartbeat"?

This is the most philosophically charged design question in this chapter.

Letta's heartbeat model is the most distinctive design among the three projects. Every tool call the Agent makes can carry a request_heartbeat parameter. When set to true, the system immediately grants the Agent another execution turn. The Agent can decide for itself whether it needs to keep thinking.

stateDiagram-v2 state "NanoClaw: 纯反应" as NC { [*] --> 等待消息 等待消息 --> 处理: 收到消息 处理 --> 等待消息: 完成 } state "OpenClaw: 事件 + 定时" as OC { [*] --> 空闲 空闲 --> 执行: 消息到达 / cron 触发 执行 --> 空闲: 完成 空闲 --> 执行: setTimeout 唤醒 } state "Letta: 心跳链" as LT { [*] --> 接收输入 接收输入 --> 执行步骤 执行步骤 --> 执行步骤: heartbeat=true\n(≤100 步) 执行步骤 --> 完成: heartbeat=false 完成 --> [*] }

OpenClaw has no heartbeat, but achieves a similar effect through cron self-scheduling — a task can create new tasks during its own execution. The difference is that it is clock-based ("do it again in 5 minutes") rather than immediate.

NanoClaw is purely reactive. The only way for the Agent to "proactively" do something is to write to a schedule_task file via IPC.

💡 Philosophical Reflection: Where is the boundary between an Agent's "sustained thinking" and a "runaway loop"? When an Agent requests 100 consecutive heartbeats, is it "deeply thinking through a complex problem" or "stuck in a meaningless cycle"? max_chaining_steps provides a hard safety boundary, but the true answer demands more nuanced judgment.

Design Pattern Inline — Event-Driven + Heartbeat

Problem solved: How to position an Agent between "entirely passive" and "fully autonomous"? Pure event-driven is too passive (it can only wait for messages); full autonomy is too dangerous (it may loop forever).

Solution: Event-driven as the foundation (message arrival triggers execution), heartbeat mechanism as an extension (the Agent requests its own next turn). max_chaining_steps serves as the safety boundary to prevent heartbeat runaway.

C3 Crash Recovery: How to Recover Pending Tasks After an Agent Crash?

Processes will crash — it is not a question of "if" but "when."

OpenClaw's runMissedJobs() (timer.ts:360-394) scans all tasks whose nextRunAtMs has been exceeded at startup and immediately catches up. Idempotency is determined through two fields: runningAtMs (is another instance already running it?) and lastStatus (has a one-shot task already completed?). Combined with launchd/systemd Restart=always, this forms a complete "crash → restart → catch-up" chain.

NanoClaw's recoverPendingMessages() (index.ts:380-392) iterates over all groups at startup to check for unprocessed messages. ensureContainerSystemRunning() cleans up orphan containers — it scans all running containers prefixed with nanoclaw- and stops them.

Letta relies on a server-side architecture. Agent state is persisted in PostgreSQL, and the scheduler uses pg_try_advisory_lock for leader election — only one instance among many acquires the scheduling lock; if the leader crashes, another automatically takes over.

# Idempotent recovery — OpenClaw style
def recover_missed(tasks, now):
    for t in tasks:
        if t.enabled and t.next_run and now >= t.next_run:
            if t.running_at:  continue      # Another process might be running it
            if t.schedule_type == "once" and t.last_status == "ok":
                continue                     # One-shot task already completed
            execute(t)                       # Catch-up execution

# What happens without idempotency checks?
# Task was half-executed before the crash but marked as pending
# Recovery re-executes it → two daily digests sent → user gets duplicates

🔄 Idempotency is the soul of crash recovery: Without idempotency checks, crash recovery actually causes duplicate task execution — the cure becomes the disease.

Design Pattern Inline — Checkpoint + Failover

mermaid flowchart LR subgraph Normal ["Normal operation cycle"] direction LR RUN["Running<br/>processing tasks"] -->|"periodic"| SAVE["Save checkpoint<br/>JSON / SQLite"] SAVE --> RUN end subgraph Recovery ["Crash recovery cycle"] direction LR CRASH["Process terminated"] --> RESTART["Auto restart<br/>launchd / systemd"] RESTART --> RECOVER["runMissedJobs()<br/>catch up missed"] end RUN -->|"crash"| CRASH RECOVER -->|"recovered"| RUN

Problem solved: After a process crash, are registered but unexecuted tasks lost?

Solution: Periodically save state to persistent storage (JSON/SQLite/PostgreSQL); on restart, scan and catch up on missed tasks. Idempotency checks prevent duplicate execution.

C4 Resource Control: How to Prevent Conflicts Between Scheduled Tasks and Live Conversations?

When an Agent must handle both real-time messages and scheduled tasks, concurrency conflicts are inevitable.

OpenClaw uses the Lane mechanism (command-queue.ts:19-56). Each lane is an independent task queue — main handles user interactions (serial), cron handles scheduled tasks (parallelizable). The generation field increments on in-process restarts, invalidating stale task callbacks.

NanoClaw uses GroupQueue (group-queue.ts:27-33): each group runs at most one container at a time, with a global maximum of 5 concurrent. drainGroup() implements priority-chained draining: tasks > messages > other waiting groups.

Letta sidesteps the problem entirely — single-threaded chained execution. step() runs inside a while True loop; concurrency conflicts simply do not exist.

Design Pattern Inline — Command Lane

mermaid flowchart TB subgraph LANES["命令车道"] direction LR L1["Session A<br/>串行队列"] L2["Session B<br/>串行队列"] L3["Cron 车道<br/>可并行"] end REQ1["用户消息 A"] --> L1 REQ2["用户消息 B"] --> L2 CRON["定时任务"] --> L3 L1 --> RT["Agent Runtime"] L2 --> RT L3 --> RT

Problem solved: When messages from multiple sessions and scheduled tasks arrive concurrently, how do you prevent state corruption?

Solution: Route requests into independent "lanes" by type. Within a single lane, execution is serial; across lanes, execution can be parallel. OpenClaw's main lane guarantees serial user interactions while its cron lane allows parallel scheduled-task execution.

C5 Priority Preemption: Can Urgent Tasks Jump the Queue?

NanoClaw's drainGroup() has a clear priority ordering: tasks > messages > other waiting groups. Tasks have higher priority than messages because once enqueued, tasks will not be rediscovered by SQLite — if they are not executed promptly, they are lost.

OpenClaw's Lane mechanism currently lacks explicit priority preemption — a running cron task cannot be interrupted by a higher-priority incoming task. However, indirect priority control can be achieved by configuring different concurrency limits per lane.

Letta, as a single-threaded model, has no concept of preemption — everything is serialized.

# NanoClaw style — priority draining
def drain_group(group):

    task = group.task_queue.pop()
    if task:
        return execute(task)
    # 2. Then check for user messages
    msg = group.message_queue.pop()
    if msg:
        return execute(msg)

"""mini_agent/scheduler.py — 智能任务调度器

第六章 Lab · Cron + Interval + 心跳 + 事件驱动 + 崩溃恢复

单独运行: python -m mini_agent.scheduler
依赖: croniter（可选 — 无安装时 cron 任务自动跳过）
"""
from __future__ import annotations

import asyncio, json, time, os, logging
from dataclasses import dataclass, field
from typing import Callable, Awaitable

logging.basicConfig(level=logging.INFO,
                    format="%(asctime)s [sched] %(message)s")
log = logging.getLogger("scheduler")

BACKOFF = [30, 60, 300, 900, 3600]

try:
    from croniter import croniter as _croniter
except ImportError:
    _croniter = None


@dataclass
class ScheduledTask:
    id: str
    name: str
    schedule_type: str           # "cron" | "interval" | "once" | "heartbeat"
    schedule_value: str          # cron expr | ms | ISO date | ""
    handler: Callable[[], Awaitable[str]]
    enabled: bool = True
    next_run: float | None = None
    last_run: float | None = None
    last_status: str | None = None
    errors: int = 0
    anchor: float = field(default_factory=time.time)


def _compute_next(t: ScheduledTask, now: float) -> float | None:
    """计算下次执行时间（锚点对齐防漂移）"""
    if not t.enabled:
        return None
    if t.schedule_type == "cron":
        if _croniter is None:
            log.warning(f"croniter 未安装, cron 任务 '{t.name}' 已跳过")
            return None
        return _croniter(t.schedule_value, now).get_next(float)
    if t.schedule_type == "interval":
        sec = int(t.schedule_value) / 1000
        periods = max(1, int((now - t.anchor) / sec) + 1)
        return t.anchor + periods * sec
    if t.schedule_type == "once":
        ts = time.mktime(
            time.strptime(t.schedule_value, "%Y-%m-%dT%H:%M:%S"))
        return ts if ts > now else None
    if t.schedule_type == "heartbeat":
        return now
    return None


class Scheduler:
    """任务调度器 — cron/interval/heartbeat + 事件 + 崩溃恢复"""

    def __init__(self, event_dir: str = "./events"):
        self.tasks: dict[str, ScheduledTask] = {}
        self._event_handlers: dict[str, Callable] = {}
        self.event_dir = event_dir
        self._running = False
        os.makedirs(event_dir, exist_ok=True)

    def add_task(self, name: str, stype: str, sval: str,
                 handler: Callable[[], Awaitable[str]]) -> str:
        tid = f"t-{int(time.time()*1000)}-{os.urandom(2).hex()}"
        task = ScheduledTask(
            id=tid, name=name, schedule_type=stype,
            schedule_value=sval, handler=handler)
        task.next_run = _compute_next(task, time.time())
        self.tasks[tid] = task
        log.info(f"+ {name} ({stype}={sval or 'now'})")
        return tid

    def request_heartbeat(self, name: str, handler: Callable) -> str:
        return self.add_task(name, "heartbeat", "", handler)

    def on_event(self, event_type: str, handler: Callable) -> None:
        self._event_handlers[event_type] = handler

    async def start(self) -> None:
        self._running = True
        await self._recover_missed()
        log.info(f"Started — {len(self.tasks)} tasks")
        await asyncio.gather(self._tick_loop(), self._event_loop())

    def stop(self) -> None:
        self._running = False
        log.info("Scheduler stopped")

    async def _tick_loop(self):
        while self._running:
            now = time.time()
            nearest = float("inf")
            for t in list(self.tasks.values()):
                if t.enabled and t.next_run is not None \
                        and now >= t.next_run:
                    await self._execute(t)
                if t.enabled and t.next_run is not None:
                    nearest = min(nearest, t.next_run)
            wait = (max(0.1, min(nearest - time.time(), 60))
                    if nearest < float("inf") else 60)
            await asyncio.sleep(wait)

    async def _execute(self, t: ScheduledTask):
        log.info(f"▶ {t.name}")
        try:
            await asyncio.wait_for(t.handler(), timeout=600)
            t.last_status, t.errors = "ok", 0
        except Exception as e:
            t.last_status = "error"
            t.errors += 1
            log.error(f"✗ {t.name}: {e}")
        t.last_run = time.time()
        if t.schedule_type in ("once", "heartbeat"):
            t.enabled, t.next_run = False, None
        elif t.last_status == "error":
            normal = _compute_next(t, time.time())
            bo = BACKOFF[min(t.errors - 1, len(BACKOFF) - 1)]
            t.next_run = max(normal or (time.time() + bo), time.time() + bo)
        else:
            t.next_run = _compute_next(t, time.time())

    async def _event_loop(self):
        while self._running:
            try:
                for fname in os.listdir(self.event_dir):
                    if not fname.endswith(".json"):
                        continue
                    fp = os.path.join(self.event_dir, fname)
                    try:
                        evt = json.loads(open(fp, encoding="utf-8").read())
                        handler = self._event_handlers.get(evt.get("type"))
                        if handler:
                            await handler(evt.get("payload", {}))
                        os.unlink(fp)
                    except Exception:
                        err_dir = os.path.join(self.event_dir, "errors")
                        os.makedirs(err_dir, exist_ok=True)
                        os.rename(fp, os.path.join(err_dir, fname))
            except FileNotFoundError:
                pass
            await asyncio.sleep(1)

    async def _recover_missed(self):
        now = time.time()
        for t in list(self.tasks.values()):
            if t.enabled and t.next_run and now >= t.next_run:
                log.info(f"↻ 恢复遗漏: {t.name}")
                await self._execute(t)


async def _demo():
    s = Scheduler()

    async def health():
        log.info("  health ok")
        return "ok"

    async def briefing():
        log.info("  briefing done")
        return "done"

    async def heartbeat_turn():
        log.info("  heartbeat turn")
        return "continued"

    s.add_task("Health Check",   "interval", "3000",      health)
    s.add_task("Daily Briefing", "cron",     "0 9 * * *", briefing)
    s.request_heartbeat("Agent Heartbeat", heartbeat_turn)
    s.on_event("email", lambda p: log.info(f"  email event: {p}"))

    async def _auto_stop():
        await asyncio.sleep(8)
        s.stop()
    asyncio.create_task(_auto_stop())

    await s.start()


if __name__ == "__main__":
    asyncio.run(_demo())

6.7 Chapter Summary

Criterion Core Insight
C1 Timing Precision The anchor-alignment algorithm fundamentally solves time drift — calculating from anchorMs on a fixed grid, unaffected by execution duration. The naive "last run + interval" approach drifts severely over long-running periods.
C2 Self-Driven Letta's heartbeat model lets the Agent "decide to keep thinking on its own," a design that simulates a "stream of consciousness." But max_chaining_steps is an essential safety boundary — whenever you give the throttle, you must also give the brake.
C3 Crash Recovery Idempotency is the soul of crash recovery. Without idempotency checks, recovery actually causes duplicate execution. OpenClaw achieves idempotency through the runningAtMs + lastStatus dual-field approach.
C4 Resource Control Command Lane routes different types of work into independent lanes so they do not interfere. GroupQueue uses global arbitration to manage all work centrally. Letta eliminates concurrency altogether — everything is serialized.
C5 Priority Preemption NanoClaw's "tasks > messages" draining order solves the problem of "scheduled tasks that are lost if not executed promptly after enqueue." However, there is no preemption — a long-running task in progress cannot be interrupted.

Three Core Design Principles:

  1. Autonomy is a spectrum, not a switch — choose the position that fits your scenario.
  2. Whenever you give the Agent a throttle, give it a brake too — every level of autonomy requires a corresponding safety boundary.
  3. Crash recovery is a first-class citizen — design for it from day one; idempotency checks prevent recovery from becoming disaster.
