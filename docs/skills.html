7.1 技能模块：为什么需要它？

你的 Agent 很聪明，能聊天、能记忆、能按时行事。你让它"帮我查一下明天北京的天气"——它回答："抱歉，我没有查天气的能力。"

你不得不去代码里加一个天气 API 调用，重新编译部署。三天后你又想让它查邮件、查日历、查股价……每次都要改核心代码。

这就是没有技能模块的 Agent——每新增一个能力都要改源码。

技能模块解决的核心问题：

让 Agent 按需获得新能力（工具），通过注册机制而非源码修改，同时管理工具的发现、权限、错误处理和生命周期。

三个项目给出了三种截然不同的方案：

方案 项目 核心思路
编译型插件 OpenClaw SDK + 注册表 + 15 种生命周期钩子
运行时工具 Letta Python 函数即工具，API 动态注册
Shell 命令 NanoClaw Bash + MCP 是全部工具集

7.3 核心算法：hardcoded-tools

技能模块的最简实现只需 3 行：

tools = {
    "get_weather": lambda city: f"{city}: 25°C",
    "calculator":  lambda expr: str(eval(expr)),
}

对照评价标准打分：

标准 最简实现 得分
C1 工具发现 硬编码列表 ❌ 新工具需改源码
C2 错误容错 无错误处理 ❌ eval 可执行任意代码
C3 权限控制 无权限 ❌ 所有工具都可调
C4 第三方接入 不支持 ❌ 无法连接外部服务
C5 生命周期管理 无注册机制 ❌ 无法增删工具

全部不及格。接下来看生产项目如何逐条补齐。

7.4 生产级实现：从 3 个工具到 40+ 技能

C1 工具发现：怎么让 LLM 知道有哪些工具可用？

LLM 不会自己发现工具——它只能使用你在请求中告诉它的那些。

OpenClaw：分层发现 + 条件注入。 discoverOpenClawPlugins()（loader.ts:200-210）按 config > workspace > global > bundled 四层优先级扫描。shouldIncludeSkill 对每个 SKILL.md 做能力检测——OS 兼容性、二进制依赖、环境变量——只有全部满足的技能才注入系统提示。

Letta：数据库驱动 + 类型分派。 每个 Agent 关联一组 Tool 记录，11 种 tool_type 枚举决定工具性质。构建 LLM 请求时从数据库取出工具的 json_schema。运行时可通过 API 动态增删。

NanoClaw：硬编码列表 + MCP 通配符。 在 query() 中显式列出 allowedTools——文件操作（Bash、Read、Write、Edit、Glob、Grep）、网络访问（WebSearch、WebFetch）、子代理协作（Task、TeamCreate、SendMessage）、辅助工具（TodoWrite、ToolSearch、Skill、NotebookEdit）等 18 种工具 + mcp__nanoclaw__*（MCP 通配符）。Bash 本身就是"元工具"——通过它可以调用任何命令行程序。

# OpenClaw 风格 —— 四层优先级发现
def discover_plugins(search_paths):
    seen = set()
    for path in [config_dir, workspace_dir, global_dir, bundled_dir]:
        for plugin in scan_dir(path):
            if plugin.id not in seen:
                seen.add(plugin.id)
                yield plugin            # 先到先得

# Letta 风格 —— 数据库查询
def get_tools_for_agent(agent_id):
    return db.query("SELECT json_schema FROM tools_agents WHERE agent_id=?",
                    agent_id)

# NanoClaw 风格 —— 硬编码
allowed_tools = ["Bash", "Read", "Write", "Edit", "mcp__nanoclaw__*"]

C2 错误容错：工具执行失败了怎么办？

工具调用不可避免地会失败。关键在于失败信息如何传回 LLM，让它能自我修正。

OpenClaw：钩子拦截 + 错误边界。 before_tool_call 钩子允许插件在执行前拦截——可以修改参数，甚至返回 { block: true } 阻止执行。after_tool_call 可以检查结果并触发重试。

Letta：结构化结果 + 友好错误消息。 ToolExecutionResult 用 status: "success" | "error" 标记。get_friendly_error_msg() 生成对 LLM 友好的错误描述——包含函数名和异常类型，但不含 traceback（traceback 放在 stderr 供调试）。

# Letta 风格 —— 结构化错误 + 友好消息
@dataclass
class ToolExecutionResult:
    status: str        # "success" | "error"
    func_return: str   # 返回值 or 错误描述（给 LLM 看）
    stderr: str        # traceback（给开发者看）

def get_friendly_error_msg(func_name, exception):
    return f"Error in {func_name}: {type(exception).__name__}: {exception}"
    # 不含 traceback → 节省 LLM 上下文窗口

NanoClaw：stderr 即反馈。 Bash 的非零退出码 + stderr 直接返回给 Claude。AI 自己决定是否重试。

设计模式就地展开——Observer / Hook System（观察者）

解决的问题：工具执行前后需要插入检查、日志、重试等逻辑，但这些逻辑不应该和工具本身耦合。

解决的方法：组件通过钩子订阅事件，解耦发布者与订阅者。OpenClaw 的 before_tool_call / after_tool_call 等 15 种生命周期钩子，分为 void（纯通知）和 modifying（可修改数据）两种模式。NanoClaw 的 startIpcWatcher() 是基于文件系统的观察者——容器写入文件"发布"请求，宿主轮询"订阅"并处理。

C3 权限控制：怎么限制工具的权限？

一个能发邮件的工具如果不受限制，可能给所有联系人群发广告。

OpenClaw：ACL + 工具白名单。 resolvePluginTools() 支持 toolAllowlist 参数：标记为 optional 的工具只有在白名单内才传给 LLM，非 optional 工具始终包含。内置冲突检测——多个插件注册同名工具时自动报告。结合 before_tool_call 钩子，可根据用户身份动态阻止工具调用。

Letta：审批机制 + 沙箱 + ToolRule。 default_requires_approval 字段控制是否需要人工确认。自定义工具在 SandboxToolExecutor 中执行。ToolRule 系统支持工具调用链约束（工具 A 执行后必须执行工具 B）。

NanoClaw：Docker 容器即沙箱。 整个 Agent 运行在容器内，容器就是安全边界。allowedTools 列表就是全部权限声明。

C4 第三方接入：第三方工具怎么接入？（MCP）

MCP（Model Context Protocol） 是行业对"AI 的 USB 接口"的标准化尝试——任何工具只要实现 MCP 协议（基于 JSON-RPC 2.0），就能被任何支持 MCP 的 Agent 使用。

NanoClaw 的 MCP 集成最深。 核心功能（send_message、schedule_task 等 7 个工具）就通过 MCP Server 暴露。添加第三方服务只需在 mcpServers 字典中加一项。

Letta 把 MCP 工具视为 ToolType.EXTERNAL_MCP，ToolCreate.from_mcp() 生成 Python 包装函数。

OpenClaw 的 MCP 支持通过插件系统实现，可按需加载。

⚠️ MCP 的现实检验：MCP 目前仍处于早期。协议只覆盖 Tool 调用，Resource 和 Prompt 两个能力在三个项目中都未被使用。最关键的限制：MCP 没有定义认证标准——每个 Server 自行处理 API Key 和 OAuth。

设计模式就地展开——Registry（注册表）

mermaid flowchart TB P1["插件 A<br/>registerTool()"] -->|registerTool| REG["Plugin Registry<br/>tools / hooks / channels"] P2["插件 B<br/>registerHook()"] -->|registerHook| REG P3["MCP Server<br/>JSON-RPC 桥接"] -->|registerTool| REG REG --> RESOLVE["resolvePluginTools()<br/>白名单 + 冲突检测"] RESOLVE --> LLM["注入 LLM<br/>tools[] → prompt"]

解决的问题：多个来源（内置、插件、MCP）的工具如何统一管理？如何检测命名冲突？如何按白名单过滤？

解决的方法：一个中央注册表动态管理所有扩展——注册、查找、解析冲突。OpenClaw 的 createPluginRegistry() 存储 13 种注册类别。注册表是插件系统的"真相之源"。

C5 生命周期管理：工具的注册、更新、卸载？

OpenClaw：启动时加载 + 文件系统管理。 loadOpenClawPlugins() 在启动时通过 jiti 即时编译加载 TypeScript 模块。插件的"安装/卸载"就是文件系统操作——拷贝/删除 extensions/ 目录中的文件夹。更新需要重启。

Letta：API 动态管理 + 数据库持久化。 ToolManager.create_tool() 通过 REST API 创建工具，源码存入数据库的 source_code 字段。随时可更新可删除，无需重启。

NanoClaw："Don't add features. Add skills." NanoClaw 的扩展哲学最激进——不是通过代码添加功能，而是通过 .claude/skills/ 目录中的 Markdown 文件教 AI 如何使用 Bash 和 MCP 完成新任务。

设计模式就地展开——Dynamic Loading（动态加载）

解决的问题：如何在不重新编译的情况下加载新模块？

解决的方法：运行时加载模块代码。OpenClaw 用 jiti 在运行时直接加载 TypeScript，无需预编译。Letta 用 importlib 动态导入 Python 函数，derive_openai_json_schema() 通过 AST 解析自动生成 Schema。

"""mini_agent/skills.py — 技能扩展系统

第七章 Lab · 工具注册表 + 动态加载 + 技能目录

单独运行: python -m mini_agent.skills
依赖: 仅 Python 标准库
"""
from __future__ import annotations

import json, os, time, importlib.util
from dataclasses import dataclass, field
from typing import Callable

from mini_agent.runtime import Tool, ToolRegistry


@dataclass
class Skill:
    """一组相关工具的集合"""
    name: str
    description: str
    version: str = "1.0.0"
    tools: list[Tool] = field(default_factory=list)

    def tool_names(self) -> list[str]:
        return [t.name for t in self.tools]


class SkillRegistry:
    def __init__(self):
        self._skills: dict[str, Skill] = {}

    def register(self, skill: Skill) -> None:
        self._skills[skill.name] = skill

    def get(self, name: str) -> Skill | None:
        return self._skills.get(name)

    def list_skills(self) -> list[Skill]:
        return list(self._skills.values())

    def all_tools(self) -> list[Tool]:
        tools = []
        for s in self._skills.values():
            tools.extend(s.tools)
        return tools

    def to_registry(self) -> ToolRegistry:
        reg = ToolRegistry()
        for tool in self.all_tools():
            reg.register(tool)
        return reg

    def summary(self) -> str:
        lines = []
        for s in self._skills.values():
            lines.append(f"[{s.name}] {s.description}")
            for t in s.tools:
                lines.append(f"  - {t.name}: {t.description}")
        return "\n".join(lines)


class SkillLoader:
    @staticmethod
    def load_from_dir(skill_dir: str) -> Skill | None:
        manifest_path = os.path.join(skill_dir, "manifest.json")
        if not os.path.exists(manifest_path):
            return None

        manifest = json.loads(
            open(manifest_path, encoding="utf-8").read())
        skill = Skill(
            name=manifest["name"],
            description=manifest.get("description", ""),
            version=manifest.get("version", "1.0.0"))

        handlers_module = None
        handlers_path = os.path.join(skill_dir, "handlers.py")
        if os.path.exists(handlers_path):
            spec = importlib.util.spec_from_file_location(
                "handlers", handlers_path)
            if spec and spec.loader:
                handlers_module = importlib.util.module_from_spec(spec)
                spec.loader.exec_module(handlers_module)

        for tool_def in manifest.get("tools", []):
            handler = None
            if handlers_module:
                handler = getattr(
                    handlers_module, tool_def["handler"], None)
            if not handler:
                _name = tool_def["name"]
                def _stub(_n=_name, **kw):
                    return f"[stub] {_n}({kw})"
                handler = _stub

            skill.tools.append(Tool(
                name=tool_def["name"],
                description=tool_def.get("description", ""),
                parameters=tool_def.get("parameters", {}),
                handler=handler))

        return skill

    @staticmethod
    def discover(skills_dir: str) -> list[Skill]:
        skills = []
        if not os.path.isdir(skills_dir):
            return skills
        for name in sorted(os.listdir(skills_dir)):
            path = os.path.join(skills_dir, name)
            if os.path.isdir(path):
                skill = SkillLoader.load_from_dir(path)
                if skill:
                    skills.append(skill)
        return skills

7.1 The Skill Module: Why Do We Need It?

Your Agent is smart — it can chat, remember things, and act on schedule. You ask it to "check tomorrow's weather in Beijing" — and it replies: "Sorry, I don't have the ability to check the weather."

You have no choice but to add a weather API call in the code, recompile, and redeploy. Three days later, you want it to check emails, calendars, stock prices… and every time, you have to modify the core code.

This is an Agent without a skill module — every new capability requires source code changes.

The core problem the skill module solves:

Allow the Agent to acquire new capabilities (tools) on demand through a registration mechanism rather than source code modification, while managing tool discovery, permissions, error handling, and lifecycle.

Three projects offer three fundamentally different approaches:

Approach Project Core Idea
Compiled plugins OpenClaw SDK + Registry + 15 lifecycle hooks
Runtime tools Letta Python functions as tools, API-driven dynamic registration
Shell commands NanoClaw Bash + MCP is the entire toolset

7.3 Core Algorithm: hardcoded-tools

The simplest implementation of a skill module requires only 3 lines:

tools = {
    "get_weather": lambda city: f"{city}: 25°C",
    "calculator":  lambda expr: str(eval(expr)),
}

Scoring against the evaluation criteria:

Criterion Simplest Implementation Score
C1 Tool Discovery Hardcoded list ❌ New tools require source changes
C2 Error Tolerance No error handling ❌ eval can execute arbitrary code
C3 Permission Control No permissions ❌ All tools callable without restriction
C4 Third-Party Integration Not supported ❌ Cannot connect to external services
C5 Lifecycle Management No registration mechanism ❌ Cannot add or remove tools

All failing. Let's see how production projects address each criterion.

7.4 Production Implementation: From 3 Tools to 40+ Skills

C1 Tool Discovery: How Does the LLM Know Which Tools Are Available?

The LLM doesn't discover tools on its own — it can only use the ones you tell it about in the request.

OpenClaw: Layered discovery + conditional injection. discoverOpenClawPlugins() (loader.ts:200-210) scans across four priority layers: config > workspace > global > bundled. shouldIncludeSkill performs capability detection on each SKILL.md — OS compatibility, binary dependencies, environment variables — only skills that pass all checks are injected into the system prompt.

Letta: Database-driven + type dispatch. Each Agent is associated with a set of Tool records; 11 tool_type enum values determine the tool's nature. When building the LLM request, tool json_schema is fetched from the database. Tools can be dynamically added or removed at runtime via API.

NanoClaw: Hardcoded list + MCP wildcard. Tools are explicitly listed as allowedTools in query() — file operations (Bash, Read, Write, Edit, Glob, Grep), web access (WebSearch, WebFetch), sub-agent collaboration (Task, TeamCreate, SendMessage), utilities (TodoWrite, ToolSearch, Skill, NotebookEdit), totaling 18+ tools + mcp__nanoclaw__* (MCP wildcard). Bash itself is the "meta-tool" — through it, any command-line program can be invoked.

# OpenClaw style — four-layer priority discovery
def discover_plugins(search_paths):
    seen = set()
    for path in [config_dir, workspace_dir, global_dir, bundled_dir]:
        for plugin in scan_dir(path):
            if plugin.id not in seen:
                seen.add(plugin.id)
                yield plugin            # 先到先得

# Letta style — database query
def get_tools_for_agent(agent_id):
    return db.query("SELECT json_schema FROM tools_agents WHERE agent_id=?",
                    agent_id)

# NanoClaw style — hardcoded
allowed_tools = ["Bash", "Read", "Write", "Edit", "mcp__nanoclaw__*"]

C2 Error Tolerance: What Happens When a Tool Fails?

Tool calls will inevitably fail. The key is how failure information is passed back to the LLM so it can self-correct.

OpenClaw: Hook interception + error boundaries. The before_tool_call hook allows plugins to intercept before execution — modifying arguments or even returning { block: true } to prevent execution. after_tool_call can inspect results and trigger retries.

Letta: Structured results + friendly error messages. ToolExecutionResult marks outcomes with status: "success" | "error". get_friendly_error_msg() generates LLM-friendly error descriptions — including the function name and exception type, but excluding the traceback (traceback goes to stderr for debugging).

# Letta style — structured errors + friendly messages
@dataclass
class ToolExecutionResult:
    status: str        # "success" | "error"
    func_return: str   # 返回值 or 错误描述（给 LLM 看）
    stderr: str        # traceback（给开发者看）

def get_friendly_error_msg(func_name, exception):
    return f"Error in {func_name}: {type(exception).__name__}: {exception}"
    # 不含 traceback → 节省 LLM 上下文窗口

NanoClaw: stderr as feedback. Bash's non-zero exit code + stderr are returned directly to Claude. The AI decides on its own whether to retry.

Design Pattern In-Place — Observer / Hook System

Problem solved: Logic such as checks, logging, and retries needs to be inserted before and after tool execution, but this logic should not be coupled with the tool itself.

Solution: Components subscribe to events via hooks, decoupling publishers from subscribers. OpenClaw's before_tool_call / after_tool_call and 13 other lifecycle hooks are divided into two modes: void (pure notification) and modifying (can alter data). NanoClaw's startIpcWatcher() is a filesystem-based observer — the container writes a file to "publish" a request, and the host polls to "subscribe" and process it.

C3 Permission Control: How Do We Restrict Tool Permissions?

A tool that can send emails, if unrestricted, could spam every contact with advertisements.

OpenClaw: ACL + tool allowlist. resolvePluginTools() supports a toolAllowlist parameter: tools marked as optional are only passed to the LLM if they appear on the allowlist; non-optional tools are always included. Built-in conflict detection automatically reports when multiple plugins register tools with the same name. Combined with the before_tool_call hook, tool calls can be dynamically blocked based on user identity.

Letta: Approval mechanism + sandbox + ToolRule. The default_requires_approval field controls whether human confirmation is needed. Custom tools execute within SandboxToolExecutor. The ToolRule system supports tool call chain constraints (tool A must be followed by tool B).

NanoClaw: Docker container as sandbox. The entire Agent runs inside a container, and the container is the security boundary. The allowedTools list is the entire permission declaration.

C4 Third-Party Integration: How Do External Tools Connect? (MCP)

MCP (Model Context Protocol) is the industry's attempt at standardizing "the USB port for AI" — any tool that implements the MCP protocol (based on JSON-RPC 2.0) can be used by any Agent that supports MCP.

NanoClaw has the deepest MCP integration. Core functionality (send_message, schedule_task, and 5 other tools) is exposed through an MCP Server. Adding a third-party service only requires adding an entry to the mcpServers dictionary.

Letta treats MCP tools as ToolType.EXTERNAL_MCP; ToolCreate.from_mcp() generates Python wrapper functions.

OpenClaw's MCP support is implemented through the plugin system and can be loaded on demand.

Design Pattern In-Place — Registry

mermaid flowchart TB P1["Plugin A<br/>registerTool()"] -->|registerTool| REG["Plugin Registry<br/>tools / hooks / channels"] P2["Plugin B<br/>registerHook()"] -->|registerHook| REG P3["MCP Server<br/>JSON-RPC bridge"] -->|registerTool| REG REG --> RESOLVE["resolvePluginTools()<br/>whitelist + conflict check"] RESOLVE --> LLM["Inject to LLM<br/>tools[] → prompt"]

Problem solved: How are tools from multiple sources (built-in, plugins, MCP) managed uniformly? How are naming conflicts detected? How is allowlist filtering applied?

Solution: A central registry dynamically manages all extensions — registration, lookup, and conflict resolution. OpenClaw's createPluginRegistry() stores 13 registration categories. The registry is the "source of truth" for the plugin system.

C5 Lifecycle Management: Tool Registration, Updates, and Uninstallation?

OpenClaw: Load at startup + filesystem management. loadOpenClawPlugins() uses jiti to JIT-compile and load TypeScript modules at startup. Plugin "installation/uninstallation" is a filesystem operation — copying/deleting folders in the extensions/ directory. Updates require a restart.

Letta: API-driven dynamic management + database persistence. ToolManager.create_tool() creates tools via REST API, with source code stored in the database's source_code field. Tools can be updated or deleted at any time without a restart.

NanoClaw: "Don't add features. Add skills." NanoClaw's extension philosophy is the most radical — instead of adding functionality through code, it teaches the AI how to accomplish new tasks using Markdown files in the .claude/skills/ directory, leveraging Bash and MCP.

Design Pattern In-Place — Dynamic Loading

Problem solved: How do you load new modules without recompiling?

Solution: Load module code at runtime. OpenClaw uses jiti to directly load TypeScript at runtime without pre-compilation. Letta uses importlib to dynamically import Python functions, with derive_openai_json_schema() automatically generating schemas via AST parsing.

"""mini_agent/skills.py — 技能扩展系统

第七章 Lab · 工具注册表 + 动态加载 + 技能目录

单独运行: python -m mini_agent.skills
依赖: 仅 Python 标准库
"""
from __future__ import annotations

import json, os, time, importlib.util
from dataclasses import dataclass, field
from typing import Callable

from mini_agent.runtime import Tool, ToolRegistry


@dataclass
class Skill:
    """一组相关工具的集合"""
    name: str
    description: str
    version: str = "1.0.0"
    tools: list[Tool] = field(default_factory=list)

    def tool_names(self) -> list[str]:
        return [t.name for t in self.tools]


class SkillRegistry:
    def __init__(self):
        self._skills: dict[str, Skill] = {}

    def register(self, skill: Skill) -> None:
        self._skills[skill.name] = skill

    def get(self, name: str) -> Skill | None:
        return self._skills.get(name)

    def list_skills(self) -> list[Skill]:
        return list(self._skills.values())

    def all_tools(self) -> list[Tool]:
        tools = []
        for s in self._skills.values():
            tools.extend(s.tools)
        return tools

    def to_registry(self) -> ToolRegistry:
        reg = ToolRegistry()
        for tool in self.all_tools():
            reg.register(tool)
        return reg

    def summary(self) -> str:
        lines = []
        for s in self._skills.values():
            lines.append(f"[{s.name}] {s.description}")
            for t in s.tools:
                lines.append(f"  - {t.name}: {t.description}")
        return "\n".join(lines)


class SkillLoader:
    @staticmethod
    def load_from_dir(skill_dir: str) -> Skill | None:
        manifest_path = os.path.join(skill_dir, "manifest.json")
        if not os.path.exists(manifest_path):
            return None

        manifest = json.loads(
            open(manifest_path, encoding="utf-8").read())
        skill = Skill(
            name=manifest["name"],
            description=manifest.get("description", ""),
            version=manifest.get("version", "1.0.0"))

        handlers_module = None
        handlers_path = os.path.join(skill_dir, "handlers.py")
        if os.path.exists(handlers_path):
            spec = importlib.util.spec_from_file_location(
                "handlers", handlers_path)
            if spec and spec.loader:
                handlers_module = importlib.util.module_from_spec(spec)
                spec.loader.exec_module(handlers_module)

        for tool_def in manifest.get("tools", []):
            handler = None
            if handlers_module:
                handler = getattr(
                    handlers_module, tool_def["handler"], None)
            if not handler:
                _name = tool_def["name"]
                def _stub(_n=_name, **kw):
                    return f"[stub] {_n}({kw})"
                handler = _stub

            skill.tools.append(Tool(
                name=tool_def["name"],
                description=tool_def.get("description", ""),
                parameters=tool_def.get("parameters", {}),
                handler=handler))

        return skill

    @staticmethod
    def discover(skills_dir: str) -> list[Skill]:
        skills = []
        if not os.path.isdir(skills_dir):
            return skills
        for name in sorted(os.listdir(skills_dir)):
            path = os.path.join(skills_dir, name)
            if os.path.isdir(path):
                skill = SkillLoader.load_from_dir(path)
                if skill:
                    skills.append(skill)
        return skills


def create_builtin_skills(memory=None) -> list[Skill]:
    def _safe_calc(expression: str = "") -> str:
        if all(c in "0123456789+-*/.() " for c in expression):
            try:
                return str(eval(expression))
            except Exception as e:
                return f"Error: {e}"
        return "Error: 非法字符"

    sys_skill = Skill("system", "系统工具集", tools=[
        Tool("get_time", "获取当前时间", {},
             lambda: time.strftime("%Y-%m-%d %H:%M:%S")),
        Tool("calculator", "数学计算",
             {"expression": {"type": "string",
                             "description": "数学表达式"}},
             _safe_calc),
    ])

    skills: list[Skill] = [sys_skill]

    if memory:
        mem_skill = Skill("memory", "记忆管理工具", tools=[
            Tool("search_memory", "搜索历史记忆",
                 {"query": {"type": "string", "description": "搜索词"}},
                 lambda query="": "\n".join(
                     r["text"] for r in memory.recall(query, 3)
                 ) or "无结果"),
            Tool("remember", "记住重要信息",
                 {"label": {"type": "string", "description": "类别"},
                  "content": {"type": "string", "description": "内容"}},
                 lambda label="human", content="": (
                     memory.remember(label, content) or
                     f"已记住: {content}")),
        ])
        skills.append(mem_skill)

    return skills

7.7 Chapter Summary

Criterion Key Insight
C1 Tool Discovery Four-layer priority scanning (OpenClaw) enables precise filtering of 36 extensions and 40+ skills; database-driven (Letta) allows runtime tool add/remove; hardcoded + Bash (NanoClaw) substitutes a meta-tool for tool discovery.
C2 Error Tolerance LLM-friendly error messages are critical — raw tracebacks waste context window. Letta's separation of func_return (for the LLM) + stderr (for the developer) is a good practice.
C3 Permission Control Tool allowlist (OpenClaw), approval + sandbox (Letta), container isolation (NanoClaw) — three levels of granularity, matching three product positioning strategies.
C4 Third-Party Integration MCP works well as a "peripheral interface" (connecting third parties), but not as a "kernel interface" (internal framework calls). All three projects retained their native tool mechanisms.
C5 Lifecycle The boundary between "declaration vs. implementation" reveals the framework's values: the greater the separation (OpenClaw), the heavier the framework; the tighter the coupling (NanoClaw), the lighter the framework but the deeper the AI dependency.
